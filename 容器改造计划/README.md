# LobsterAI 多实例容器化改造计划

> 目标：在一台服务器上，以 Docker 容器（推荐）或虚拟机的形式，同时托管 **N 个（目标 50 个）** 相互隔离的 LobsterAI 实例，每个实例的状态（Skill、文件、会话、记忆、配置、密钥）可持久化保存、可备份、可迁移。

本目录是该改造的完整方案与逐步操作手册。所有结论均基于对当前源码（`src/main`、`scripts`、`package.json`、`electron-builder.json`）的实际侦察，而非泛化经验。若源码演进导致某处过期，请以源码为准。

---

## 一、三个核心问题的结论

### 1. 是否可行？——**可行**

LobsterAI 是一个 Electron 桌面应用，其状态几乎全部集中在一个可控目录（`userData`，Linux 上为 `$HOME/.config/LobsterAI`）下，运行期依赖清晰（一个 Electron 主进程 + 一个 Node 版 OpenClaw 网关子进程），且所有本地服务默认只绑定 `127.0.0.1`。这三点使它非常适合"一容器一实例 + 一卷一状态"的模型。

**关键洞察：容器的隔离能力天然化解了大部分"单机多实例"难题。**
侦察阶段被标为"硬阻断"的两点——单实例锁与共享 `userData`——本质上只在"同一台主机、同一个用户目录里跑 50 个进程"时才成立。一旦每个实例装进独立容器（独立的文件系统命名空间、独立的 `HOME`、独立的网络命名空间/回环），这些问题自动消失，无需改代码：

| 侦察标记的"阻断" | 在独立容器下 | 结论 |
|---|---|---|
| `app.requestSingleInstanceLock()` 单实例锁 | 锁基于 `userData` 目录，每容器各一份 | 非问题 |
| `userData` 目录默认共享 | 每容器独立 `HOME` → 独立 `~/.config/LobsterAI` | 非问题 |
| SQLite 文件锁冲突 | 每容器独占自己的 DB | 非问题 |
| OpenClaw 状态目录/端口冲突 | 每容器独立状态目录 + 独立回环 | 非问题 |
| 各类本地端口冲突（18789、动态端口等） | 每容器独立网络命名空间 | 非问题 |

真正需要投入工作的，是下面三件事（详见各分册）：

1. **让 GUI 在无显示器的服务器上可运行、且可被用户访问**：Electron 没有 headless 模式，渲染进程需要一个显示服务器。方案是每容器内置一个虚拟显示（Xvfb）+ 远程访问通道（noVNC，浏览器即可操作）。
2. **打包一个能在 Linux 容器里跑起来的镜像**：安装 Chromium/GTK 系统库、Node 24、构建 Linux 版 OpenClaw runtime、编译 linux-x64 原生模块（better-sqlite3、sharp 等）。
3. **规模化与外部策略**：50 实例的资源规划（约 100–200 GB 内存量级），以及 Youdao 云端账号/配额/遥测/合规这类**非技术**约束（详见 06 分册）。

### 2. 技术难度如何？——**中等（偏运维/打包，而非深度改代码）**

- **代码改动面很小**：核心隔离靠容器天然获得，无需重构。需要动的基本是"设置几个已存在的环境变量 + 少量可选的健壮性补丁"（详见 05 分册），代码侵入度低。
- **主要工作量在 DevOps**：写一个正确的 Linux Dockerfile（Electron in Docker + Xvfb + noVNC 是成熟且有大量先例的路径）、构建流水线、编排（docker-compose / 后续 K8s）、反向代理路由、监控。
- **主要不确定性在外部策略**：Youdao 后端是否允许同账号大量并发、配额/限流/ToS——这是需要与服务方确认的**业务问题**，不是代码问题。

**难度评级：中等偏上。** 没有需要推翻现有架构的"大手术"；风险集中在打包细节、资源密度与外部合规三处，均在合理可控范围内。

## 二、推荐架构总览

```
┌─────────────────────────────────────────────────────────────────────┐
│  一台服务器 (Linux, 建议 ≥256GB RAM / 32+ vCPU / NVMe)                 │
│                                                                       │
│   ┌──────────────┐   反向代理 (Traefik/Caddy/nginx)                    │
│   │ 用户浏览器     │──▶ https://inst-07.lobster.example.com ──┐         │
│   └──────────────┘                                          │         │
│                                                             ▼         │
│   ┌──────── 容器 inst-07 (一实例) ────────────────────────────────┐    │
│   │  supervisord (PID1, tini 转发信号)                            │    │
│   │   ├─ Xvfb           :99   (虚拟显示)                          │    │
│   │   ├─ openbox        (轻量窗口管理器, 可选)                     │    │
│   │   ├─ x11vnc         (把 :99 暴露为 VNC)                        │    │
│   │   ├─ websockify/noVNC :6080 (VNC→WebSocket, 浏览器访问)        │    │
│   │   └─ LobsterAI (Electron 主进程)                             │    │
│   │        └─ OpenClaw 网关子进程 (127.0.0.1:18789, 容器内私有)     │    │
│   │                                                              │    │
│   │  HOME=/home/lobster  ← 挂载持久卷 vol-inst-07                  │    │
│   │     ~/.config/LobsterAI/  (DB / openclaw 状态 / SKILLs / 记忆) │    │
│   │     ~/projects/          (用户文件 / 会话工作目录)             │    │
│   └──────────────────────────────────────────────────────────────┘    │
│   ┌──────── 容器 inst-08 ────┐  ┌──── inst-09 ────┐   ... × 50         │
│   └──────────────────────────┘  └─────────────────┘                    │
└─────────────────────────────────────────────────────────────────────┘
```

**核心设计决策（推荐值，可在 01 分册中调整）：**

| 决策点 | 推荐 | 理由 |
|---|---|---|
| 隔离单元 | **Docker 容器**（非 VM） | 50× 密度下 VM 太重；容器共享内核，足够隔离本用例 |
| GUI 访问方式 | **Xvfb + noVNC**（浏览器远程桌面） | 改动最小，用户体验接近原生桌面；替代方案见 01 分册 |
| 持久化 | **每容器一个卷，挂在 `HOME`** | 一个卷囊括全部状态（配置/DB/技能/文件），最简洁 |
| 状态隔离 | 靠容器 + 独立卷天然获得 | 无需改代码 |
| 编排 | 起步用 **docker-compose**，规模化后 **K8s/Nomad** | 50 个用 compose 可管理；再大用编排器 |
| 账号/密钥 | 每实例独立账号与 Provider Key（强烈建议） | 规避共享配额耗尽与限流；见 06 分册 |

## 四、给决策者的一句话摘要

> 这件事**可行且难度可控**。它主要是一个"把桌面应用打进带虚拟显示的 Linux 容器、用 noVNC 远程访问、每实例一个持久卷、用 compose/K8s 编排 50 份"的**打包与运维工程**，几乎不需要改动 LobsterAI 的核心代码。最大的两个现实约束是**内存密度**（50 × 约 2–4GB ≈ 需要一台大内存机器）和**Youdao 云端账号/配额/合规策略**（需与服务方确认，属业务而非技术问题）。
>
> **推进方式：分版本、勿一口吃胖。** 按 [00 · 版本路线图](00-版本路线图.md) 走 **MVP（单容器自用可用，约 2–5 天）→ V1（多实例基本可用）→ V2（50 规模生产可用）→ V3（平台化精益求精）**。先把 MVP 当 Go/No-Go 关卡打通，验证成立后再逐版放大——每一版都是能停靠、能用、能改的里程碑。
