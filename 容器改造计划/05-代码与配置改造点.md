# 05 · 代码与配置改造点

**核心结论：改动面很小。** 在"一容器一实例"模型下，隔离由容器天然提供，大部分"多实例难题"无需改代码。本文把需要动的东西分成三档：**必做（几乎都是设环境变量）**、**按需**、**可选的健壮性补丁**。每项附源码依据。

---

## A. 必做（多为环境变量，零/极少代码改动）

### A1. userData / 状态隔离 —— 无需改代码 ✅
- `configureUserDataPath()` 依 `app.getPath('appData')` 推导，Linux 上即 `$HOME/.config/LobsterAI`。
  - 依据：`src/main/main.ts:1113-1124`。
- 每容器独立 `HOME`（Dockerfile 已设 `HOME=/home/lobster`）→ 状态天然隔离。**不改代码。**

### A2. 单实例锁 —— 一容器一实例下无需改代码 ✅
- `app.requestSingleInstanceLock()` 的锁基于 `userData` 目录；每容器各一份，不冲突。
  - 依据：`src/main/main.ts:3124-3127`。
- **仅当**你想在**同一个容器/同一 `userData` 内**跑多个实例（不推荐）才需要绕过——见 C1。默认不改。

### A3. 关闭 GPU（容器无显卡）—— 设环境变量 ✅
- 设 `LOBSTERAI_DISABLE_GPU=1`（Dockerfile 已设），触发 `disable-gpu` + 软件光栅 + `disableHardwareAcceleration()`。
  - 依据：`src/main/main.ts:1147-1151, 1212-1217`。
- Linux 上 `no-sandbox`、`disable-dev-shm-usage` 已自动启用，无需处理。
  - 依据：`src/main/main.ts:1206-1211`。

### A4. Provider Key / 凭据注入 —— 用环境变量/密钥文件，不进镜像 ✅
- `openclaw.json` 用 `${VAR}` 占位，运行时由环境变量填充；库内 provider 配置同理可由每实例配置提供。
  - 依据：`src/main/libs/openclawConfigSync.ts`（provider/env 占位解析）。
- 操作：每实例 `env_file: secrets/lobster-NN.env` 注入（04 分册），**不要**把 Key 写进镜像层。

## B. 按需（取决于你是否用到这些能力）

### B1. 遥测 / 使用分析 —— 建议在服务端部署时关闭
- 每实例生成随机安装 UUID 上报 `rlogs.youdao.com`；受 `usageAnalyticsEnabled` 开关控制。
  - 依据：`src/renderer/services/installationId.ts`、`src/renderer/services/logReporter.ts:235-238`。
- 50 实例 × 高频上报 + 同 IP 可能触发风控/限流（见 06 分册）。**建议关闭**：在应用设置里关掉使用分析，或（若要默认关闭）把该配置项默认值改为 false / 由环境变量控制。
- **动作**：确认关闭入口（设置项/配置键 `usageAnalyticsEnabled`），在实例首启后置为关闭；如需批量默认关闭，做一处小改（默认值或读环境变量）。

### B2. 自动更新检查 —— 服务端托管场景建议禁用
- 应用会周期性向 `api-overmind.youdao.com` 检查更新并带 UUID。
  - 依据：`src/main/libs/appUpdateCoordinator.ts:94-140`、`src/main/libs/endpoints.ts:39-43`。
- 托管环境里镜像由你统一升级（04 分册滚动更新），客户端自动更新既无意义又放大同 IP 请求。**建议禁用/弱化**自动检查（配置或小改），避免"更新风暴"。

### B3. IM 入站 webhook 端口 —— 仅当使用入站 IM 渠道
- 多数本地服务只绑回环、容器内私有，无需处理。**例外**是需要"外部可达"的 IM webhook（如 POPO 硬编码 `3100`）。
  - 依据：`src/main/im/types.ts:447`（POPO webhookPort）。
- 若用到：为该实例的 webhook 端口做容器端口映射 + 反代路由，并在 IM 平台侧填对应公网回调地址；多数纯 GUI 使用场景用不到。

### B4. Codex（旧版 OpenAI）OAuth 端口 1455 —— 仅当使用 Codex
- 硬编码 `127.0.0.1:1455`，容器内私有不冲突；但若多个实例都要跑 Codex OAuth 且共享宿主网络才有问题——容器网络隔离下无碍。
  - 依据：`src/main/libs/openaiCodexAuth.ts:28`。一般无需处理。

### B5. ⚠️ 容器内登录闭环 —— noVNC 方案的隐藏必做项
- 登录 `auth:login` 走 `shell.openExternal(finalUrl)` 打开**外部浏览器**做 OAuth，再重定向回本地回调服务 `http://127.0.0.1:<port>/auth/callback`。
  - 依据：`src/main/main.ts:4565-4604`、`src/main/libs/authLocalCallbackServer.ts:5-7`。
- **问题**：容器/无头环境默认**没有浏览器**，`xdg-open` 直接失败 → 用户无法登录。
- **首选解法（无需改代码）**：镜像内安装浏览器并设为默认（`chromium` + `update-alternatives --set x-www-browser` + `BROWSER=chromium`，[02 分册](02-镜像构建步骤.md) 已包含）。因为 Electron ↔ 浏览器 ↔ `127.0.0.1` 回调都在**同一容器的 loopback** 内，用户经 noVNC 在容器内浏览器完成 OAuth 即可闭环。
- **降级解法（需小改/运维）**：若上面仍不通——① **预置 token**：把一次成功登录得到的 `kv.auth_tokens` 注入卷；② **小改登录**：增加"贴 token / 设备码"式登录入口，绕开外部浏览器（代码改造，属本表 C 类可选补丁）。
- **务必在 [08 · 尖刺 A3](08-实施路线图与验收清单.md) 先验证**，再据结论固化镜像。

## C. 可选的健壮性 / 便利性补丁（非必须）

> 遵循仓库 `AGENTS.md` 的"补丁策略"：优先在 LobsterAI 侧集成点用环境变量/配置解决；确需改行为再做**小而有据**的改动，并保持每个改动文件 ESLint 干净。以下均为可选增强，不做也能跑。

### C1.（可选）同容器多实例的单实例锁旁路
仅当你想在一个容器里跑多个实例（**不推荐**，与"一容器一实例"相悖）。可加一个受环境变量保护的旁路：
```ts
// src/main/main.ts:3124 附近
const bypassLock =
  process.env.LOBSTERAI_DISABLE_SINGLE_INSTANCE_LOCK === '1';
const gotTheLock = bypassLock ? true : app.requestSingleInstanceLock();
```
- 默认关闭；仅在明确需要时置 `1`。注意：绕过后必须保证各实例 `userData` 不同，否则会 SQLite 冲突。

### C2.（可选）优雅关闭以确保 SQLite 刷盘
容器停止时应让主进程收到 `SIGTERM` 并 `app.quit()`，触发 DB 正常关闭/WAL checkpoint：
- 已由 supervisord `stopsignal=TERM` + `stopwaitsecs=25`（02 分册）+ tini 信号转发覆盖。
- 若发现异常退出导致 WAL 残留，可在主进程补一个 `process.on('SIGTERM', () => app.quit())` 兜底（确认现有 `before-quit`/窗口关闭逻辑是否已妥善关库）。

### C3.（可选）容器友好的日志级别
- 主进程默认 `[OpenClaw]` 等日志较多；托管大量实例时可用现有环境变量收敛（如不开 `ELECTRON_ENABLE_LOGGING`），并在 Docker 侧限制日志大小（04 分册）。

### C4.（可选）默认工作目录固定在 HOME 下
- 会话/Agent 工作目录默认 `os.homedir()`（`src/main/coworkStore.ts:37-39`），在容器内即 `/home/lobster`，已在持久卷内，无需改。
- 若产品上想强制工作目录在某子目录（如 `~/projects`），可在配置初始化时设默认值。属产品偏好，非必须。

## D. 汇总：改动清单一览

| 项 | 类型 | 是否必做 | 方式 |
|---|---|---|---|
| userData 隔离 | 隔离 | ✅ 必做 | 容器独立 `HOME`，**零代码** |
| 单实例锁 | 隔离 | ➖ 无需 | 一容器一实例下不冲突 |
| 关 GPU | 运行 | ✅ 必做 | `LOBSTERAI_DISABLE_GPU=1` 环境变量 |
| sandbox/dev-shm | 运行 | ➖ 无需 | Linux 已自动处理 |
| Provider Key 注入 | 凭据 | ✅ 必做 | `env_file` 运行时注入，不进镜像 |
| **容器内登录浏览器** | 凭据 | ✅ 必做(noVNC) | 镜像内置 `chromium` 设默认；失败则预置 token/改登录（B5） |
| 关遥测 | 合规 | 🔶 建议 | 配置项 `usageAnalyticsEnabled` / 小改默认值 |
| 关自动更新 | 合规 | 🔶 建议 | 配置/小改，避免更新风暴 |
| IM webhook 端口 | 网络 | ⭕ 按需 | 端口映射 + 反代，仅用入站 IM 时 |
| 单实例锁旁路 | 便利 | ⛔ 可选 | 环境变量守卫的小补丁（仅同容器多实例时） |
| 优雅关闭兜底 | 稳定 | ⛔ 可选 | supervisord 已覆盖，必要时补 SIGTERM 处理 |

> **要点**：真正"必做"的三项里有两项只是设环境变量，一项是注入密钥的部署实践——**核心代码几乎不用动**。这正是本方案难度可控的根本原因。

外部（非代码）约束请务必读 [06-认证配额与合规风险.md](06-认证配额与合规风险.md)。
