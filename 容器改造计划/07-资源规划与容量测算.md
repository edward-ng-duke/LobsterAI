# 07 · 资源规划与容量测算

目标：给出 50 实例同机托管的服务器选型、密度上限与调优点。**内存是本方案的第一硬约束。**

---

## 1. 单实例资源足迹（估算，需以实测为准）

一个实例 = Electron 主进程 + Chromium 渲染进程 + GPU/工具进程 + OpenClaw Node 网关 + Xvfb + x11vnc + noVNC。

| 组件 | 内存（估） | 说明 |
|---|---|---|
| Electron 主进程 | 150–300 MB | Node + 主逻辑 |
| Chromium 渲染进程 | 300–700 MB | 界面越复杂/会话越多越高 |
| Electron 辅助进程（GPU/utility） | 100–250 MB | 软件渲染下 GPU 进程较轻 |
| OpenClaw 网关（Node） | 300–800 MB | `--max-old-space-size=4096` 是**上限**非常驻；随会话/技能增长 |
| Xvfb | 50–120 MB | 视分辨率 |
| x11vnc + websockify/noVNC | 30–100 MB | 有活跃连接时略升 |
| **合计（空闲~轻载）** | **≈ 1.2–2.2 GB** | |
| **合计（活跃会话/多技能）** | **≈ 2.5–4 GB** | 峰值 |

- **CPU**：空闲极低；**冷启动**（Electron + 网关起来，2–12s）和**推理请求转发**时会有短时占用。软件渲染下界面动画/滚动也吃 CPU。按 **0.5–1.5 vCPU/实例**（软配额）规划。
- **磁盘**：镜像共享（只读层复用）；每实例卷 **5–30 GB**（DB + SKILLs + 工作区 + 用户文件），见 03 分册。
- **GPU**：**不需要**（代码可关硬件加速，无本地大模型）。用软件渲染（SwiftShader）。
- **网络**：noVNC 是像素流，活跃交互时每实例 **~0.5–5 Mbps**（取决于分辨率与操作频率）；空闲近 0。

> ⚠️ 这些是基于架构的估算。**务必用单容器 `docker stats` 在真实负载下实测**（03 分册冒烟步骤），再据实测定 compose 的 `memory` 上限与单机密度。

---

## 2. 50 实例的服务器选型

以"活跃峰值 2.5–4 GB/实例"取中值 **~3 GB** 规划（并预留内核/反代/监控/缓冲）：

| 资源 | 计算 | 建议 |
|---|---|---|
| **内存** | 50 × 3 GB = 150 GB + 系统/反代/监控 ~16–32 GB + 缓冲 | **≥ 192 GB，推荐 256 GB** |
| **CPU** | 50 × (0.5–1.5) vCPU，考虑不会全部同时峰值 | **≥ 32 vCPU（推荐 48–64）** |
| **磁盘** | 50 × (5–30 GB) 卷 + 镜像 + 日志 | **≥ 1 TB NVMe SSD（推荐 1.5–2 TB）** |
| **磁盘类型** | SQLite 需 POSIX 锁 | **本地 NVMe / 云块存储**，勿用 NFS |
| **网络** | 50 × noVNC 像素流峰值 | **≥ 1 Gbps**（并发交互多时考虑 2.5–10 Gbps） |
| **内核参数** | 大量进程/句柄 | 调高 `fs.file-max`、`ulimit -n`、`vm.max_map_count`、`kernel.pid_max` |

**密度经验值**：
- 保守（活跃用户多）：**单机 20–30 实例**更稳。
- 目标 50：需 **256 GB 内存**级机器，且做好内存上限、错峰启动、监控告警。
- 若单机放不下 50，按内存**横向分摊到 2–3 台**，反代统一入口（04 分册）。

## 3. 密度调优点

1. **给每容器设 `memory` 硬上限**（04 分册）：先 4g 观测，逐步收敛到 2.5–3g，避免个别实例膨胀吞掉整机。
2. **限制 OpenClaw 网关堆**：默认 `--max-old-space-size=4096`（`openclawEngineManager.ts`）。若单实例常驻远低于 4G，可评估在集成侧调低（需谨慎，避免 OOM 崩溃循环）。
3. **降低 Xvfb 分辨率**：`SCREEN_GEOMETRY=1366x768x24` 比 `1600x900` 省内存与带宽；按用户体验取舍。
4. **noVNC 压缩/画质**：调 x11vnc/noVNC 的压缩与色深，降带宽。
5. **错峰启动**：分批 up，避免 50 个冷启动的 CPU/IO 风暴（04 分册）。
6. **`disable-dev-shm-usage` 已启用**：省 `/dev/shm`；仍建议 `--shm-size=1g` 兜底。
7. **关遥测/自动更新**：省网络与后端压力（05/06 分册）。
8. **休眠空闲实例（进阶）**：对长期无人访问的实例，可停容器释放内存、保留卷，用户访问时再拉起（需在反代/调度层做"按需唤醒"，属进阶优化）。

## 4. 监控与告警阈值（建议）

| 指标 | 告警阈值（建议） | 动作 |
|---|---|---|
| 宿主可用内存 | < 15% | 停止新建实例；排查膨胀实例 |
| 单实例 RSS | > 设定上限 90% | 观察是否内存泄漏；必要时重启该实例 |
| 宿主 load / CPU | 持续 > 80% | 降密度 / 加机器 |
| 卷用量 | > 80% | 扩卷 / 清理 / 归档 |
| 容器重启次数 | 短时间多次 | 查 supervisord/应用日志，定位崩溃 |
| OpenClaw 网关健康 | 探活失败 | 网关重启已内建退避；连续失败则告警 |

## 5. Docker 容器 vs 虚拟机（为何推荐容器）

| 维度 | Docker 容器（推荐） | 虚拟机 |
|---|---|---|
| 单实例开销 | 低（共享内核，仅进程级） | 高（每台一套 OS，+数百 MB~GB 内存与磁盘） |
| 50× 密度 | 可行（内存是主约束） | 昂贵（50 套 OS 开销叠加） |
| 启停速度 | 秒级 | 慢 |
| 隔离强度 | 命名空间/cgroup（足够本用例） | 更强（硬件级） |
| 运维 | 镜像统一、编排成熟 | 较重 |
| 适用 | **本方案默认** | 仅当需要更强隔离/合规硬要求时 |

> 若安全合规要求"硬隔离"，可折中用 **gVisor / Kata Containers**（容器接口 + 更强隔离），兼顾密度与隔离，但会牺牲部分性能与增加复杂度。

---

## 6. 成本直觉（仅数量级）

- **单台 256 GB / 48–64 vCPU / 2 TB NVMe** 服务器（自建或云）即可承载 ~50 实例的**基础设施**成本。
- **真正的大头往往是模型调用（LLM API）**：50 个活跃实例并发调用 LLM 的 token 费用，可能远超服务器成本。**务必对每实例做用量计量与限额**（06 分册 R2），否则成本不可控。

容量确定后，按 [00-版本路线图.md](00-版本路线图.md) 分版本推进（本册主要支撑 **V2 规模化**），逐条验收见 [08-实施路线图与验收清单.md](08-实施路线图与验收清单.md)。
