# 01 · 可行性分析与架构设计

本文回答"为什么可行、有哪些坑、最终选什么架构"。所有判断基于对当前源码的侦察，关键处附 `文件:行` 依据。

---

## 1. 六个维度的侦察结论

### 1.1 启动与 GUI 模型（最关键）

- LobsterAI 是 Electron 桌面应用，`createWindow()` 总会创建 `BrowserWindow`，**没有 headless / 无窗口模式**。渲染进程（Chromium）需要一个显示服务器。
  - 依据：`src/main/main.ts` 的 `createWindow()`（约 9449 行）、多个 IPC 处理器假设 `mainWindow` 存在。
- 存在单实例锁 `app.requestSingleInstanceLock()`；抢锁失败即 `app.quit()`。
  - 依据：`src/main/main.ts:3124-3127`、`second-instance` 事件 `:3199`。
  - **重要澄清**：Electron 的单实例锁**基于 `userData` 目录**。每个容器有独立 `HOME` → 独立 `userData` → 独立锁文件，因此**一容器一实例时锁根本不冲突**，无需改代码。它只在"同一 `userData` 下起第二个实例"时才拦截。
- `userData` 由 `configureUserDataPath()` 设为 `app.getPath('appData')/LobsterAI`，Linux 上即 `$HOME/.config/LobsterAI`。
  - 依据：`src/main/main.ts:1113-1124`。
  - **含义**：只要每容器 `HOME` 不同（容器本就如此），状态目录天然隔离，**无需注入路径变量**。
- Linux 上已默认 `no-sandbox` 与 `disable-dev-shm-usage`；GPU 硬件加速默认开启，可用 `LOBSTERAI_DISABLE_GPU=1` 关闭。
  - 依据：`src/main/main.ts:1206-1216`、`disableGpu` 读取 env `:1147-1151`。

> **结论**：GUI 强制存在是唯一的实质性技术门槛，解决办法是"虚拟显示 + 远程访问"，不是改代码。单实例锁/路径共享在容器模型下均非问题。

### 1.2 状态持久化

- 状态高度集中在 `userData`（`~/.config/LobsterAI`）下：SQLite 主库 `lobsterai.sqlite`、`openclaw/state/`（含 `openclaw.json`、`workspace-main`、`workspace-{agentId}`、记忆、`gateway-token`、`gateway-port.json`）、`SKILLs/`、IM 媒体 `nim-inbound/`/`qq-inbound/`、`codex/`、`backups/`。
  - 依据：`src/main/sqliteStore.ts:36`、`src/main/libs/openclawEngineManager.ts:201-212`、`src/main/libs/openclawMemoryFile.ts:47`、`src/main/skillManager.ts:1357`、`src/main/im/nimMedia.ts:21`、`src/main/im/qqMediaDownload.ts:12`。
- 少量状态在 `userData` 之外：迁移回滚目录 `~/.config/LobsterAI-migration-rollbacks`（`main.ts:1129`），用户可见工作目录默认 `os.homedir()`（`coworkStore.ts:37-39`），会话附件临时目录在 `app.getPath('temp')`。
- **含义**：把整个 `HOME` 作为一个持久卷，即可一网打尽全部状态（含上述"之外"的部分）。详见 03 分册。

### 1.3 OpenClaw 运行时/网关

- 每实例内启动一个 Node 版 OpenClaw 网关子进程，`--bind loopback --port <动态>`，默认 18789，失败在 18789–18868 内扫描。
  - 依据：`src/main/libs/openclawEngineManager.ts:38-39, 586, 1267-1298`。
  - **含义**：容器内私有回环 + 独立网络命名空间 → 端口天然不冲突。
- runtime 在构建期生成：克隆 `github.com/openclaw/openclaw` 指定 tag、npm 安装、打包成单文件 `gateway-bundle.mjs`（把冷启动从 80–100s 降到 2–12s），外置若干原生依赖（sharp、better-sqlite3、koffi、playwright、ffmpeg-static 等）。
  - 依据：`scripts/build-openclaw-runtime.sh`、`scripts/bundle-openclaw-gateway.cjs:72-101`、`scripts/ensure-openclaw-version.cjs`。
  - **含义**：构建镜像时需要网络访问；原生依赖必须按 **linux-x64/arm64** 目标平台准备。
- 网关设 `--max-old-space-size=4096`（4GB V8 堆上限，是上限非常驻）；健康探针每 600ms；崩溃重启退避 `[3,5,10,20,30]s`，最多 5 次。
  - 依据：`openclawEngineManager.ts:40-42, 494-542, 1300-1390`。

### 1.4 认证 / 授权 / 多租户

- Youdao 云端 OAuth（`lobsterai-server.youdao.com`），token 存本地 SQLite `kv` 表；每实例生成随机安装 UUID 上报遥测（`rlogs.youdao.com`）；配额按账号（非按设备）。**未发现**硬件绑定、机器指纹或客户端侧"每账号并发数/设备数"限制。
  - 依据：`src/main/main.ts:4565-4646, 3475-3484`、`src/main/libs/endpoints.ts:29-43`、`src/renderer/services/installationId.ts`、`src/renderer/services/logReporter.ts`、`src/main/authQuota.ts:33-38`。
- **含义（关键的非技术风险）**：50 实例共用一个账号会**共享配额**、并可能因**同 IP 高频**触发后端限流/风控；同账号并发是否被服务端允许属**未知的服务端策略**。→ 详见 06 分册。遥测可通过 `usageAnalyticsEnabled` 关闭。

### 1.5 端口 / 网络 / 本地服务

- 所有本地服务**只绑 `127.0.0.1`**：OpenClaw 网关、HTML 预览服务、MCP 桥、OAuth 回调、token 代理、OpenAI 兼容代理等，且多为动态端口。
  - 依据：`htmlPreviewServer.ts:317`、`mcpBridgeServer.ts:179`、`authLocalCallbackServer.ts:228`、`openclawTokenProxy.ts:47`、`coworkOpenAICompatProxy.ts:2921`。
- 少量硬编码端口：Vite 5175（仅 dev）、Codex OAuth 1455（可选功能）、POPO IM webhook 3100（仅用入站 IM webhook 时）。
  - 依据：`vite.config.ts:9`、`openaiCodexAuth.ts:28`、`src/main/im/types.ts:447`。
- **含义**：容器网络命名空间隔离下，端口冲突基本消失。唯一需要"对外可达"的是 IM 入站 webhook，需按实例做端口映射/反代（多数部署用不到）。

### 1.6 资源足迹与构建

- Node 锁定 `>=24.15 <25`；Electron 40.2.1；无需 GPU（可软件渲染）；无本地大模型。
  - 依据：`package.json:71, 227`、`electron-builder.json:259`。
- 原生模块需按平台构建：`better-sqlite3`、`sharp/@img/*`、`node-nim`、`koffi` 等（`asarUnpack` 解包）。
  - 依据：`electron-builder.json:260-265`、`prune-openclaw-runtime.cjs:101`。
- 每实例约 **2–4 GB** 内存量级（Electron/Chromium + 网关 + 缓存/DB），加 Xvfb ~100MB。Linux 是构建目标（`dist:linux`，AppImage/deb），但相对 mac/win **测试较少**。
  - 依据：`package.json:97-124`、`electron-builder.json:175-251`；`README.md` 只写了 macOS/Windows。
- Linux 系统库未在仓库中显式声明（需自行在镜像里补齐 GTK/NSS/X11 等）。

## 2. GUI 访问方式：三方案对比

由于 GUI 强制存在，"如何让用户操作这 50 个实例"是架构主决策。三条路线：

### 方案 A（推荐）：Xvfb + noVNC —— 每容器一个远程桌面
- 容器内跑虚拟显示 Xvfb，Electron 渲染到它；x11vnc 把该显示暴露为 VNC；websockify/noVNC 转成 WebSocket，用户用**浏览器**即可操作完整原生界面。
- **优点**：改动几乎为零；100% 保真原生 UI、所有功能可用；技术成熟、先例极多。
- **缺点**：每实例常驻一套 Chromium 渲染，内存开销高（密度受限于内存）；交互是像素流，带宽/延迟略高于纯 Web。
- **适用**：本需求（"启动任意多个可交互的 Lobster 实例并托管")的最佳匹配。

### 方案 B：重构为前后端分离的 Web 应用
- 把 React 渲染层从 Electron 里剥离，用浏览器直连；主进程逻辑改成 HTTP/WS 服务。
- **优点**：密度最高、体验最"云原生"。
- **缺点**：**工作量巨大**——渲染层通过 `preload.ts` + `contextBridge` 与主进程深度耦合（大量 IPC 通道），等于把桌面应用重写成 Web 应用。短期不现实。
- **适用**：长期产品化方向，非本次改造范围。

### 方案 C：Headless / 纯 API·IM 驱动（无 GUI 交互）
- 只跑主进程逻辑 + OpenClaw 网关，通过 IM 渠道（钉钉/飞书/Telegram…）或 API 使用，不提供图形界面。
- **优点**：密度较高（省掉渲染/显示层）。
- **缺点**：GUI 强制存在，需改代码绕过窗口创建，且很多能力经 IPC 暴露、失去可视化操作；"用户添加 Skill/文件"的可视化体验没有了。
- **适用**：如果最终用途其实是"跑一批无人值守的 IM 机器人"，可专门评估此路线（会改成另一套计划）。

> **本计划默认采用方案 A**。若你的真实目标更接近 C（无人值守机器人），请告知，我会据此改写 02/04 分册。

## 3. 最终架构与组件清单（方案 A）

**一容器一实例**，容器内进程栈（由 supervisord 托管，tini 作 PID1 转发信号）：

| 组件 | 作用 | 端口/显示 |
|---|---|---|
| `tini` | PID1，转发 SIGTERM，回收僵尸进程 | — |
| `supervisord` | 进程编排与拉起顺序、崩溃重启 | — |
| `Xvfb` | 虚拟显示（无物理显示器） | `DISPLAY=:99` |
| `openbox`（可选） | 轻量窗口管理器，改善窗口行为 | — |
| `x11vnc` | 把 `:99` 暴露为 VNC | `5900`（容器内） |
| `websockify`+`noVNC` | VNC→WebSocket，浏览器访问 | `6080`（对外，经反代） |
| `LobsterAI`（Electron） | 应用本体，`HOME` 指向持久卷 | 窗口在 `:99` |
| OpenClaw 网关（子进程） | 由应用自行拉起 | `127.0.0.1:18789`（容器私有） |

**外层（宿主/编排）：**
- 每实例一个命名持久卷，挂在容器 `HOME`（如 `/home/lobster`）。
- 反向代理（Traefik/Caddy/nginx）按子域名/路径把 `6080` 路由到对应容器；建议叠加认证（Basic Auth/OAuth）与 TLS。
- 资源限制：每容器设 `mem_limit`/`cpus`，防止单实例 OOM 拖垮全机。
- 监控：容器存活、内存、网关健康、noVNC 可达性。

**数据流：**
用户浏览器 → HTTPS(反代) → 容器 noVNC:6080 → x11vnc → Xvfb:99 ←渲染← Electron 渲染进程 ↔(IPC) Electron 主进程 ↔(HTTP,回环) OpenClaw 网关 → 各 LLM/云端 API。

---

## 4. 可行性小结

- **技术上**：方案 A 无需重构，唯一门槛（GUI）由 Xvfb+noVNC 成熟解决；隔离由容器天然提供；状态由单卷持久化。→ **可行**。
- **工程量**：集中在镜像打包（Linux 系统库 + runtime + 原生模块）与编排/反代/监控。→ **中等**。
- **主要风险**：内存密度（07 分册）与 Youdao 账号/配额/合规（06 分册）。→ **可控但需正视**。
