# 03 · 状态持久化与卷映射

目标：容器销毁/重建后，某实例的全部用户状态（Skill、文件、会话、记忆、配置、密钥、IM 绑定）原样保留；且可备份、可迁移到另一台机器。

---

## 1. 核心策略：一个卷挂在 `HOME`

LobsterAI 的持久状态**几乎全部**位于 `userData`（Linux = `$HOME/.config/LobsterAI`）下，少量在 `HOME` 其它位置（工作目录、迁移回滚目录）。因此最简洁、最不易漏的方案是：

> **给每个容器一个独立命名卷，挂在容器的 `HOME`（`/home/lobster`）。**

这样 `~/.config/LobsterAI/`（配置/DB/技能/记忆/OpenClaw 状态）、`~/projects/`（用户文件/会话工作目录）、`~/.config/LobsterAI-migration-rollbacks/`（回滚）全部落在同一个卷上，一并持久化、一并备份。

```yaml
# docker-compose 片段（详见 04 分册）
volumes:
  - lobster_home_07:/home/lobster
```

镜像内 `HOME=/home/lobster` 已在 Dockerfile 设定；无需改任何代码即可让状态落到卷上（`configureUserDataPath()` 会据 `HOME` 推导出 `~/.config/LobsterAI`，见 `src/main/main.ts:1113-1124`）。

---

## 2. 精确的持久化清单（若想细粒度挂载）

若你不想挂整个 `HOME`、而要精确控制，需要持久化的路径如下（均相对 `userData = ~/.config/LobsterAI`，除非注明）：

| 路径 | 内容 | 依据 | 必须持久化 |
|---|---|---|---|
| `lobsterai.sqlite`（含 `-wal`/`-shm`） | 会话、消息、Agent、记忆、配置、MCP、插件、订阅缓存等主库 | `sqliteStore.ts:36`、`appConstants.ts:5` | ✅ 核心 |
| `openclaw/state/openclaw.json` | 生成的 OpenClaw 运行配置（provider/agent/绑定/MCP/插件） | `openclawEngineManager.ts:208` | ✅（可再生，但省事保留） |
| `openclaw/state/workspace-main/` | 主 Agent 工作区：`MEMORY.md`/`IDENTITY.md`/`SOUL.md`/`USER.md`/`AGENTS.md`/`memory/` | `openclawMemoryFile.ts:47` | ✅ 记忆/身份 |
| `openclaw/state/workspace-{agentId}/` | 各自定义 Agent 工作区 | `openclawAgentModels.ts:224` | ✅ |
| `openclaw/state/gateway-token` | 网关鉴权 token | `openclawEngineManager.ts:206` | ⛔ 可再生 |
| `openclaw/state/gateway-port.json` | 上次网关端口 | `openclawEngineManager.ts` | ⛔ 可再生 |
| `openclaw/state/.compile-cache/` | V8 编译缓存 | 同上 | ⛔ 可再生（保留可加速） |
| `SKILLs/` | 用户添加/安装的技能（`SKILL.md` + 依赖 + `skills.config.json` + `_meta.json`） | `skillManager.ts:1357` | ✅ 核心 |
| `nim-inbound/`、`qq-inbound/` | IM 入站媒体文件 | `nimMedia.ts:21`、`qqMediaDownload.ts:12` | ⭕ 视是否用 IM |
| `codex/` | 旧版 OpenAI Codex 鉴权 | `openaiCodexAuth.ts:127` | ⭕ 视是否用 Codex |
| `backups/` | SQLite 自动备份 | `sqliteStore.ts:42` | ⭕ 建议保留 |
| `~/.config/LobsterAI-migration-rollbacks/`（**在 userData 之外**） | 数据迁移回滚快照 | `main.ts:1129` | ⭕ 建议保留 |
| 用户工作目录（默认 `os.homedir()`，可在会话/Agent 中自定义为绝对路径） | 用户项目/文件/会话 cwd | `coworkStore.ts:37-39` | ✅ 用户文件 |
| 会话附件临时目录（`app.getPath('temp')/lobsterai/attachments`，或 `{sessionCwd}/.cowork-temp`） | 上传附件的临时副本 | `main.ts:902-905` | ⛔ 临时 |

**结论**：细粒度方案至少要挂 `~/.config/LobsterAI`（去掉可再生项也行）**加上**用户工作目录。既然二者都在 `HOME` 下，**直接挂 `HOME` 最省心、最不易漏**，强烈推荐第 1 节方案。

> ⚠️ 绝对路径注意：Agent 的 `working_directory`、会话 `cwd` 以**绝对路径**存库（`openclawAgentModels.ts:208`）。因此容器内的这些路径必须与存库时一致——统一约定工作目录在 `HOME` 下（如 `/home/lobster/projects`）即可保证跨重建一致，不要用宿主机随机路径。

## 3. 密钥与凭据：用卷 + 环境变量，勿硬编码进镜像

- **Provider API Key / 云端 token**：存在库或 `openclaw.json` 里，后者用 `${VAR}` 占位由环境变量注入（`openclawConfigSync.ts`）。
  - **不要**把任何真实 Key 写进镜像层；用 compose 的 `environment`/`env_file` 或 Docker/K8s secret 在**运行时**注入到该容器。
  - 每实例一套独立 Key（强烈建议，见 06 分册），通过每实例不同的 `env_file` 提供。
- **Youdao 登录 token**：登录后写入该实例卷内的 SQLite `kv` 表，随卷持久化；每实例各自登录一次即可。
- **安全提醒**：库内 token/Key 目前为**明文**存储（`sqliteStore.ts` kv/cowork_config）。卷应设权限（仅 `lobster` 用户可读）、宿主目录加密（LUKS/云盘加密），并纳入密钥泄露的风险评估（06 分册）。

## 4. 备份与恢复

**卷即备份单元。** 每个实例一个卷，备份/恢复以卷为粒度。

### 冷备份（推荐，最稳）
先优雅停容器（触发 SQLite 刷盘与 WAL checkpoint），再打包卷：
```bash
docker stop --time 30 lobster-07          # 给足 stopwaitsecs 刷盘
docker run --rm -v lobster_home_07:/data -v $PWD/backup:/backup alpine \
  tar czf /backup/lobster-07-$(date +%F).tar.gz -C /data .
docker start lobster-07
```

### 热备份（无停机，需容忍 WAL 未 checkpoint）
```bash
docker run --rm -v lobster_home_07:/data -v $PWD/backup:/backup alpine \
  tar czf /backup/lobster-07-hot.tar.gz -C /data .
```
> 热备时 SQLite 可能有未合并的 `-wal`；恢复后由应用自动 checkpoint。要求点一致性时用冷备。应用自带 `SqliteBackupManager` 会在 `backups/` 内做健康备份（`sqliteStore.ts:42-58`），也随卷一并保留。

### 恢复 / 迁移到另一台机器
```bash
docker volume create lobster_home_07
docker run --rm -v lobster_home_07:/data -v $PWD/backup:/backup alpine \
  tar xzf /backup/lobster-07-2026-07-02.tar.gz -C /data
# 再用同镜像启动，指向该卷即可；因工作目录约定在 HOME 下，绝对路径天然一致
```

## 5. 存储后端与并发注意

- **必须是本地/块存储语义**：SQLite（better-sqlite3）依赖 POSIX 文件锁。**不要**把卷放在 NFS/对象存储/弱一致网络盘上，否则可能锁异常或损坏。用本地 SSD/NVMe 或云块存储（EBS/云盘）。
- **一卷一实例，单写者**：绝不让两个容器挂同一个卷同时运行（会 SQLite 锁冲突/损坏）。编排层用唯一卷名 + 唯一容器名保证（04 分册）。
- **磁盘规划**：每实例 `SKILLs` + DB + 工作区 + 用户文件，估 **5–30 GB/实例**（视使用深度）。50 实例预留 **0.5–1.5 TB**，并对卷做用量监控与配额（07 分册）。

## 6. 快速验收（持久化）

1. 起容器 → 登录 → 加一个 Skill、建一个会话、上传一个文件、在工作目录建一个文件。
2. `docker stop --time 30` 后 `docker rm` 容器（**保留卷**）。
3. 用同镜像 + 同卷重新 `docker run`。
4. 断言：Skill/会话/记忆/上传文件/工作目录内容**全部还在**，无需重新登录。

通过后进入 [04-编排与规模化运行.md](04-编排与规模化运行.md)。
