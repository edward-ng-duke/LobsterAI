# 🔴 阶段 A · 可行性尖刺（MVP 上半场）⏱ 3–7 天

> 目的:在**裸机**上用最便宜方式逐个证明,任一失败尽早暴露。允许脏、允许手工。**严格按 A1→A5 顺序。**

## A1 · Linux 能构建出可运行产物 🔴 ⏱ 1–3 天
**目标**:证明 `dist:linux` 在 Linux 上能完整跑通。这是最可能翻车、最该先做的一步。
**步骤**:
1. 在实验台克隆仓库,`npm ci`(触发 `postinstall` 的 `patch-package` + `install-app-deps`,按 Electron ABI 重建原生模块)。
2. `npm run dist:linux`(会先 `predist:linux` 构建 linux-x64 的 OpenClaw runtime,需联网克隆 `github.com/openclaw/openclaw` 及安装插件)。
3. 观察三类易错点:原生模块编译(`better-sqlite3`/`sharp`/`node-nim`)、OpenClaw runtime 克隆与打包、内部 registry 插件(`npm.nie.netease.com`,`optional` 可跳过)。
**退出标准**:
- [ ] `npm run dist:linux` 退出码 0。
- [ ] `release/linux-unpacked/LobsterAI` 存在且可执行。
- [ ] `release/linux-unpacked/resources/cfmind`(OpenClaw runtime)存在。
**风险与降级**:🔴 高。⤵ 若插件安装因内部 registry/网络失败,先从 `package.json` 的 `openclaw.plugins` 移除 `optional` 项、或配代理;⤵ 若 runtime 克隆不稳,预构建 `vendor/openclaw-runtime/` 并用 `OPENCLAW_SKIP_ENSURE=1`;⤵ 若原生模块编译失败,锁定缺失的系统头文件/编译器版本。**此步失败=方案受阻,必须先解决。**

## A2 · 无头（Xvfb）渲染 🔴 ⏱ 0.5–1 天
**目标**:证明 Electron 界面能在虚拟显示上起来、不崩,OpenClaw 网关能健康。
**前提**:A1 通过。
**步骤**:
1. 实验台安装 Electron 运行所需系统库(见 [02 分册](02-镜像构建步骤.md) apt 列表)。
2. `xvfb-run -s "-screen 0 1600x900x24" ./release/linux-unpacked/LobsterAI --no-sandbox --disable-gpu`。
3. 另开 `x11vnc -display :99`(或截图)确认界面确实渲染;查看主进程日志确认 OpenClaw 网关 `127.0.0.1:18789` 健康探针通过。
**退出标准**:
- [ ] 应用启动无崩溃,`xvfb-run` 下界面可被 VNC/截图看到。
- [ ] 日志显示 OpenClaw 网关 ready(健康探针通过)。
- [ ] 无致命的 `chrome-sandbox`/GPU/`/dev/shm` 报错(必要时 `--shm-size`/tmpfs)。
**风险与降级**:🟡 中。⤵ 缺库→按报错补 `lib*`;⤵ shm 报错→挂 tmpfs `/dev/shm`;⤵ 网关起不来→查 `resources/cfmind` 是否完整、node ABI 是否匹配。

## A3 · ⚠️ 登录闭环 🔴 ⏱ 1–2 天（最容易被忽略）
**目标**:在无头+远程环境完成 OAuth 登录并把 token 落库。**这是隐藏地雷**:登录用 `shell.openExternal()` 开外部浏览器,再重定向回 `127.0.0.1` 本地回调(`src/main/main.ts:4588`、`authLocalCallbackServer.ts:5-7`),容器/裸机默认**没有浏览器**会直接失败。
**前提**:A2 通过。
**步骤**:
1. 实验台安装浏览器并设为默认:`apt install -y chromium`(或 `firefox-esr`),`xdg-settings set default-web-browser chromium.desktop`(或 `update-alternatives --set x-www-browser $(which chromium)`)。
2. 在 `xvfb-run` 会话里启动应用,点登录:验证 `shell.openExternal` 是否在**容器内浏览器**打开 Youdao 登录页。
3. 完成 OAuth,确认重定向到 `http://127.0.0.1:<port>/auth/callback`(同一网络命名空间内的 loopback,可达),`auth:exchange` 成功、token 写入 SQLite `kv`。
4. 重启应用,确认仍是登录态(`auth:getUser` 返回)。
**退出标准**:
- [ ] 点击登录能在容器内浏览器打开 Youdao 登录页。
- [ ] OAuth 回调成功,主界面进入已登录状态。
- [ ] 重启后免登录(token 已随 HOME 持久)。
**风险与降级**:🔴 高。⤵ 若 `xdg-open`/默认浏览器关联仍不通→在启动脚本显式设 `BROWSER=chromium`;⤵ 若外部浏览器路径彻底走不通→评估**预置 token**(把一次登录得到的 `kv.auth_tokens` 注入卷)或**小改登录**支持"贴 token/设备码"方式(代码改造,记入 [05 分册](05-代码与配置改造点.md));⤵ 记录最终采用的方案,B 阶段镜像据此固化(**镜像必须内置浏览器**)。

## A4 · 真实 agent 回合 🟡 ⏱ 0.5 天
**目标**:证明配好 provider 后能跑通一次完整 LLM 对话(端到端打通模型链路)。
**前提**:A3 通过。
**步骤**:
1. 在应用内配置一个可用的 provider(Youdao 云端额度 或 自建/第三方 Key,经 `env` 注入)。
2. 新建会话,发一条消息,等待助手完整回复。
**退出标准**:
- [ ] 助手回复正常渲染,无鉴权/配额/网关错误。
- [ ] (如启用)工具调用/技能路由可用。
**风险与降级**:🟡 中。⤵ 429/配额→换 Key 或降并发;⤵ 网关报错→回到 A2 排查 runtime。

## A5 · 状态持久化闭环 🟡 ⏱ 0.5 天
**目标**:证明全部用户状态可持久且重启还原。
**前提**:A4 通过。
**步骤**:
1. 加 1 个 Skill、上传 1 个文件、在工作目录(HOME 下)建 1 个文件、留 1 条会话记忆。
2. 结束进程,用**同一 HOME 目录**重新启动。
3. 逐项核对是否还在。同时 `docker`/`ps`+`/usr/bin/time -v` 或后续 `docker stats` 采内存峰值。
**退出标准**:
- [ ] Skill / 会话 / 记忆 / 上传文件 / 工作目录内容全部还在,免登录。
- [ ] 已记录空闲与活跃内存峰值(喂给 [07 分册](07-资源规划与容量测算.md))。
**风险与降级**:🟢 低。⤵ 若丢状态→核对 `userData` 是否落在 HOME 下、绝对路径是否一致([03 分册](03-状态持久化与卷映射.md))。

### ✅ 阶段 A 出口（Go/No-Go）
- [ ] A1–A5 全绿。→ **方案技术可行,进入阶段 B。** 任一红灯:先解决,或调整方案(含回退到 [01 方案 C](01-可行性分析与架构设计.md))。

# 🟡 阶段 B · 镜像固化（MVP 下半场）⏱ 2–4 天

> 把 A 阶段手工跑通的东西固化成"一条命令可复现"的镜像。参照 [02](02-镜像构建步骤.md)/[03](03-状态持久化与卷映射.md)。

## B1 · builder 阶段 🟡 ⏱ 0.5–1 天
- [ ] 写多阶段 Dockerfile 的 `builder`(node:24-bookworm + git/python3/build-essential),`npm ci && npm run dist:linux` 在镜像内可复现;或改为拷入 A1 预构建产物以加速。
- [ ] 处理构建期网络/代理(GitHub/npm/内部 registry)。

## B2 · runtime 阶段 🟡 ⏱ 0.5–1 天
- [ ] `debian:bookworm-slim` + Electron 系统库 + `xvfb x11vnc novnc websockify openbox supervisor tini` + CJK 字体。
- [ ] **按 A3 结论内置浏览器**(`chromium` 或 `firefox-esr`)并设为默认——**这是登录能用的前提,别漏**。
- [ ] 拷入 `release/linux-unpacked` 到 `/opt/lobsterai`;非 root 用户;`HOME=/home/lobster`。

## B3 · 进程编排与优雅停机 🟡 ⏱ 0.5 天
- [ ] supervisord 编排 Xvfb→openbox→x11vnc→noVNC→LobsterAI 的启动顺序;tini 作 PID1 转发信号。
- [ ] `stopsignal=TERM` + `stopwaitsecs` 确保停机时 SQLite 刷盘(见 [05 分册 C2](05-代码与配置改造点.md))。

## B4 · noVNC 接入 + 冒烟 🟢 ⏱ 0.5 天
- [ ] `docker build` 成功;`docker run --shm-size=1g -p 6080:6080 -v vol:/home/lobster` 后浏览器访问 `:6080/vnc.html` 见界面。
- [ ] 重跑 A3/A4:容器内登录 + 跑通一次对话。

## B5 · 单卷持久化验证 🟢 ⏱ 0.5 天
- [ ] `docker stop --time 30 && docker rm`(**保留卷**)→ 同镜像同卷重启 → 状态全在、免登录(等价 A5,但在容器内)。

## B6 · 采数 + 运行手册 🟢 ⏱ 0.5 天
- [ ] `docker stats` 记录单容器空闲/活跃内存峰值。
- [ ] 产出《单容器运行手册》(build/run/stop/backup 命令)。

### ✅ MVP 出口
- [ ] `docker run` 一条命令得到可用实例,浏览器可操作、可登录、能对话,重启状态不丢,内存足迹已知。

# 🟢 阶段 C · 多实例 + 入口（V1）⏱ 3–5 天

> 参照 [04](04-编排与规模化运行.md)/[05](05-代码与配置改造点.md)。技术风险低,是标准编排。

## C1 · 多实例编排 🟢
- [ ] `gen-compose.sh` 生成 5 实例,每实例独立卷 `lobster_home_NN` + 独立 `env_file`(独立 Provider Key)。
- [ ] `docker compose up -d` 全部起来。

## C2 · 隔离验证 🟢
- [ ] 在 A 实例加 Skill/传文件,确认**不影响** B 实例。
- [ ] 确认各实例 SQLite/OpenClaw 状态互不干扰。

## C3 · 反向代理 + 子域名 + TLS 🟢
- [ ] Traefik/Caddy 按子域名路由到各容器 `6080`;签发 TLS。
- [ ] 验证 WebSocket(noVNC)经反代正常。

## C4 · 入口认证 🟢
- [ ] Basic Auth / SSO 中间件(noVNC 本身无鉴权,入口必须挡住)。

## C5 · 资源与日志上限 🟢
- [ ] 每容器 `memory`/`cpus` 上限(初值 4g,后续收敛)。
- [ ] Docker 日志 `max-size/max-file` 上限。

## C6 · 合规基线 + 稳定性 🟢
- [ ] 关闭遥测 + 自动更新([05 分册 B1/B2](05-代码与配置改造点.md))。
- [ ] 批量优雅停机验证无 DB 损坏。

### ✅ V1 出口
- [ ] 5 实例经认证 HTTPS 子域名访问、互不串扰、≥24h 稳定、遥测/更新已关。
- [ ] **决策点**:账号拓扑(此版可暂共享,V2 前须定为每实例独立)。

## D6 · 生命周期脚本 🟢
