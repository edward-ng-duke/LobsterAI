# 00 · 版本路线图与里程碑（先读这份）

> 本文是整个改造的**主线**。它不是简单地把工作切成"MVP/V1/V2/V3 四个盒子",而是**按这个项目真实的难度与风险分布来排期**——把风险最高、最不确定的部分前置、拆细、单独验证,把低风险的重复性工作放到后面。
>
> 详细的"怎么做"在 02–07 分册;**任务级的拆解、命令、退出标准、估时**在 [08 分册](08-实施路线图与验收清单.md)。本文讲**难度在哪、为什么这么排、每个阶段是什么**。

---

## 1. 我的判断：难度到底在哪（真实风险分布）

先说结论,再谈排期。基于对源码的侦察,这个项目的技术风险**严重前置**,集中在"让**单个**容器从零到完全可用"这一段,而不是"扩到 50"。

```
风险热度（技术不确定性 × 影响）：

  阶段A 可行性尖刺  ████████████████████  ~60–70% 的技术风险在这里
  阶段B 镜像固化    ██████                中低（工程活,路径成熟）
  阶段C 多实例+入口 ████                  低（标准反代/编排）
  阶段D 规模化到50  ████████              中（容量不确定 + 运维量）
  阶段E 平台化      ███                   低-中（渐进,可选）
  ── 合规/账号 ──   ████████████          非技术,但可能是硬阻断（贯穿）
```

**为什么风险前置在阶段 A？** 因为有几件事**现在没人验证过**,且任何一件失败都会推翻整体方案或逼出代码改造:

| 未验证的硬骨头 | 为什么风险高 | 依据 |
|---|---|---|
| **Linux 能否构建出可运行产物** | 仓库主要在 mac/win 测试,`README` 只写了这两个平台;Linux 版 OpenClaw runtime 要联网克隆、编译原生模块、装内部 registry 插件,任一环节可能翻车 | `package.json:104-105,123`、`README` 平台声明 |
| **无头（Xvfb）下能否正常渲染** | Electron 无 headless 模式,渲染进程必须有显示服务器;软件渲染 + no-sandbox 在容器里能否稳定起来是未知 | `src/main/main.ts:1206-1216`（无 headless） |
| **⚠️ 容器里能否完成登录** | 登录用 `shell.openExternal()` 打开**外部浏览器**做 OAuth,再重定向回 `127.0.0.1` 本地回调服务。容器里默认**没有浏览器**,`xdg-open` 直接失败 → 无法登录。这是最容易被忽略的地雷 | `src/main/main.ts:4565-4604`、`authLocalCallbackServer.ts:5-7` |
| **状态能否真正持久且重启还原** | 单卷挂 HOME 理论可行,但 SQLite WAL/优雅关闭、绝对路径一致性需实测 | `sqliteStore.ts`、`coworkStore.ts:37-39` |

**相对地,低风险的是**:扩到 50 主要是"把已经跑通的东西复制 N 份 + 反代 + 资源规划",是标准的容器编排,几乎没有未知数——真正的约束是**内存**(硬件)和**合规**(业务),不是技术。

**由此得出两条与"天真计划"不同的判断,直接影响排期:**

1. **先在裸 Linux 机器上用 `xvfb-run` 验证"应用能无头跑通",再谈容器化。** 不要把"应用能不能无头工作"和"我的 Dockerfile 对不对"两个变量搅在一起——那样出问题时你分不清是应用问题还是打包问题。所以阶段 A 的多数尖刺**不需要 Dockerfile**,在一台普通 Linux VM 上就能做,又快又好定位。
2. **登录流程是需要专门攻克的地雷,不是一个勾选框。** 它几乎肯定要求"在镜像里装一个浏览器 + 设为默认",甚至可能要小改代码。必须**单独立一个尖刺**先打通,否则你会做出一个"能跑但没人能登录"的容器。

> 这两条就是我对"别一口吃胖"的具体化:**把 MVP 再拆成"先验证可行性(阶段A)、再固化成镜像(阶段B)"两段**,而不是笼统地"做个 MVP"。

## 2. 版本与阶段总览

我保留你熟悉的 **MVP → V1 → V2 → V3** 版本名作为里程碑,但把 **MVP 内部按风险拆成阶段 A + B**(因为最难,最该拆细):

| 里程碑 | 阶段 | 定位 | 实例数 | 风险 | 预估 |
|---|---|---|---|---|---|
| —— | **M0** | 前置决策 | — | —— | 0.5–1 天 |
| **MVP** | **A 可行性尖刺** | 在裸机上逐个证明"能行"(A1–A5) | 1 | 🔴 高 | 3–7 天 |
| **MVP** | **B 镜像固化** | 把跑通的东西固化成可复现镜像(B1–B6) | 1 | 🟡 中低 | 2–4 天 |
| **V1** | **C 多实例+入口** | 3–10 实例 + 反代 + 认证(C1–C6) | 3–10 | 🟢 低 | 3–5 天 |
| **V2** | **D 规模化到 50** | 生产就绪:容量/监控/备份/合规(D1–D6) | 50 | 🟡 中 | 1–2 周 |
| **V3** | **E 平台化** | 弹性/自助/密度/安全,渐进(E1–…) | 50+ | 🟢 低-中 | 持续 |

**每个里程碑都能停靠**:到 MVP 你就有一个可自用的容器化 Lobster;到 V1 能小范围对外;到 V2 能规模化运营;V3 是锦上添花。

```
M0 ─▶ A1─A2─A3─A4─A5 ─▶ B1…B6 ─▶ C1…C6 ─▶ D1…D6 ─▶ E…
决策   └──── MVP ────┘    V1        V2       V3
       裸机验证可行  固化镜像  多实例   50规模   平台化
```

## 3. 能力演进矩阵

| 能力维度 | MVP(A+B) | V1(C) | V2(D) | V3(E) |
|---|---|---|---|---|
| 实例数量 | 1 | 3–10 | 50 | 50+ / 多机 |

## 4. 前置决策（M0，进入阶段 A 前拍板）

1. **GUI 访问方式**（最关键）：默认 **Xvfb + noVNC**。若真实用途是**无人值守 IM 机器人 / 纯 API**(不需要图形界面),那是另一条分支([01 方案 C](01-可行性分析与架构设计.md)),阶段 A/B 做法不同——先确认。
2. **登录/账号方案**：确认能拿到可登录的 Youdao 账号用于阶段 A3;并预判"同账号多开"策略(见 [06](06-认证配额与合规风险.md))。
3. **目标架构**：x64 / arm64(原生模块不可跨架构)。
4. **后端依赖**：Youdao 云端 还是 自建/私有模型服务(决定 D5 合规工作量)。

## 5. 各阶段定位（任务级拆解见 [08](08-实施路线图与验收清单.md)）

### 🔴 阶段 A · 可行性尖刺（MVP 上半场,风险最高,3–7 天）
**目的**:用最便宜的方式(多数在裸 Linux VM 上,不写 Dockerfile)逐个证明核心假设,任一失败尽早暴露。**按风险从高到低排序**:

- **A1 Linux 构建出可运行产物** — `dist:linux` 能完整跑通,产出 `release/linux-unpacked/LobsterAI` + `resources/cfmind`。
- **A2 无头渲染** — `xvfb-run ./LobsterAI --no-sandbox` 界面能起、不崩、OpenClaw 网关健康。
- **A3 ⚠️ 登录闭环** — 在裸机装浏览器,验证 `shell.openExternal` → OAuth → `127.0.0.1` 回调 → token 落库的完整链路;若不通,评估"预置 token / 改造登录"降级方案。
- **A4 真实 agent 回合** — 配 provider key,跑通一次完整 LLM 对话。
- **A5 状态持久化闭环** — 加 Skill/传文件/写工作目录,重启进程指向同一 HOME,状态还原。

**退出即 Go/No-Go**:五个尖刺全绿 → 方案技术可行,进入固化。任一红灯 → 先解决或调整方案(甚至回到方案 C)。

### 🟡 阶段 B · 镜像固化（MVP 下半场,2–4 天）
把 A 阶段手工跑通的东西固化成**一条命令可复现**的镜像:多阶段 Dockerfile、Xvfb+x11vnc+noVNC+浏览器+supervisor、单卷挂 HOME、优雅停机、健康检查、运行手册。见 [02](02-镜像构建步骤.md)/[03](03-状态持久化与卷映射.md)。
**退出**:`docker run` 得到可用实例,浏览器操作,重启状态不丢,内存足迹已采数。

### 🟢 阶段 C · 多实例 + 入口（V1,3–5 天）
compose 生成器、每实例独立卷/凭据、Traefik 反代 + 子域名 + TLS + 认证、资源上限、隔离验证、关遥测/更新。见 [04](04-编排与规模化运行.md)/[05](05-代码与配置改造点.md)。
**退出**:5 实例经认证 HTTPS 访问、互不串扰、≥24h 稳定。

### 🟡 阶段 D · 规模化到 50（V2,1–2 周）
服务器选型 + 内核调参、错峰启动、压测收敛资源上限、监控告警、自动备份 + 恢复演练、合规落地(独立账号/Key/IP/ToS/加密)、生命周期脚本、必要时多机分摊。见 [06](06-认证配额与合规风险.md)/[07](07-资源规划与容量测算.md)。
**退出**:50 实例 ≥72h 稳定,监控/备份/合规齐备。

### 🟣 阶段 E · 平台化（V3,渐进,子项独立）
K8s、空闲休眠/按需唤醒、自助开通、自动伸缩、密钥管理器 + 强隔离、用量计量。无单一退出标准,按投入产出择优推进。

## 6. 依赖关系与关键路径

- **强顺序**:A1→A2→A3→A4→A5 必须逐个通过(后者依赖前者的产物/环境);A 全绿才进 B;B 通了才进 C;C 通了才进 D。
- **可并行**:阶段 D 里的"合规确认(D5)"和"ToS/账号审阅"属外部沟通,**建议在 M0 就并行启动**,别等到 D 才发现账号策略不允许(那会推翻很多)。硬件采购(D1)同理提前。
- **关键路径**:**A1(Linux 构建)** 和 **A3(登录)** 是最可能拖长/翻车的两个节点,把最好的人先压在这两处。

---

## 7. 推进原则

1. **风险先行,不按"大小"排。** 先做最可能失败的 A1/A3,而不是先做"看起来是起点"的 Dockerfile。早失败 = 省钱。
2. **A 阶段允许脏、允许手工、允许一次性脚本。** 目的是"证明能行",不是"做得漂亮";漂亮留给 B。
3. **不跳阶段。** 每个阶段的退出标准就是护栏;上一阶段没绿,下一阶段的工作大概率白做。
4. **每阶段产出可交付物**(手册/报告/脚本),这样你随时能停下来"稍微改改就用"。
5. **合规与硬件提前并行**,别让它们成为终点前的意外关键路径。
