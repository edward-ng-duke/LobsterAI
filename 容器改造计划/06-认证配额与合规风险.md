# 06 · 认证、配额与合规风险（非技术，但可能是最大阻碍）

> 本文的风险**不是代码问题**，而是账号/配额/后端策略/合规问题。技术上 50 实例能跑通，但能否**合规且稳定**地长期运行，取决于这里。**上线前务必先确认。**

---

## 1. 认证与账号模型（现状）

- LobsterAI 通过 Youdao 云端 OAuth 登录（`lobsterai-server.youdao.com`），access/refresh token 存本地 SQLite `kv` 表。
  - 依据：`src/main/main.ts:4565-4646, 3475-3484`、`src/main/libs/endpoints.ts:29-32`。
- **无**硬件绑定、机器指纹、客户端侧"每账号并发/设备数上限"。安装 UUID 为随机生成，非硬件派生。
  - 依据：`src/renderer/services/installationId.ts:27`；全仓库无 license/activation/hwid 绑定逻辑。
- 配额按**账号**（非按设备）：登录后拉取 `api/user/quota`，媒体生成受订阅/积分门控。
  - 依据：`src/main/authQuota.ts:33-38`、`src/main/main.ts:4683-4699`。

**含义**：客户端**不**阻止你跑 50 个实例；但服务端是否允许、如何计费/限流，属于**未知的服务端策略**，必须与 Youdao/服务方确认。

## 2. 关键风险清单

| # | 风险 | 触发条件 | 影响 | 缓解 |
|---|---|---|---|---|
| R1 | **同账号并发是否被允许（未知）** | 50 实例共用一个 Youdao 账号 | 可能被判异常、封号或强制单会话 | **每实例独立账号**；或先向服务方书面确认并发策略 |
| R2 | **共享配额被瞬间耗尽** | 多实例共用一个账号/一套 Key | 一个实例把额度用光，其余全部被限 | 每实例独立账号 + 独立 Provider Key；对高消耗操作限流 |
| R3 | **同 IP 高频被风控/限流** | 50 实例 NAT 出同一公网 IP，登录/刷新/遥测/更新/调用叠加 | 后端按 IP 限流甚至封 IP，波及全部实例 | 关遥测与自动更新（05 分册 B1/B2）；分摊出口 IP / 加代理池；错峰 |
| R4 | **刷新 token 互相踢下线** | 多实例共享同一 refresh token 且后端滚动失效 | 一个实例刷新导致另一个 401 | 每实例独立登录、独立 token；勿跨实例复制凭据 |
| R5 | **遥测放大 50×** | 未关 `usageAnalyticsEnabled` | `rlogs.youdao.com` 收到 50× 事件，疑似 bot | 关闭使用分析（05 分册 B1） |
| R6 | **更新检查风暴** | 未关自动更新，50 实例同时启动/周期检查 | `api-overmind.youdao.com` 峰值，触发限流 | 禁用/弱化自动更新（05 分册 B2），由你统一升级镜像 |
| R7 | **ToS / 使用条款** | 大规模多开是否违反服务条款 | 账号/IP 处置、法律风险 | **上线前做条款审阅**；必要时走商务/企业授权 |
| R8 | **明文凭据、攻击面 ×50** | token/Provider Key 明文存 SQLite | 任一容器被攻破即泄露该实例全部凭据 | 卷加密 + 权限收紧；每实例独立最小权限 Key；密钥轮换；容器最小权限（非 root，已做） |

依据补充：R3/R6 的端点见 `endpoints.ts:39-43`；R5 见 `logReporter.ts`；429 限流错误已被识别 `src/common/coworkErrorClassify.ts:18`；明文存储见 `sqliteStore.ts` kv/cowork_config。

## 3. 强烈建议的账号/凭据拓扑

> **默认按"每实例完全独立"设计**，把共享带来的耦合风险降到最低：

- **每实例独立 Youdao 账号**（若用 Youdao 云端能力）→ 规避 R1/R2/R4。
- **每实例独立 Provider API Key**（OpenAI/Anthropic 等，或走你自建的网关做集中计量与限流）→ 规避 R2，并获得每实例用量可见性。
- **出口 IP 分散**：若条件允许，把 50 实例分摊到多个出口 IP（多网卡/代理池/多机），或与服务方报备白名单 → 规避 R3。
- **关遥测 + 关自动更新**（05 分册）→ 降 R3/R5/R6。
- **凭据只在运行时经 `env_file`/secret 注入，卷加密存储** → 降 R8。

如果这些实例其实用的是**你自建/私有的模型服务**（不依赖 Youdao 云端配额），则 R1/R2/R7 大幅减轻——此时主要注意 R3/R8 与你自己网关的限流。请据实际后端选择。

## 4. 上线前的"确认清单"（建议逐条落实）

- [ ] 与 Youdao/服务方确认：**同账号并发实例**是否允许、上限、计费方式。
- [ ] 审阅**服务条款**，明确大规模托管是否合规；必要时申请**企业/商用授权**。
- [ ] 决定账号拓扑：**每实例独立账号**还是共享（默认独立）。
- [ ] 决定 Provider Key 拓扑：独立 Key / 经自建网关集中管理。
- [ ] 关闭遥测与自动更新（或改默认值）。
- [ ] 规划出口 IP 与限流策略，避免同 IP 风控。
- [ ] 制定凭据存储与轮换、卷加密、容器最小权限方案。
- [ ] 明确数据合规：50 个用户的数据隔离、留存、删除、导出责任边界。

---

## 5. 一句话结论

> **技术能跑，合规要谈。** 50 实例在技术上没有账号/授权层面的硬阻断（无设备绑定、无客户端并发限制），但**同账号并发、配额、同 IP 限流、服务条款**是真实的运营风险。按"每实例独立账号 + 独立 Key + 关遥测/更新 + 分散 IP + 卷加密"设计，并在上线前与服务方确认条款，即可把风险控制在可接受范围。这些属于**业务/运营决策**，请在投入规模化前先拍板。
