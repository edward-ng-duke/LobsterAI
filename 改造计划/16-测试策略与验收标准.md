# 测试策略与验收标准

> 本文档是 LobsterAI「桌面单机 → 多租户 SaaS Web」改造计划的**全局验收/测试口径统一收口**。它面向 QA/测试负责人、后端与前端工程师、SRE、安全负责人与发布门禁评审人。各专项文档（`03`~`15`）的「验收标准」章节回引本文的测试方法、准入门与用例集编号；本文不重复各章的业务实现细节，只定义「怎么测、测到什么程度算过、每个阶段门要求哪些测试全绿」。
>
> 一句话立场：**这是一个多租户、跑不可信代码、强流式、要保证 schema 迁移正确且可选导入历史数据、要按量计费的平台**。因此测试的第一性原理是——**把「隔离不破、流式不丢、schema 不漂 / 导入零差、计费不错、沙箱不逃」这五条红线做成自动化、可重复、进发布门禁的硬门**，而不是靠人工点 `electron:dev` 走查。
>
> 权威边界：安全渗透用例编号（`PEN-*`）的**定义权威在 `14-安全合规与多租户隔离.md` §12**，本文只负责把它们纳入测试计划与阶段门；配额/计费用例的**结算口径权威在 `09-模型代理与计费.md`**；R-DATA-01 的**方案权威在 `06-数据模型迁移.md` §6.2**，且按附录 C §8 拆为 **V3 SaaS schema 迁移校验** 与 **V5 桌面存量导入双读（启用导入工具时）**；阶段门与工作量的**排期权威在 `17-分阶段路线图与工作量估算.md`**，风险↔门绑定在 `18-风险登记册.md` §4。跨文档决策（阶段门命名、契约事实源、RLS/计费/迁移口径）与**阶段门唯一对照表的总权威在 `附录C-决策基线与接口契约总纲.md`（D1–D16 / §8）**，本文已就其口径就地对齐。

### 0.2 本文要解决什么

1. 把现有 Vitest 单测资产**平移复用**（`main.ts` 抽库后行为不变靠它守，见 `04` §3），并向上补齐集成/契约/E2E/安全/迁移/性能层。
2. 用 **Playwright E2E 替代手动 `electron:dev` 走查**——SaaS 目标是浏览器端，手动走查不可重复、不进门禁。
3. 定义五类「必过」专项测试：**契约测试、跨租户越权、迁移校验/桌面导入双读、流式补发/断连恢复、沙箱安全**，外加负载/性能/冷启动压测基线。
4. 给出每阶段 **Definition of Done（DoD）** 与准入门，并与 `18` §4 / `17` 的阶段门逐条对齐。
5. 定义测试数据与环境分层、覆盖率目标。

| 层 | 只负责 | 不负责（交给谁） |

## 3. 集成测试：service ↔ 真实基础设施

### 3.1 环境（Testcontainers）

集成测试起**真实依赖容器**，杜绝 mock 掩盖 RLS/锁/队列真实行为：

```mermaid
flowchart LR
  T["Vitest 集成用例"] --> PG[("Postgres + RLS<br/>Testcontainer")]
  T --> RD[("Redis<br/>Testcontainer")]
  T --> S3[("MinIO(S3)<br/>Testcontainer")]
  T --> BQ["BullMQ(基于 Redis)"]
  PG -.每用例 SET LOCAL app.tenant_id.-> RLS[RLS 策略生效验证]
```

- 每个套件用 Prisma migrate 建库（含 RLS 迁移 SQL，`06` §5.3 的 RLS raw SQL 模板），应用连接用**非 superuser、非 BYPASSRLS** 角色（否则测不出 RLS，见 `14` §3.3 PEN-ISO-4）。
- 每用例事务隔离 + 显式 `SET LOCAL app.tenant_id`，验证连接池不串上下文。

### 3.2 RLS 与租户过滤集成用例（对应越权红线的下层证明）

| 用例 | 断言 |
|---|---|
| A 租户 session，切 `SET LOCAL app.tenant_id=B` 查询 | 返回 0 行（RLS 拒绝） |
| Prisma tenant extension 自动注入 `where tenant_id` | 缺省 tenant 的裸查询被拦或空结果 |
| **关闭应用层 extension**（模拟 bug），仅靠 RLS | 仍 0 行/报错（双层任一失效不越权，`14` §3.1） |
| 连接池并发多租户请求 | 无 `SET LOCAL` 串上下文（PEN-ISO-3 的自动化版） |

### 3.3 会话串行化与消息一致性（`04` §7）

- **会话级分布式锁**：两个并发「发消息」请求，第二个得 `409 SESSION_BUSY`（或正确排队），不产生 `sequence` 冲突。
- **看门狗续期**：模拟长生成（> 锁 TTL）时锁不提前释放，无并发闯入。
- **`replaceConversationMessages` 事务性**：并发读分页不读到半更新状态。
- **sequence 生成竞态**：持锁写路径使用 `cowork_sessions.next_message_sequence` + `SELECT ... FOR UPDATE` 分配，压力下无重号/乱序；`beforeMessageId` 插入触发间隙不足时只重排本 session。

### 3.4 流式广播（Redis pub/sub + Stream）集成

- `XADD` + `PUBLISH` 到 `stream:{tenantId}:{sessionId}`，多个 Gateway 订阅者都能收到（多端同看）。
- **补发**：`subscribe sinceSeq` 触发 `XRANGE` 回放 `> sinceSeq`；`sinceSeq`/`seq` 是 Redis Stream id 或等价单调游标字符串，断连窗口内事件不丢（详见 §6）。
- 幂等：重复 `seq` 帧被客户端/Gateway 去重。

### 3.5 对象存储与签名 URL（MinIO）

- 上传只签 PUT、读只签 GET，**拒签 DELETE/List**（`14` §5.4）。
- 签名 URL 过期返回 403；篡改 `tenantId`、`previewSessionId`、`artifactId`、`path`、`method` 或 `kid` 任一签名输入均返回 403（PEN-CONTENT-3/4 的集成前置版）。
- S3 key 强制 `tenants/{tid}/...` 服务端拼接，拒绝客户端传完整 key（PEN-ISO-6 前置版）。

### 3.6 BullMQ 任务租户上下文

- job payload 带 `tenant_id`，worker 内重建 RLS 上下文；worker 内越权查询受 RLS 约束（PEN-ISO-8 前置版）。
- worker 幂等（job id = 业务键），重复触发不产生重复副作用。

### 3.7 Sandbox 镜像与状态卷集成

原容器改造计划中的 A1-A5 尖刺不再作为独立路线图，但其验证项应进入 V1 的 Sandbox 测试：

| 用例 | 断言 |
|---|---|
| Linux runtime 构建 | `lobster-openclaw-runtime` 在目标架构上可复现构建；OpenClaw bundle、插件、原生模块完整；容器启动到 gateway ready |
| 镜像内容检查 | 生产 runtime 镜像不含 Electron Renderer、Xvfb、x11vnc/noVNC、Chromium UI、真实密钥或租户数据 |
| state/PVC 映射 | `openclaw.json`、gateway token、workspace、memory、logs/cache 按 `07` §5.5 归属落位；没有未映射的隐式 `HOME` 依赖 |
| 优雅停机 | Pod 收到 SIGTERM 后 gateway 可关闭、state 刷盘、租约释放或等待 reconcile；不会留下半写文件 |
| 资源采样 | 空闲、首 turn、长响应、MCP/Skills 执行均记录 CPU/RSS/重启次数，为 `07` resourceClass 和 `15` HPA/水位提供实测输入 |
| 调试 noVNC | 生产环境无 noVNC Ingress/Service；若测试启用调试镜像，必须通过 `14` PEN-SBX-4 |

## 4. 契约测试（REST/WS + 桥双跑一致性）

契约测试是「防漂移」的核心——它保证**前端桥、后端 API、`libs/shared/contracts` 三者永不脱节**，是渐进式迁移与双跑（`03` §10.2）的支柱。现有 `src/shared/*/constants.ts` 是 PR-1 抽取输入，不是目标态字段契约源。

### 4.1 三类契约

```mermaid
flowchart TB
  SC["libs/shared/contracts<br/>(OpenAPI/AsyncAPI/Zod 契约事实源)"]
  LEG["src/shared/*/constants.ts<br/>(现状 as const 抽取输入)"]
  LEG --> SC
  SC --> R["① REST 契约<br/>路径/方法/请求体/响应信封/错误结构"]
  SC --> W["② WS 帧契约<br/>ClientFrame/ServerFrame/StreamEventType"]
  SC --> B["③ 桥双跑一致性<br/>ElectronBridge 接口: Electron桥 == Web桥"]
  R --> A["附录A IPC↔接口映射逐条覆盖"]
  W --> A
```

## 6. 流式补发 / 断连恢复测试（专项，红线）

流式是本平台体验核心，弱网/多副本下**不能丢帧、不能重复、不能串会话**。对应风险 `R-STREAM-01`（`18`）。

### 6.1 测试矩阵

```mermaid
sequenceDiagram
  participant C as 测试 WS Client
  participant GW as Gateway
  participant RS as Redis(Stream+PubSub)
  Note over C,RS: 场景A 断连补发
  C->>GW: subscribe {sid, sinceSeq="1719900000000-0"}
  GW->>RS: XRANGE > "1719900000000-0"
  RS-->>GW: 事件 "1719900000001-0".."1719900000020-0"
```

|---|---|---|

## 8. 迁移校验与桌面导入双读（R-DATA-01 分门）

> 方案权威在 `06-数据模型迁移.md` §6.2；对应风险 `R-DATA-01`（`18`）。**附录 C §8 已把 R-DATA-01 拆成两条独立门**：**(a) SaaS schema 迁移校验（Prisma migration 正确性/幂等）= `V3` 早硬门**；**(b) 桌面存量 SQLite → Postgres 导入双读 = `V5` 可选门**。注意口径：**SaaS 是全新自建后端，不做在线数据迁移**（`05` §9 注）；本节的「双读对账」主要针对 (b) 的可选导入工具正确性，(a) 的 schema 迁移校验随 `V3` 集成测试执行。

### 8.1 V3 SaaS schema 迁移校验（硬门）

V3 的 R-DATA-01 不跑桌面 SQLite 导入双读；执行者只需要证明 SaaS Postgres schema 的 migration 可重复、可升级、RLS/tenant scope 正确：

- 从空库执行全量 migration + seed，表/索引/外键/RLS policy 与 `06` schema 清单一致。
- 从上一版本 schema 执行增量 migration，重复执行不产生脏状态；失败后按 `15` 的数据库回滚/恢复 Runbook 可回到迁移前快照。
- 每个 tenant-scoped 表强制 `tenant_id`/workspace 归属约束；RLS bypass negative cases 必须失败。
- `Prisma migrate`/DDL、RLS policy、BullMQ 相关表、对象存储元数据引用能被集成测试覆盖；任何 DDL 漂移或 RLS 漏洞阻断 V3。

### 8.2 桌面存量导入双读方法（dry-run + 对账）

```mermaid
flowchart LR
  SRC[(桌面 SQLite 快照)] --> IMP[导入器 dry-run]
  IMP --> DST[(Postgres 目标租户)]
  SRC --> RA[读源: 行数/关键字段/引用]
  DST --> RB[读目标: 行数/关键字段/引用(重映射后)]
  RA & RB --> DIFF{对账}
  DIFF -- 差异>0 --> FAIL[阻断: 定位映射/引用断链]
  DIFF -- 0 差异 --> PASS[通过 0 差异门]
```

### 8.3 桌面导入校验维度与判定（0 差异门）

| 维度 | 校验 | 门 |
|---|---|---|
| 行数对账 | 每张业务表源↔目标行数一致（约 19-20 张：`agents`/`cowork_sessions`/`cowork_messages`/`mcp_servers`/… 见 `06`） | 差异必须为 **0** |
| 引用完整性 | 消息→会话、记忆来源→记忆、子代理消息→run 等引用**重映射后**仍指向存在父行（`06` §6.2） | 无断链 |
| id 重映射 | 老 id→新 UUID 映射唯一、引用列同步改写 | 无冲突、无遗漏 |
| 内容抽样 | 抽样消息内容/元数据/JSON 字段字节级一致 | 抽样 0 差异 |
| 幂等 | 同一快照重复导入 | 不产生重复（幂等） |
| 特殊表口径 | 桌面导出中的 `scheduled_tasks`/`scheduled_task_runs` 为**历史遗留表**；目标 SaaS 权威是 `11` 定义的 Postgres `scheduled_tasks`/`scheduled_task_runs` + BullMQ 调度，沙箱内 OpenClaw cron 禁用/不下发。`scheduled_task_meta` 的 `origin`/`binding` 并入目标任务主表，不保留独立目标表 | 按 `11` 口径校验迁移路径；不得把 gateway cron 或遗留 SQLite 表校验为长期活跃权威 |

**准入门**：SaaS schema 迁移校验为 **`V3` 硬门**（随 `V3` 集成测试执行，见附录 C §8）；桌面存量导入双读 **0 差异**（行数 + 引用 + 抽样）为 **`V5` 可选门**，仅在启用导入工具时必过（附录 C §8 / `18` §4）。禁止把 V5 桌面导入双读前移为 V3 硬门。

## 9. 沙箱安全测试（专项，红线）

> 用例权威在 `14` §12.2（SANDBOX/NET）；对应风险 `R-ISO-01`（`18`，逃逸测试 100% 拦截是 PoC→单租户阶段门）。这些测试需**真实集群 + 探针 Pod**。

### 9.1 沙箱与网络必过用例

| 编号（→14 §12） | 测试动作 | 通过判定 |

| PEN-NET-1 | Pod 内 `curl 169.254.169.254`（IMDS） | egress 代理 + NetworkPolicy 拒绝（`14` §4.1） |
| PEN-NET-2 | 访问 RFC1918 内网 / 另一 Pod IP | 拒绝（默认拒绝入站/横向） |

### 9.2 执行方式

- 用**受限探针 Pod**（在 sandbox namespace 起一个带攻击脚本的容器）跑 PEN-SBX/PEN-NET，断言全部被拦。
- gVisor/Kata 逃逸 PoC 与运行时升级联动（`14` §8 供应链、`15` CI/CD 镜像扫描）。
- 供应链门控（PEN-SUP-*）在 SkillModule/McpModule 集成测试 + E2E J8 覆盖；公开分享 abuse（PEN-ABUSE-4..8）在分享服务集成测试 + 浏览器 CSP E2E 覆盖。

## 10. 负载/性能与冷启动压测基线

> 结算/配额熔断口径见 `09`，编排并发/预热容量见 `07`，容量/SLO/SLA 与告警见 `15`。对应风险 `R-PERF-01`/`R-COST-01`（`18`）。本节定**压测场景与测量方法**；**性能阈值（首帧 p95、冷启 p50/p95 等）统一以 `17` V4 权威表为唯一数值来源**（附录 C §8 V4 门），本节不再自给可能与之量级冲突的数值。

### 10.1 压测场景与基线指标（示例目标，实测按 `17`/`15` 调校）

| 场景 | 工具 | 输入规模/向量 · 关键指标 | 阈值 / pass-fail（数值权威见 `17` V4） |
|---|---|---|---|
| REST QPS | k6 | 目标 QPS 阶梯加压；会话/消息 CRUD p95 延迟、错误率 | p95 延迟与错误率阈值见 `17` V4；超阈 = fail |
| WS 流式吞吐 | 自研流式 client | N 并发连接 × 增量帧速率；端到端首帧延迟、稳定吞吐 | 首帧 p95 阈值见 `17` V4（统一消除跨文档量级冲突）；吞吐无堆积 = pass |
| 并发会话 | k6 + 流式 client | 加压至套餐并发上限；活跃流式会话数、Redis 广播延迟 | 达套餐上限不劣化（`07` §6）；广播延迟阈值见 `17` V4 |
| **冷启动**（沙箱 Pod） | 编排压测 | 批量新会话 claim；claim→gateway ready 分位延迟 | p50/p95/p99 与预热命中率阈值见 `17` V4（§10.2） |
| 会话锁竞争 | 并发发消息 | 同会话 M 并发发消息；`409` 比例、锁等待 | 串行正确、无 sequence 冲突（§3.3，pass/fail 二值） |
| 大附件 | k6 上传 | 边界附件（安全线上下）；拦截行为、S3 直传成功率 | 超安全线按 `04` §6.4 / `08` 拦截、直传不撑爆（pass/fail） |
| 出站/计费计量 | egress 审计 | 已知字节量出站；`egress_bytes{tenant}` 准确度 | 与计费对账一致（`09`，允许误差阈值见 `17` V4） |

### 10.2 冷启动专项

冷启动是 SaaS 体验短板（桌面无此问题）：

```mermaid
flowchart LR
  REQ[新会话请求] --> CLAIM{预热容量命中?}
  CLAIM -- 命中 --> READY[毫秒级复用净室 Pod]
  CLAIM -- 未命中 --> BOOT[冷启: 调度+拉镜像+gateway ready]
  READY & BOOT --> MEASURE[度量: p50/p95/p99 冷启延迟]
```

- 压测**预热容量命中率**与**冷启分位延迟**（`07` 预热容量机制），基线先纳入 `15` §7.3 SLO，并作为 §7.4 SLA 违约计量输入；不得只写对外 SLA 而缺内部 SLI/SLO 证据。
- 冷启退化告警（p95 超 `17` V4 权威阈值）接入 `15` 监控。

### 10.3 计费正确性压测（对齐 `09`）

- 并发调用下扣费**不多扣不漏扣**（幂等 + 事务）；超配额触发降级（拒新写/提示升级），**不影响读取与已有会话**（`09` 降级策略，E2E J10）。
- 上游模型/ASR provider 故障切换演练（`R-VENDOR-01`）：切换后计费归属正确、无重复扣费；ASR 失败/取消仍 release 或写 0 credits usage/audit。

| 环境 | 用途 | 依赖 | 数据 |

### 12.2 测试数据集（多租户为一等公民）

- **至少两个租户**（Tenant A / Tenant B）+ 每租户多用户多角色（owner/admin/member/viewer），供越权（§7）与 RBAC 用例。
- 覆盖数据形态：长会话（分页边界）、大量消息（`sequence` 压力）、带附件会话、含 artifact（HTML/SVG/文档）、含记忆/dreaming 数据、含 MCP/技能配置（含 secref 字段）、含定时任务（历史遗留表 + gateway cron meta，`11` 口径）。

**Seed 数据要求**：

- 每次运行必须从固定 seed 生成确定性 ID（或输出 `id-map.json`），同一 fixture 在本地与 CI 的期望结果一致。
- 基础 seed 至少包含：`tenant_a` / `tenant_b` 两个活跃租户、`tenant_suspended` 一个禁用租户、每租户 owner/admin/member/viewer 四类角色、跨租户同名 session/agent/file/artifact/MCP/skill/task。
- 计费 seed 必须含四桶余额组合：只够 daily、daily 不足需跨 monthly/granted/topup、余额为 0、并发 `hold`、结算小于 hold 的正向 `settle` 回补、调用失败/未产生 usage 的 `release`、usage missing 的估算 `settle` 与后续 `refund(reason=usage_correction)` / `adjust` 修正，以及支付退款/拒付的 `refund(reason=payment_refund|chargeback, credits<0)` clawback；`credit_accounts` fixture 使用 `daily_balance/monthly_balance/granted_balance/topup_balance` 作为权威快照，`*_used` 只能作为期望响应里的派生字段。
- `GET /api/v1/billing/account` fixture 必须断言顶层兼容字段与四桶明细一致：`creditsRemaining=Σbalance`、`creditsLimit=daily.limit+monthly.limit+granted.total+topup.balance`、`creditsUsed=creditsLimit-creditsRemaining`；不得手填与 breakdown 不一致的示例数。
- 安全 seed 必须含「合法访问对照组」与「越权攻击组」成对样本，报告中按 pair 输出，防止只测失败路径而漏正常路径。

**Testcontainers 夹具规则**：

- 集成安全套件统一由 Testcontainers 起 Postgres + Redis + MinIO；Postgres 使用非 superuser、非 `BYPASSRLS` 应用角色，迁移后显式检查目标表均 `ENABLE ROW LEVEL SECURITY` + `FORCE ROW LEVEL SECURITY`。
- 每个测试 schema/Redis 前缀/S3 bucket 前缀必须带 run id；夹具初始化输出 `{ runId, tenantIds, userIds, bucketPrefix, redisPrefix }`，供断言和报告复用。
- RLS bypass negative cases 必须存在：关闭 Prisma tenant extension、尝试裸 SQL、尝试 `SET app.tenant_id` 跨租户、缺失 `SET LOCAL`、并发请求复用连接池、worker 内未设 tenant 上下文。通过判定是 0 行/报错/404，且报告写明是哪一层拦截。
- Redis/S3/MinIO 负向用例必须覆盖缺 tenant 前缀、传完整他租户 key、签名 URL 篡改、过期 URL、跨租户 Stream subscribe。

**Playwright 夹具规则**：

- Playwright 使用同一 seed 输出的账号与租户映射，不在测试中临时手工创建不可追踪数据；登录态、storage state 与 traces 以 run id 命名。
- E2E 必须覆盖正常旅程 + 越权旅程成对断言：A 租户能打开自己的 session/artifact/file；同一路径换成 B 租户资源或同租户未授权 workspace 时应 404 且 UI 不泄露名称、大小、预览缩略图。同租户角色不足且无需隐藏存在性的动作才允许 403。
- 内容安全夹具必须在浏览器内断言：iframe 不能读主应用 token、沙箱预览外链按 CSP 阻断、公开分享 noindex/下架状态可观测。

**Sandbox escape corpus 规则**：

- `sandbox_escape` 样本按 metadata 管理：`id`、`category`（privilege/network/filesystem/runtime/supply-chain）、`runtimeClass`、`command` 或脚本入口、`expectedBlock`、`requiredCapability`、`source`、`lastReviewedAt`。
- corpus 不存真实云凭证、不访问真实内网；IMDS/RFC1918/DNS rebinding 使用测试网段或本地假服务。需要真实集群的样本标记 `requiresCluster: true`，只在 Nightly/阶段门环境跑。
- 每次 gVisor/Kata/runtime 镜像升级必须跑全量 corpus；新增逃逸 CVE 时先加入 corpus，再补修复与门禁。

**计费对账夹具规则**：

- 对账输入固定为 `usage_events.jsonl` + `expected_ledger.json` + `expected_bucket_balances.json`；`expected_ledger.json` 每条分录必须带 `reason` 与 `bucketDeltas` 且 ΣbucketDeltas=`credits`；`expected_bucket_balances.json` 必须按 `daily_balance/monthly_balance/granted_balance/topup_balance` 断言 `credit_accounts` 物化快照 = 初始余额 + Σ`bucketDeltas`，不得用旧 `*_used` 字段当余额权威；断言维度包括 `hold/settle/release/refund/topup/grant/adjust` 分录、`refund` 的 `usage_correction` 正向返还与 `payment_refund/chargeback` 负向 clawback、四桶余额、同 `(tenant_id, request_id, entry_type, reason)` 幂等重放、同一 `request_id` 不同 `reason` 可共存、跨租户同 `request_id` 不碰撞。
- 并发压测必须输出 `billing-reconciliation.json`：总输入 tokens、按模型单价计算 credits、Redis 扣减、DB ledger、差异、允许误差、失败样本 requestId。
- `stream cancel`、provider timeout、usage missing、重试后迟到 usage 都必须有夹具；通过判定是不多扣、不漏扣、释放/退款/调整可追踪、四元幂等键唯一。

**CI 报告格式**：

所有安全/隔离/计费夹具套件必须产出机器可读 JSON，作为 CI artifact 上传，字段最少包含：

```jsonc
{
  "suite": "tenant-isolation",
  "runId": "20260708T120000Z-abc123",
  "contractVersion": "1.2.0",
  "gitSha": "…",
  "environment": "ci|staging|nightly",
  "summary": { "passed": 42, "failed": 0, "skipped": 1 },
  "cases": [
    {
      "id": "PEN-ISO-4",
      "fixture": "tenants/basic-cross-tenant",
      "tenant": "tenant_a",
      "targetTenant": "tenant_b",
      "expected": "0 rows or 404",
      "actual": "0 rows",
      "status": "passed",
      "evidence": ["trace.zip", "query.log"]
    }
  ],
  "reconciliation": { "ledgerDiff": "0", "bucketDiff": "0" }
}
```

- 报告 schema 失败视为测试失败；PR 页面只展示摘要，完整 trace/log/video/JSON 作为 artifact 保留。
- 阶段门评审只接受同一 run id 下的 contract diff、PEN 报告、Playwright trace、billing reconciliation、sandbox corpus 结果，避免拼凑不同环境的证据。

## 13. 测试覆盖率目标

| 层 | 覆盖率/量化目标 | 度量 |
|---|---|---|
| 单元（抽库纯逻辑） | 行/分支覆盖 **≥ 80%**（`04` §3.2 A 类逻辑更高，关键路径接近 100%） | Vitest coverage |
| 集成（RLS/锁/队列/S3 关键路径） | 关键路径 **100% 有用例**（非行覆盖，而是场景覆盖） | 用例清单核对 |
| 契约 | `附录A` 每条 IPC 通道 **100% 有映射契约用例**；桥双跑全接口 | 通道清单穷举 |
| E2E | 关键旅程 J1-J11 **按阶段 100% 覆盖**（J11 ASR 自 V5/ASR 后端落地后阻断） | 旅程清单 |
| 安全（PEN-*） | `14` §12 全部编号 **100% 有自动化/半自动用例** | 编号核对 |
| 迁移 | 双读 **0 差异**（非覆盖率，是绝对门） | 对账报告 |

补充规则：

- 覆盖率**门禁不倒退**：PR 不得使总覆盖率下降（CI 比对基线）。
- 覆盖率不是唯一目标——**红线专项（越权/沙箱/迁移/流式/计费）以「用例是否存在且通过」为准**，不用行覆盖率替代。
- 安全/沙箱用例即使少（金字塔顶），**权重最高**：任一失败阻断发布（§11 阶段门）。

## 14. CI/CD 集成（测试何时跑）

```mermaid
flowchart LR
  PR[Pull Request] --> L[eslint 改动文件]
  PR --> U[npm test: Vitest/纯逻辑单测]
  PR --> PG[当前阶段已落地门禁]
  PG --> CT[PR-1+ 契约测试 + 桥双跑]
  PG --> IT[PR-2+ 集成(Testcontainers 独立 job)]
  PG --> ISO[V3+ 越权必过集 PEN-ISO/AUTH]
  U & PG --> COV[覆盖率门禁不倒退]
  COV --> MERGE{合并}
  MERGE --> E2E[E2E(Playwright) staging]
  MERGE --> NIGHTLY[Nightly: 负载/沙箱/schema迁移/导入双读/混沌]
  E2E & NIGHTLY --> GATE[阶段门评审 §11]
```

| 触发 | 跑什么 | 阻断? |
|---|---|---|
| 每次 PR | eslint（改动文件）、`npm test`（Vitest/纯逻辑单测快环）、覆盖率不倒退 + **当前阶段已落地的阻断门禁**：PR-0 检查入口/manifest/fixture 不得 no-op；PR-1 起契约+桥双跑；PR-2 起 Testcontainers/RLS 最小集成 job；V3 起完整越权必过集与安全/隔离/计费 fixture 报告 schema。不得把尚未 scaffold 的未来套件写成绿灯占位，也不得把 PR-2+ 容器集成默默并入当前 `npm test` | 是 |
| 合并到主干 | 上述 + 当前阶段已引入的 E2E（Playwright，staging，J1-J11 按阶段启用）+ Playwright trace/video/report artifact；Playwright 未进入 PR-0/PR-1 前只允许以失败/blocked 明示缺口，不允许假绿 | 是 |
| Nightly/定期 | 负载/性能、冷启动压测、沙箱安全（探针 Pod + sandbox escape corpus）、SaaS schema 迁移校验、启用导入工具时的桌面导入双读、计费对账、混沌、完整渗透 | 阶段门前必绿 |
| 阶段门评审 | §11 各门必过测试全绿，风险清账（`18` §4） | 是（不过不放行） |

> E2E/负载/沙箱需集群环境，不进 PR 快环（保 PR 反馈 < 10min），但**进合并门与阶段门**。这与 `15` CI/CD（GitHub Actions + Helm、蓝绿/金丝雀、一键回滚）联动。
