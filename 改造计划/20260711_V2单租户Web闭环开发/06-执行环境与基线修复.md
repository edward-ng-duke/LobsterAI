## 2. 合并事故与集成纪律教训

PR #1 使用 GitHub 默认 merge commit（`3babe5c6`，第一父为旧 main `801fb837`），导致 P02 证据链（codeEvidenceSha `352d0caa`、stageEvidenceSha `2f8e33e4`）整体落在 main 的第二父一侧。P02 trusted provenance 要求证据 SHA 位于 HEAD 的 first-parent 链上，因此 `tests/db/p03-merge-evidence.test.ts` 与 `tests/db/reviewer-round2-red.test.ts` 的 live 校验在合并后的 main 上确定性失败。此外，分支末尾的 `31d09818`/`1a53e396`（contracts 产品代码）落在证据冻结之后，即使不合并也已再次失效该链。

这不是校验器误报：它按设计把"证据冻结后又有代码变动/历史被 merge 改写"视为证据过期。教训固化为 [04-风险决策](04-风险决策与阻断项.md) 的 DEC-10。

## 6. V1 阶段本地测试集群策略（DEC-12 草案）

V101～V107 需要真实集群证据。在用户提供云上集群之前，本工作站的可行路径：

- kind（Docker 内 K8s）作为 BLK-06 的测试集群载体；Calico 提供真实 NetworkPolicy 语义（V103）；本地 PVC/hostPath StorageClass + MinIO 容器（V104）；`ctr`/`docker` 导入 P03 镜像或起本地 registry 容器（BLK-05 测试侧）。

- 上述均为测试阶段基础设施（允许联网下载并固化）；生产断公网口径不变。Day-0 冻结文档由 V101 Planner 以本机实际值起草，用户对 DEC-12 有一票否决权；若用户提供真实云集群，V1 证据以云集群重跑为准。

## 7. 协调者集成纪律（自本文件起生效）

1. 主线线性：后续任务直接在 main 线性提交（或 rebase/fast-forward 合入）；禁止产生 merge commit，除非同批完成 P02 证据重冻结。
2. 每个任务落码后的集成收尾顺序固定为：产品/测试/CI 变更全部落定 → 本地 targeted/full 与 P03 live source 解析 → 推送同一 candidate 并等待 required CI/机器 artifact Green → 下载验 hash → P02 证据重冻结 → Reviewer/Tester/协调者记录 → 只读 provenance/formal checker 复验。不得先冻结 P02 再等待 CI，也不得在冻结后为修 CI 追加 product/test/workflow commit。
3. 提交批次顺序：产品/测试/CI代码 → 计划与任务文档 → local/CI 机器证据稳定 → P02 evidence-only 冻结提交 → 总表/角色记录。若后续提交推进 P03 product source，重做 P03 live/CI 与 P02 freeze；若仅总表/角色 allowlist 变化，只做只读复验。
4. 全仓命令一律使用 nvm Node 24（`PATH=~/.nvm/versions/node/v24.18.0/bin:...`），供应链命令追加 `~/.local/bin`。
