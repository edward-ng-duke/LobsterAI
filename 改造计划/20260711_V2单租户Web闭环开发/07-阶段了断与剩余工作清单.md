## 1. 审阅结论与已纠正缺口

本轮不是文字润色，而是关闭以下执行缺口：

1. **默认测试不可持续：** 主 CI 的 `test` job 在干净 checkout 直接运行 `npm test`，不会先生成 `.reports`，也不会生成 workspace 包入口指向的 ignored `dist`；它还依赖可被 path filter 跳过的 `build-main` job，并使用浅 checkout。原计划无法修复独立 CI 的 evidence mutation、workspace 包解析或 P02 first-parent 历史读取。现由 B00-3B 将 mutation tests 改为合成 fixture，由 B00-3C 在 Vitest 启动前串行清理/编译 workspace 并执行 contracts `--check`，同时让主 CI test job 独立、full-history、`npm ci` 可重放；live evidence 继续由独立 gate 验证。
2. **P02 双平台方案不可落地：** 当前 `postgres-image.json` 是单平台 schema v1，`preflight.mjs`、`run-integration.mjs`、`validate-static.mjs` 都直接读取单对象。现将 B00-4 明确为 schema v2 + 集中 selector + 两平台真实运行，而不是“多写一个 digest”。
3. **局部复核与任务签收混淆：** B00-3/B00-4 不再写局部 `APPROVED_FOR_TEST`/`PASSED`。Developer 先构建完整候选，B00-7 才执行任务级 Reviewer/Tester。
4. **P02/P03 顺序成环：** 最终顺序固定为 P03 产品与 live evidence → P02 重冻结 → P02 接受 → P03 接受。P02 只消费 P03 机器证据，不要求 P03 已协调接受；P03 最终签收则要求 P02 已接受。
5. **冻结后文档可能再次打断 evidence：** B00-6 后只允许 P02/P03 角色记录、P02 evidence 目录和权威总表的普通修改。`04`～`07`、产品、测试、CI 或脚本若再变，必须从 B00-6 重做。
6. **证据只在本机不够耐久：** P03 最终签收必须同时记录本机 live report hash 与 `saas-scaffold` CI run/artifact 引用；raw `.reports` 仍 ignored，不入库。
7. **P04 解锁口径不精确：** P03 接受后才能启动 P04 Developer；Day-0 未齐时只能完成可控 fixture/纯逻辑，真实 preflight 仍 `BLOCKED`，不得交 Tester 或勾总表。
8. **P03 外部门被误当可延期：** B00-5 拆为 5A local/CI 与 5B internal registry publish/signature。5A 可供 P02 重冻结；5B 缺 PLAT/SEC 输入时 P03 必须 BLOCKED。
9. **V2 会自我作废 V107：** V107 拆为 Phase A 开发授权与 V208 最终候选 Phase B requalification；正常 V2 提交不撤销 Phase A。
10. **契约与配置 owner 缺口：** V201/V207 分别内置 C201/C207 additive checkpoint；V205 内置 Phase 0 server configuration domains，不新增总表行。

## 3. B00 不可违反的执行不变量

1. 所有命令使用 `PATH="$HOME/.nvm/versions/node/v24.18.0/bin:$HOME/.local/bin:$PATH"`；不使用系统 Node 22。
2. 只允许一个 writer 在同一线性分支推进 B00；禁止 merge commit、并行产品提交和冻结期间 rebase。
3. 生产仍完全断公网；测试/镜像构建可联网下载并固化。不得把公网 provider、registry、OIDC、S3 或包源写成生产 fallback。
4. `npm test` 必须在没有 `.reports/`、没有 P03 本地镜像、没有预存 workspace `dist/dist-types/*.tsbuildinfo` 的干净 checkout 通过；测试前置只能使用当前 tracked source 重建输出，不得把 ignored 产物当输入。这只证明 hermetic 逻辑。
5. `npm run docker:build:check` 与 `npm run supply-chain:check` 必须在真实 Docker/live evidence 上通过；这只证明当前候选机器事实。两类证据缺一不可。
6. PostgreSQL manifest 只接受 exact tag + per-platform exact digest；Node、Docker 与所选平台必须一致，未知平台返回 `BLOCKED/2`。
7. 任一 Red 必须因目标缺陷失败；工具缺失、网络错误、旧 evidence 或语法错误不算 Red。
8. ignored `.reports`、镜像、cache、build output 和 secret 永不强制入库；提交只含源码、测试、最小合成 fixture、durable 摘要和角色记录。
9. B00-6 之后只提交 allowlist 路径；任何其它修改使 Reviewer/Tester 结论立即过期。

## 8. 失败回退与停止条件

- B00-3A mutation 仍可绕过：停在 P03 Developer，禁止重建镜像。
- B00-3B 仍依赖 `.reports`、旧 Git object 或本地 Docker：停在 P03 Developer，主 CI 不得标绿。
- B00-3C 在预存 `dist` 移除后仍无法收集套件、contracts `--check` 发生写入，或 TSC/Vite 用例仍共享主 checkout 输出：停在 P03 Developer，不以本机残留产物重试冒充 Green。
- B00-4 无法取得可验证 amd64 exact digest，或 arm64 CI 回归：P02 保持 `BLOCKED`，不使用浮动 tag/跨架构 emulation 冒充通过。
- B00-5A 任一镜像/CI/Artifact失败或 B00-5B publish/signature 非 PASS：保留失败摘要；前者不进入 evidence freeze，后者不影响已接受 P02但阻断 P03/P04。
- B00-6 任一 source/hash/first-parent/cleanup 不一致：废弃该 evidence chain，修复后从 code candidate 重新开始，不手改 JSON。
- Reviewer/Tester 任一 `CHANGES_REQUESTED`/`FAILED`：回原任务 Developer；修复若触及冻结禁止路径，P03 live evidence 与 P02 evidence 都重建。
- 任何 secret、真实 credential、tenant 数据或公网生产 fallback 出现：立即停止、删除敏感 artifact、轮换受影响凭据并登记安全事件。
- 不使用 `reset --hard`、`checkout --`、`clean`、强推或手工删除历史失败记录做“回滚”；只用新 commit 前向修复或经用户明确授权的安全回退。
