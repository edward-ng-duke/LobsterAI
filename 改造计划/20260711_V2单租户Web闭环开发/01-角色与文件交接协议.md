# 角色与文件交接协议

## 1. 角色隔离

每个任务使用独立角色上下文。Planner、Developer、Reviewer、Tester 不复用同一个 Agent；不同任务也不复用已销毁 Agent，避免上下文污染。同一任务若被退回，优先唤回该任务原 Developer；原 Agent 不可用时，新 Developer 必须先完整阅读该任务四份文件和相关 diff。

| 角色 | 可做 | 不可做 | 写入文件 |
|---|---|---|---|
| Planner | 澄清依赖、源码边界、失败测试、验收和回滚 | 写业务代码、替 Reviewer 通过 | `开发任务.md` |
| Developer | 先写失败测试，再实现；运行验证；记录 commit/diff/风险 | 自审为通过、勾总体表 | `开发记录.md` |
| Reviewer | 独立读计划、diff、测试；指出文件/行/优先级和复现方式 | 代写大块实现、跳过失败测试证据 | `审核意见.md` |
| Tester | Reviewer 通过后补独立 corner-case/回归/E2E/安全测试 | 用 Developer 自测替代独立验证 | `测试报告.md` |
| 总体协调者 | 校验依赖、四角色结论、证据和阶段门；更新总表 | 仅凭口头结论勾选 | `00-总体完成表.md` |

## 2. 文件状态机

`TODO → DEV_IN_PROGRESS → REVIEW_PENDING → CHANGES_REQUESTED → REVIEW_PENDING → TEST_PENDING → TEST_FAILED → REVIEW_PENDING/DEV_IN_PROGRESS → ACCEPTED`。

1. Planner 创建/冻结 `开发任务.md`，记录版本号和计划提交 SHA。
2. Developer 在 `开发记录.md` 记录 Red（失败测试）→ Green（最小实现）→ Refactor；每轮追加，不覆盖历史。
3. Reviewer 在 `审核意见.md` 写 `CHANGES_REQUESTED` 或 `APPROVED_FOR_TEST`。任何 P0/P1、未满足验收标准、缺失 Red 证据均必须退回。
4. Tester 只有在 `APPROVED_FOR_TEST` 后开始；新增测试属于交付的一部分。失败时写清复现并退回 Developer，修复后必须重新 Reviewer。
5. Tester 写 `PASSED` 后，协调者核对任务依赖和证据，才可在总体表勾 Developer/Reviewer/Tester/协调者四格及 Todo。

## 3. 多轮沟通格式

每轮标题统一为 `Round N · YYYY-MM-DD HH:mm · Agent/commit`。意见必须含：严重级、需求/用例 ID、文件与行、期望、实际、复现命令、需要的完成证据。Developer 逐条回复 `fixed / rejected with evidence / blocked`；不得删除历史意见。

## 4. Commit 与并行规则

- 每个任务至少一个独立 Conventional Commit；脚手架用 `chore(...)`，功能用 `feat(...)`，修复用 `fix(...)`，测试用 `test(...)`。提交消息英文。
- Reviewer 修复意见尽量形成独立 commit；Tester 新增测试可独立 `test(...)` commit。禁止把多个无依赖功能塞进一个提交。
- 可并行：P01 与 P03（P00 后）；V101/V103/V104/V105/V106（相应脚手架后）；V201 与 V202（V1 go 后）；V205/V206 可在 V204 契约稳定后并行。
- 不可并行越过硬门：P00 未过不得写业务；P00-P04/Day-0 未过不得启动真实 V1；V107 未 go 不得实现 V2 真实主链路。
- 本轮用户已明确要求“多 commit”，构成本次 V2 开发的提交授权。执行者应按任务/评审修复/测试边界分批实际提交，不再等待额外确认；仍须使用英文 Conventional Commits、保持每个提交可独立审查和回滚，禁止提交无关改动或添加 `Co-Authored-By`。

## 5. 证据规则

- Red 证据必须显示测试因缺少目标行为失败，而不是语法、环境或无关基线失败。
- Green 证据必须包含命令、退出码、时间、环境、commit SHA；集群证据还要有 namespace、K8s/RuntimeClass/镜像 digest。
- 截图和视频只证明 UI 观感，不替代契约、数据库、网络和安全断言。
- 若全量测试有既有失败，必须证明目标分支基线与改动分支差异；阶段门仍要求全部必过测试为绿。
- 证据目录用 `YYYYMMDD_主题`，禁止 `outputs/`。

## 6. 变更控制

发现计划冲突时先在 `04-风险决策与阻断项.md` 新增决策记录。不得自行缩小 V2、绕过 V1、将公网服务当临时依赖或把 unsupported 静默成功。重大架构变化需 Planner 更新任务、Reviewer 确认测试影响后继续。
