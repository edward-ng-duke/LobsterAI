### BLK-05 可执行解除协议

1. **PLAT 输入：** `registryHost`、五个 repository 路径、TLS CA bundle 的 secret/config reference、push robot 与 runtime pull robot 的 secret reference、允许的 build/CI source CIDR、immutability/retention policy。值必须是批准内网地址；文档与 Git 只记录非秘密值和 ref，不记录 token/password。
2. **SEC 输入：** 二选一并冻结：`key` 模式的 cosign public-key/trust-root ref 与 signer key ref，或 `keyless` 模式的内部 OIDC issuer、精确 certificate identity/subject；另给出 Rekor/透明日志是否为批准内网服务、撤销与轮换验证方式。模式和值缺一不可，不能由 P03 Developer 自签临时根。
3. **P03 执行：** 以 exact five 本地/CI 已验证 digest 发布到批准 repository，逐镜像执行远端 digest readback、签名、独立 verify，并生成 `publish-signature-report.json`。建议入口固定为：

   ```bash
   node scripts/verify-registry-release.mjs \
     --input docs/supply-chain/registry-signing-input.json \
     --evidence .reports/supply-chain/20260712_PR3部署供应链证据 \
     --report .reports/supply-chain/20260712_PR3部署供应链证据/publish-signature-report.json
   ```

   该 runner 必须实际调用并记录等价命令：`docker login ... --password-stdin`（凭据来自 secret ref）、`docker push`、`docker buildx imagetools inspect <repo>@<digest>`，以及所选模式对应的 `cosign sign`/`cosign verify`。`verify` 必须绑定 exact digest 与批准 public key，或同时绑定精确 `--certificate-identity` 和 `--certificate-oidc-issuer`；禁止只验 tag、只看签名存在或把 signer 与 verifier 共用的未批准临时 key 当信任根。
4. **退出语义：** 输入/ref/内网 registry/cosign/权限不可用为 `BLOCKED/2`；digest readback 不同、签名 identity/trust-root 不符、撤销/过期、五镜像缺项或报告篡改为 `FAILED/1`；仅当五镜像 `publishedDigest == localDigest == ciDigest` 且五个签名独立验证为批准 identity 时 `PASS/0`。任何非零都禁止 P03 Reviewer 给任务级批准。
