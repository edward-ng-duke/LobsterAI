### 8.5 B00-5B：内部 registry publish 与签名验证

本步只在 B00-5A 的 exact-five digest 冻结后执行。当前 registry/TLS/robot/signing 输入尚未提供，因此状态必须保持 `BLOCKED_EXTERNAL_REGISTRY_INPUT`，不能预写 READY/PASS。

- **PLAT 输入：** `registryHost`、五个 repository、TLS CA ref、push/pull robot secret refs、网络 allowlist、immutability/retention policy。
- **SEC 输入：** 唯一 `signingMode=key|keyless`，以及相应 public-key/trust-root ref 或内部 OIDC issuer + 精确 certificate identity、撤销/轮换策略。
- **P03 执行：** 只消费非秘密配置与 secret refs，不创建临时信任根，不把凭据写入 Git/report/stdout。

唯一计划入口：

```bash
node scripts/verify-registry-release.mjs \
  --input docs/supply-chain/registry-signing-input.json \
  --evidence .reports/supply-chain/20260712_PR3部署供应链证据 \
  --report .reports/supply-chain/20260712_PR3部署供应链证据/publish-signature-report.json
```

runner 必须逐镜像实际执行等价的 `docker login --password-stdin`、`docker push`、`docker buildx imagetools inspect <repo>@<digest>` 与选定模式的 `cosign sign`/`cosign verify`。key 模式 verify 绑定批准 public key；keyless 模式同时绑定精确 certificate identity 与内部 OIDC issuer。报告记录 local/CI/published digest、repository、verified identity/trust-root fingerprint 与 runId，不记录 secret。

输入/ref/registry/cosign/权限不可用为 `BLOCKED/2`；远端 digest 不等、签名/identity/trust-root/撤销校验失败、五镜像缺项或报告篡改为 `FAILED/1`；仅五镜像 `publishedDigest == localDigest == ciDigest` 且独立 verify 命中批准 identity 时 `PASS/0`。

### 9.2 B00-3A 精确 Red 与 NUL parser 契约

首个 Red 固定为：在临时 Git repo 的 baseline 后提交 `改造计划/含 空格\t引号“与换行\n记录.md`，分别设置 repo-local `core.quotepath=true` 和 `false`，断言 `resolveProductSourceSha(root) === baseline`。当前按行/Tab 切分的实现必须因目标缺陷失败，而不是因 Git/文件系统错误失败。

Green 的唯一命令形状是：

```text
git diff --name-status -z -M -C <first-parent> <commit>
git diff-tree --root --no-commit-id -r --name-status -z -M -C <root-commit>
```

`-M` 不会产生 copy status，因此 §8.2 的 `-M` 表述在本节明确更正为 `-M -C`。实现必须将 name-status 调用与普通文本 `gitOutput()` 分开：`spawnSync` 不设 `encoding`、直接取 `Buffer`，不得对 stdout 调用 `trim()`、按行拆分或依赖 `core.quotepath`。

新导出的 parser 名称固定为 `parseGitNameStatusZ(output: Buffer): string[] | undefined`，并按以下契约测试：

- 空 Buffer 返回 `[]`；非空输出必须以 NUL 结尾，否则返回 `undefined`。
- status 只接受单字节 `A/D/M/T/U/X/B` 及带相似度的 `R0..R100`/`C0..C100`；`R*`/`C*` 消费两个非空 path，其余消费一个非空 path。
- status 以 ASCII 解析，path 用 `TextDecoder('utf-8', { fatal: true })` 逐字段解码；非法 UTF-8、缺 path、多 path、空 path 或未知 status 均返回 `undefined`，不做 NFC/NFD 归一化。
- Git 非零、`spawnSync.error`、signal、输出超限或 parser `undefined` 必须使 `changedPathsAgainstFirstParent()` 返回 `undefined`；resolver 随即 fail closed 到当前 commit。

Red 批次还必须直接喂给 parser：`R100\0old\0new\0`、`C100\0old\0new\0`、截断 rename、无尾 NUL、未知 `Z`、非法 UTF-8；并用真实 Git 复测中文 NFC/NFD、引号、Tab、换行、frozen→product 和 product→frozen rename。

### 9.3 B00-3B hermetic fixture 的可调用契约

`tests/helpers/supply-chain-evidence-fixture.ts` 对测试只暴露 `createSupplyChainEvidenceFixture(options?)`，返回 `{ root, reportPath, evidenceDirectory, sourceSha, runChecker, readReport, writeReport, mutateEvidenceDocument, cleanup }`。每个测试独占 root，`afterEach` 必须调用 `cleanup`。

创建顺序固定为：

1. 复制 checker/schema/inventory 及 inventory discovery 所需最小资产树，创建 `package.json` 中与真实声明一致的 10 个 plugin。
2. `git init --initial-branch=main`，配置 fixture 用户，提交上述 synthetic source，动态取 `sourceSha`；不写 alternates，不从主仓库拷贝 `.git`。

### 9.4 两个 CI workflow 的必选 DAG

`tests/deploy-ci-evidence.test.ts` 用 `js-yaml` 解析 workflow，不只用宽松 substring，并锁定：

- `.github/workflows/ci.yml` 的 `test.needs` 只依赖无条件的 `lint`，不依赖 `changed-files`/`build-main`/`build-renderer`，且 `test` 无 job-level `if`；checkout `fetch-depth: 0`，顺序包含 `npm ci` 后 `npm test`，不允许 `npm install`、`continue-on-error` 或 `|| true`。`pull_request` 无 paths 过滤，因此 scripts/tests-only PR 也必须运行。
- `.github/workflows/saas-scaffold.yml` 的 push 和 pull_request paths 都覆盖 helper 与 `tests/fixtures/supply_chain/**`；`scaffold` checkout 保持 full history，并依次执行 `docker:build:check` → `supply-chain:check` → `helm:lint` → `test "$(stat -c '%a' '.reports/supply-chain/20260712_PR3部署供应链证据/docker-build-check.json')" = 600` → upload exact-five artifact。upload 没有 `if: always()`/`continue-on-error`，`if-no-files-found: error`、`include-hidden-files: true`、`retention-days: 14`、名称包含 `${{ github.sha }}`。
- `p03EvidenceRelativeDirectory` 由 `scripts/check-supply-chain.mjs` 唯一导出；Docker checker 导入它，workflow 和 CI 测试与它的字面值对账。任意一处路径漂移必须使 CI test Red。

主 `CI` 中的必选结果是 `lint=success` 和 `test=success`；条件 build job 可按 path 语义 skip，但不得再导致 `test` skip。`SaaS Scaffold` 必须同时 `scaffold=success` 和 `db-integration=success`；只有 artifact 而 workflow/job 失败不算证据 Green。

### 9.5 B00-5A exact-five 绑定、验收与回退触发器

local 与 CI 都必须记录 `checkoutSha` 和 `productSourceSha`；两者在同一 run 内可因 frozen test/workflow commit 不同，但 local/CI report 的 `productSourceSha` 必须相同，并且各自 checkout 必须是该 product source 的 first-parent descendant。CI 候选优先使用与 local 完全相同的 checkout SHA；若仅因触发 workflow 追加 frozen commit，必须在记录中列出两 SHA 之间的全部 name-status 并由 resolver 证明 product source 未变。

下载 artifact 后以 NUL-safe 文件枚举生成排序的 SHA-256 manifest，与 `docker-build-check.json` 中 5 个 SPDX/Grype hash 交叉对账；只记录下载文件内容 hash，不把 artifact ZIP 元数据或下载后权限当证据。report mode `0600` 在 upload 前的 Linux job 中单独断言和记录，因 artifact 打包不保证保留 mode。

以下任一发生就废弃当前 P03 live candidate，从 B00-5A 重来：

- `scripts/**`、`docker/**`、`apps/**`、`libs/**`、`openclaw-extensions/**`、`resources/**`、`SKILLs/**` 或其他 resolver 判为 image input 的变更；
- exact-five set/platform/digest、SBOM/Grype hash、Critical 计数、plugin installed/skipped、local image inspect 或 product source 任一漂移；
