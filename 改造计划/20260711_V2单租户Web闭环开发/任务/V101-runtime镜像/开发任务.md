### 1.2 Day-0 真实环境硬门

- 每个 RuntimeClass × architecture 的资源/启动延迟预算、重复次数和 owner。预算须在首次动态运行前版本化，不得看完结果后改阈值。

本机 Docker、kind/minikube 的 runc Pod、模拟 `kubectl` 或预制日志只能验证静态规则/编排器，**绝不能写成 gVisor/Kata 集群兼容证据**。缺 RuntimeClass、对应原生架构节点、集群权限或任一 Day-0 值时，相应矩阵单元必须 `BLOCKED`，V101 不交 Tester、不勾总表。

### 1.3 结果语义

- `PASS`：真实执行且全部断言通过，具有同 runId 的新鲜、完整、可重算证据；只有 PASS 可返回 0。
- `FAILED`：环境完整但功能、安全、性能或内容断言失败；非零，不能降级成 BLOCKED。
- `BLOCKED`：缺前置、Day-0、工具、RuntimeClass、原生节点、权限或批准内网输入；专用非零码并穷举 `blocked: missing <decision>`。

`SKIPPED`、空矩阵、所有 case 为 stub、本机容器成功、命令 exit 0 但无 artifact 均不得汇总 PASS。

## 2. 目标、范围与非目标

### 2.1 目标

### 2.2 范围内

- `docker/openclaw-runtime/`：生产镜像、entrypoint、真实 gateway readiness；debug Dockerfile 只用于隔离调试，不进生产证据。
- 必要的 OpenClaw Linux build/patch/plugin/extension/channel-dependency/prune 脚本，以及新增 V101 专用的 image content/Kit lock/native module/compatibility/resource probe。
- `apps/runtime-orchestrator/poc/` 中 P04 预留的 runtime-image/compatibility/resource 扩展点，以及最小 sandbox Pod values/manifest 参数化；不改 P04 的三态、证据和 cleanup 核心语义。
- `tests/runtime-image-*.test.ts`、`tests/fixtures/runtime_image/`（可按 P04 已验收结构微调）：纯逻辑、隔离副本 mutation、可控假进程和报告 schema 测试。
- `docs/poc/YYYYMMDD_V101_OpenClaw运行时镜像与兼容矩阵证据/`：证据契约/报告；实际运行目录还须含唯一 runId，禁止 `outputs/`。

### 2.3 非范围

- 不运行/声称 V102 真实 model turn、V103 egress 完整 allow/deny、V104 PVC/S3 一致性、V105 桥契约或 V106 Config Sync golden。

## 3. 镜像、Kit 锁定与内容审计契约

### 3.1 可复现输入与不可变标识

- 唯一版本输入是根 `package.json.openclaw.version/plugins`、版本范围内 patch、`package-lock.json`、内网同步的 OpenClaw source SHA 和已验收 supply-chain manifest；禁止 `latest`/浮动 range/未定 git ref。
- 产出 OCI index digest 与 `linux/amd64`、`linux/arm64` 各自 manifest/image digest；报告绑定 source SHA、Dockerfile hash、base image digest、OpenClaw pinned tag/source SHA、patch hash、lockfile hash、builder/BuildKit 版本。
- 从**实际镜像文件系统**生成 `runtime-content-lock.json`，至少列出 OpenClaw bundle/version、本地 extensions、全部 production-eligible plugins、Kit/MCP/skills 默认资产、native addons（路径/版本/架构/libc/SHA-256）。不得只抄声明清单自证。
- Kit 清单与 P03 inventory 双向对账，包含现有 built-in Kit 及 computer-use Kit 资产；漏项、幽灵项、重复 ID、大小写碰撞、symlink 越界、integrity 漂移均阻断。

### 3.2 内容与密钥审计

- 允许的运行时根仅含 OpenClaw/Node 执行面和已审核依赖；明确拒绝 Electron Renderer、Chromium GUI、Xvfb/x11vnc/noVNC、Darwin/Windows 二进制、宿主缓存、`.npmrc`、registry auth、PEM/JWT/token、租户 prompt/workspace/state。
- 扫描 image filesystem、layer/history、env/labels、SBOM 和报告；敏感命中只输出脱敏类型/路径哈希，不回显原文。
- 以镜像内数字 UID/GID 非 root 用户启动；根文件系统只读，仅 `/state`、`/workspace`、`/tmp` 为明示可写挂载，禁提权、drop `ALL`、`RuntimeDefault` seccomp、禁 SA token。
- `OPENCLAW_GATEWAY_TOKEN` 缺失时 fail closed；真实 readiness 必须证明 gateway WS 协议就绪，TCP 开端口/恒真脚本不算 ready。

### 3.3 SBOM、漏洞与签名

- 每个 platform digest 以 P03 固定 Syft 生成机器可读 SBOM，以 P03 `scanner-policy.json` 固定 Grype 扫描；SBOM/scan 同时绑定 platform digest 和 source SHA。
- 无 SBOM、digest/source 不匹配、扫描失败、Critical 发现或过期 waiver 都是 FAILED；waiver 必须 owner/reason/expiresAt/CVE+package+digest 精确匹配，不得宽泛放行。
- 内部 registry/签名信任根未冻结时，本地 build/SBOM 可继续，但 push/sign/verify/集群运行均 BLOCKED；不得推公网或用假 cosign 证据补齐。

## 4. Linux 多架构与原生模块验证

- 用 buildx/等价受控 builder 从同一 source SHA 构建 `linux/amd64` 与 `linux/arm64`；`TARGETARCH`→`linux-x64/linux-arm64` 映射必须 fail closed，其它架构不得默认为 x64。
- 对镜像内所有 `.node`/可执行文件做 ELF architecture/libc/interpreter/needed libraries 审计，`ldd` 不得出现 `not found`；拒绝 Darwin/PE、wrong-arch、musl/glibc ABI 混用和非预期 setuid/capability。
- 在对应原生节点分别执行 native addon `require/import`、OpenClaw 启动、plugin 加载与 stdio MCP 探针。QEMU/仿真可作 build smoke，不能替代对应架构原生节点的动态验收；缺节点时该 architecture 单元 BLOCKED。
- 拔掉一个必需 native addon、放入错架构 `.node`、删除 shared library、禁用可选 plugin 和给出未锁版 plugin 都要有可观测且 fail-closed 的负向结果。

## 5. gVisor/Kata 工作负载兼容与资源矩阵

### 5.1 矩阵维度与探针

矩阵主键为 `sourceSha + imageDigest + architecture + runtimeClass + resourceClass + probeId`，每个不可重复。至少覆盖：

| 维度 | 要求 |
|---|---|
| RuntimeClass | `gvisor` 默认、`kata` 回退；`runc` 只可作诊断性基线，不得汇总安全 PASS |
| Architecture | linux/amd64、linux/arm64；各用对应 platform digest 与原生节点 |
| ResourceClass | `small`=0.25/1 CPU、512Mi/1.5Gi、2Gi；`standard`=0.5/2 CPU、1Gi/3Gi、5Gi；`large`=1/4 CPU、2Gi/6Gi、10Gi；V8 old-space 小于 memory limit |
| 基础执行 | `node`、POSIX shell、spawn/exec/信号、多子进程/stdin/stdout/stderr、大量小文件与目录遍历 |
| OpenClaw | 非 root+只读根启动到真实 WS ready、pinned version/bundle/plugin 加载、SIGTERM 有界退出 |
| 扩展执行 | 离线 npm 包解压/已固化 `npx`、代表性 stdio MCP、代表性 skill 脚本、Kit 资产可读与 integrity 对账 |
| 安全/韧性 | 写系统路径、提权、读 SA token 失败；无 HOME/不可写 HOME；缺 plugin/native module；只读根；OOM/crash/timeout/health 假阳性 |

`npm/npx` 探针只能使用镜像内已固化资产或批准内网 registry 的预取包；生产 Pod 不得安装/下载依赖。

### 5.2 功能、性能与路由结论

- 每个矩阵单元按 Day-0 冻结的重复次数执行，启动采样不少于 10 次；记录 success rate、p50/p95/p99 ready latency、空闲/启动/探针 CPU、RSS/working set、ephemeral bytes、restart/OOM/exit/signal。
- 功能接受线为必需探针 **100% 成功**、无 OOM/意外重启、无密钥/跨 Pod 状态；资源与延迟须不超过 Day-0 预先冻结预算。资源预算未冻结则性能结论 BLOCKED，不用“已记录”代替 pass/fail。
- gVisor 单元全过才得写 `route=gvisor`；gVisor 不兼容/超预算时必须在同镜像/架构/资源档重放 Kata，Kata 全过才得写 `route=kata`；两者都失败或无可用 RuntimeClass 时不得回退 runc，结论为 V1 no-go/BLOCKED。
- OOM 负向探针须证明 cgroup 限制生效、Pod 状态和终止原因可观测、同 namespace 其它探针不受影响；OOM 用于证明隔离而不是功能 PASS。
- SIGTERM 必须在 Day-0 冻结 grace 内退出、无孤儿子进程/半写 lock，且不依赖可写根或公网。V101 只验证 runtime 进程行为，不声称 V104 持久化/租约已验收。

## 6. 生产完全断公网验证

- build/test 环境可在受控阶段下载并固化依赖，候选镜像在生产同构运行时不得执行 `npm install`/`pnpm`/`yarn`/`npx` 下载/`pip`/`curl`/`wget`/`git clone`。
- 先用 Docker `--network=none` 做候选 smoke，再在真实 RuntimeClass Pod 用 default-deny-all egress 运行 gateway ready 与本地执行探针；任意 DNS/连接公网或运行时安装尝试都是 FAILED。
- 若某探针确需内网服务，必须由 P04 Day-0 按精确 Service selector/port 显式注入并在报告分类；不得宽泛放行 namespace/RFC1918。V103 负责完整 allow/deny 网络证据，V101 只证明 runtime 镜像无公网也能启动/运行本地执行面。
- 生产证据不得出现 Docker Hub/GitHub/npmjs/公网 CVE feed 实时访问；SBOM/CVE DB/基础镜像/包由离线同步或批准内网源事先固化。

## 11. 回滚、BLOCKED 准则与 Reviewer / Tester 交接

### 11.1 回滚

- 镜像只回退到上一个已审计 platform/index digest，并同步回退 SBOM、signature、content lock、OpenClaw patch 及路由矩阵；禁止 tag 漂移或只回退生成物。
- gVisor 工作负载回退 Kata 只能使用已验证的精确 tuple；未验证 tuple 不得类推。两者都失败时隔离候选镜像并提 V1 no-go，不回退 runc。
- 集群 cleanup 只按 runId+managed label 删本轮资源，禁止 namespace-wide delete；故障证据按 retention 保留并脱敏。

### 11.2 BLOCKED / FAILED 准则

- 缺 P00–P04 验收、Day-0、内部 registry/信任根、RuntimeClass、原生架构节点、权限、预算或工具：`BLOCKED`，非零，不交 Tester。
- 环境完整但 build/content/native/security/readiness/资源/断网断言失败：`FAILED`，退 Developer；不得通过放开 root/HOME/公网、删探针或改报告补绿。
- 意外公网可达、Secret 入镜像/证据或 cleanup 越界：立即停止，轮换可能泄漏凭据，保留脱敏失败证据并退 Developer。

### 11.3 角色交接

- Developer 按 Round 追加 Red→Green→Refactor、分 commit、命令/退出码/环境/证据，对 AC01–AC22 逐项定位；不得自审通过或勾总表。
- 独立 Reviewer 必须审前置硬门、范围、22 AC、有效 Red、镜像实际内容锁、native/Kit 对账、供应链、断公网、证据脱敏/新鲜度和真实 RuntimeClass attestation。任一 P0/P1、缺 Red、本机 Docker 冒充集群、空矩阵/可伪绿均 `CHANGES_REQUESTED`。
- 仅 Reviewer `APPROVED_FOR_TEST` 后独立 Tester 开始。Tester 至少新增一个 Developer 未覆盖角度：wrong-arch native addon、Kit 大小写/Unicode、假 WS readiness、不可写 HOME、SIGTERM 进程树、OOM 邻居隔离、RuntimeClass 伪 attestation、证据原子中断或敏感编码绕过。
- Tester `FAILED` 退同一 Developer，修复后必须重新 Reviewer。只有 Tester `PASSED` 且协调者独立重放静态+动态证据后，协调者才可勾总体表；各角色均不得自行勾 Todo。
