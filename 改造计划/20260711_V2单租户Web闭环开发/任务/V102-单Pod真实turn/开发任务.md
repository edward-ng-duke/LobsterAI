### 1.3 结果语义

- `PASS`：真实集群中的 gVisor Pod 从真实 WS hello/ready 到 `chat.send { sessionKey }` 和内网模型调用全部发生，必需事件及 complete 全部通过，同 runId 证据可重算；只有 PASS 返回 0。
- `FAILED`：前置和环境完整，但功能、事件、隔离、时延、资源、安全、证据或清理断言失败；非零，不能降级为 BLOCKED。
- `BLOCKED`：缺前置验收、Day-0、内网模型、RuntimeClass/节点、digest、PVC、权限、预算或工具；专用非零码并穷举 `blocked: missing <decision>`。

`SKIPPED`、Kata/runc 通过、本机 Docker、fixture/mock kubectl、Pod Running、端口可连、只有 ack、空响应、无模型侧关联证据、无 artifact 的 exit 0 都不得汇总 PASS。

## 2. 目标、范围与非目标

### 2.1 目标

在 V101 已验收的精确 gVisor tuple 上，使用 P04 已验收 harness 在一个短命 Sandbox Pod 中启动真实 OpenClaw gateway，由集群内 client 经 token 鉴权 WS 发送一条 canonical `chat.send { sessionKey, ... }`，证明请求实际到达批准内网模型端点，并收齐可关联、有序、唯一收尾的消息事件。产出可直接支撑 V1.1 判定的真实环境证据，但不代替 V101 全矩阵、V103 全网络门、V104 存储门或 V107 总决策。

### 2.2 范围内

- `apps/runtime-orchestrator/poc/` 中 P04 冻结的 turn probe/client/runner/schema/fixture 扩展点；实现精确连接、鉴权、RPC、事件归一、超时、abort、证据和 cleanup。
- `charts/lobsterai/` 中仅 V102 所需的 sandbox fixture/values/test NetworkPolicy 参数，保持 gVisor、digest-only、非 root、只读根、drop ALL、禁 SA token 和精确内网 Service allow。
- `tests/runtime-turn-*.test.ts` 与 `tests/fixtures/runtime_turn/`（可按 P04 验收后的实际命名微调）：纯逻辑、可控 WS/子进程、隔离 mutation 和报告 schema 测试。
- `docs/poc/YYYYMMDD_V102_gVisor单Pod真实OpenClawTurn证据/`：证据契约与真实 runId 子目录；禁止 `outputs/`。
- 必要时在 P01 已冻结 contract 上增加 PoC 内部 result schema 引用；不改产品 REST/AsyncAPI 字段事实源。

### 2.3 非范围

- 不实现 V2 产品 API/Cowork Service/Web bridge/Redis Stream/多副本 owning-replica，不冒充浏览器 E2E。
- 不实现 V4 Runtime Orchestrator、动态 Pod 池、预热、租约、自愈、HPA 或生产 SLO；V102 只运行 P04 短命单 Pod harness。
- 不修改 V101 镜像、OpenClaw pinned version/plugin/patch/content lock；镜像缺口退回 V101。
- 不声称 V103 全套 allow/deny、DNS rebinding、metadata/K8s API/横向隔离通过；V102 只要求这条 turn 在精确内网模型 allow 与公网 default-deny 下可运行。
- 不声称 V104 并发/断连/S3 一致性、V106 Config Sync golden 或 V1 总 go；只使用最小受控 config/workspace fixture。
- 不提交 token/provider key/kubeconfig/Secret data、原始 prompt/模型敏感响应、大日志、镜像 tar 或真实租户数据。

## 3. 真实 turn 与证据契约

### 3.1 真实性鉴证

每次合格 run 必须将同一 `runId` 关联到：

1. 脱敏的 render/server-dry-run/applied Pod spec，Pod UID、namespace、RuntimeClass 对象及 handler、调度 node UID/architecture/kernel/runtime 证据；
2. 与 V101 验收 tuple 一致的 source SHA/platform digest，以及 Pod status 返回的实际 `imageID`；tag 不作证据；
3. 真实 WS hello/ready 与鉴权成功，client 必须在集群内受控 Pod/Job 运行；`kubectl port-forward`、localhost 或从开发机直连不能作 PASS 路径；
4. 唯一 `sessionKey`/turn correlation ID，`chat.send` 请求与 ack，OpenClaw 运行时的原始脱敏事件序列和 PoC 归一结果；
5. 内网模型网关审计中同 correlation 的请求/响应/终止结果，证明不是预制 gateway event；只记录 model ID、状态、用量/时间摘要和响应 hash，不归档凭据或敏感原文；
6. NetworkPolicy 实际 applied 版本：只放行 kube-dns、内网模型 Service 和完成 turn 必需的精确已批内网回调；无公网 NAT/代理兜底。

RuntimeClass 名称、Pod annotation 或预制 JSON 不能单独证明 gVisor。V102 必须复用 V101 已验收的 runtime attestation 方法并重查实际 Pod/node/imageID；无法实施 attestation 时是 BLOCKED，不默认成功。

### 3.2 RPC、sessionKey 与事件断言

- 请求仅用真实 RPC `chat.send`，寻址是 `sessionKey`；拒绝 `session.start/session.continue`、以 `sessionId` 代替 key、空/Unicode 欺骗 key、其它 run/session 的帧。
- 稳定的必需 PoC 语义序列是：真实 ready 后 `chat.send` ack，然后同 sessionKey/run 的助手 `message`、至少一次有效 `messageUpdate`、可获取时的 `contextUsage`，最终唯一 `complete`。若当前 pinned OpenClaw 的 raw event 名与 PoC 语义名不同，由 P04/P01 已冻结 adapter 做唯一映射，不在测试中臆造新 wire 名。
- `contextUsage` 若协议明确为 optional，可记 `not_emitted_by_pinned_runtime`，但必须用 `chat.history/sessions.list`（以已冻结 contract 为准）给出同 sessionKey 的用量/历史侧证；不得用空字段写 PASS。
- complete 前的增量内容组合后必须非空，与最终消息/历史 hash 一致；complete 后不得再出现同 turn 的内容帧；错误终止不得同时被记为 complete PASS。
- 简短基准 prompt 使用无敏感合成 nonce，不用真租户语料。deterministic 内网端点可校验 nonce 回显；非确定模型不以精确文字断言，而以非空语义响应、关联 ID、事件/历史一致和模型审计证明真实。
- `chat.send` 的安全载荷线和 30,000,000-byte WS 硬限不得放宽；超限必须有确定可读失败且 Pod 不崩溃。

### 3.3 时延、资源与重放

- 分别采集 Pod scheduled→gateway WS ready、ready→`chat.send` ack、ack→首语义帧（模型首响应）、ack→complete 和 cleanup；不得把 TCP connect 写成 ready，不得把 ack 写成首 token。
- 记录 container CPU/RSS/working set/ephemeral、restart/OOM/exit/signal 和模型端计费/用量摘要；阈值引用 Day-0 预先冻结表和 V101 同 tuple 预算，未冻结阈值时性能结论 BLOCKED。
- 主成功场景在同一 Pod 内至少独立执行 3 次，使用不同 run/turn correlation；必须 100% 成功，无重启/OOM/串帧。其中至少一次续用同 `sessionKey` 验证 history 增长，一次使用第二 `sessionKey` 验证不串扰；这是单 Pod PoC，不是生产共享 Pod 设计承诺。
- 重复执行不得覆盖旧成功包；并发相同 runId 必须 fail closed，中断只能留下显式 FAILED/临时目录并进入有界 cleanup。

## 4. 生产完全断公网边界

- 候选镜像已由 V101 固化所有依赖；V102 Pod 运行期禁止 `npm/pnpm/yarn/npx` 下载、`pip`、`curl/wget` 取依赖、`git clone`、公网插件/skill/MCP 拉取。
- provider `baseUrl` 必须是批准内网 Service DNS，Pod 不持真实上游 provider key；上游凭据只由独立内网模型网关保有。`OPENCLAW_GATEWAY_TOKEN` 与模型 provider key 严格区分，两者都不进证据。
- V102 成功 run 必须应用 default-deny egress + kube-dns + 精确内网 Service selectors/ports，且运行环境无公网 NAT/代理兜底。如果为跑通 turn 需要 `0.0.0.0/0`、整 namespace、整 RFC1918、公网 provider 或开发机转发，直接 FAILED/BLOCKED，不得当临时例外。
- V102 证明这条 turn 的最小断公网可运行性；V103 仍必须独立证明公网、IMDS、K8s API、DB、Redis、横向 Pod、未批 RFC1918 全套 deny 与批准内网 allow。

## 10. 回滚、FAILED/BLOCKED 准则与角色交接

### 10.1 回滚

- 代码按 V102 分 commit 回滚；回滚 V102 turn probe/fixture/values 后重跑 P04/V101 相关静态门，不回滚或重打 V101 已验收镜像。
- 集群资源仅按 `runId + managed-by` 精确删除；禁止 namespace-wide delete，禁止删共享 PVC/bucket/其它 run 的 Secret/NetworkPolicy。合成 workspace 测试数据按 retention 清理，脱敏失败证据保留供审查。
- gVisor 功能失败时可在同 digest/architecture/resourceClass 上跑 Kata 诊断，但 V102 仍 FAILED；不改 `runtimeClassName` 把 Kata/runc 写成任务通过。若与 V101 已验收结论冲突，退 V101 复核并阻断 V107。
- 意外公网可达、Secret 泄漏或 cleanup 越界时立即中止，隔离资源，轮换可能泄漏的 token，保留脱敏审计记录，不继续跑其余场景。

### 10.2 FAILED / BLOCKED 准则

- 缺前置验收/Day-0/集群/gVisor/node/digest/PVC/内网模型/权限/预算/工具：`BLOCKED`，非零，不交 Tester、不勾总表。
- 环境完整但 ready/RPC/event/model correlation/abort/时延/资源/断公网/脱敏/证据/清理断言失败：`FAILED`，退 Developer；不得改成 BLOCKED、改阈值、放开公网/root、换 Kata/runc、删测试或用截图补绿。
- 仅 deterministic 内网模型可用时，可产出 V1 真实 OpenClaw turn 结论，但必须带限定；如协调者将 V102 本身冻结为“必须真实模型供应商”，则缺批准供应商内网网关时必须 BLOCKED，不自行改口径。

### 10.3 Developer → Reviewer → Tester 独立交接

- Developer 按 `Round N · YYYY-MM-DD HH:mm · Agent/commit` 追加 Red→Green→Refactor、命令/退出码/环境/证据，对 AC01～AC24 逐项定位；不得自审通过、不写 Reviewer/Tester 结论、不勾总表。
- 独立 Reviewer 必须审前置硬门、范围、24 AC、有效 Red、gVisor/image/model 真实性、RPC/event 序列、断公网、Secret/证据/清理、阈值和原子性。任一 P0/P1、缺 Red、无真实 run、本机/fixture/Kata 冒充 gVisor、无模型关联或可伪绿均 `CHANGES_REQUESTED`。
- 仅 Reviewer `APPROVED_FOR_TEST` 后独立 Tester 才开始。Tester 必须在新 runId 重放至少一条真实 gVisor turn，不能只检查 Developer 的证据；并新增至少一个 Developer 未覆盖角度：假 RuntimeClass attestation、模型侧无 correlation、sessionKey 串帧、complete 后续帧、abort/SIGTERM 竞态、Unicode/大帧、敏感编码绕过、原子中断或 cleanup label 逃逸。
- Tester 无集群/模型/审计读权时结论是 BLOCKED，不得使用 Developer runId 写 PASSED。Tester `FAILED` 退同一 Developer，修复后必须重新 Reviewer。只有 Tester `PASSED` 且协调者独立核对前置与真实证据后，协调者才可勾总表。
