## 5. Playwright 真实浏览器主链

### 5.1 每个浏览器必过 journey

Chromium、WebKit、Firefox 均从全新 browser context 执行，不复用 storageState/token：

1. 访问 Web HTTPS 入口，断言无 `window.electron`/native import 初始化崩溃且 zh/en shell 可用。
2. 点击登录，经实际 OIDC redirect/callback 和 PKCE S256 完成认证；state/nonce/verifier 一次消费，access 仅内存，refresh 仅 HttpOnly Cookie，URL/storage/trace无 token。
3. 从 UI 创建 session 并提交首 prompt；REST 返回 accepted requestId，PG 中 tenant-scoped session+首 user message+receipt/outbox 原子存在。
4. Web 申请 origin/user/sid/jti/session-scoped 一次性 ticket，经真实 WS Upgrade/authok/subscribe；URL/subprotocol/storage/log不含 ticket/JWT。
5. 真实 OpenClaw/gVisor/内网模型产生 `message → messageUpdate* → complete`，UI 增量可见，最终内容与 PG/history/Redis Stream 同源；终态恰一次。
6. 同一 run 触发并打开 `text/markdown/code/image/html` 五类只读 artifact（SVG作为 image安全分支另测）；metadata/version/hash/size/MIME与 S3/inline bytes一致。
7. 登出后 access/refresh/ticket/preview 失效，socket/timer/blob URL/cache清理；旧页面/后退/BFCache不能恢复内容或身份。

可控 IdP lane 还必须覆盖 callback刷新/后退、多 tab state隔离、code重放/过期、JWKS轮换/mix-up、refresh并发与 logout竞态。真实 IdP lane必须覆盖 active happy path、refresh、logout、disabled与 wrong-tenant拒绝，以及真实key rotation或SEC批准演练；只有 fixture 或只有真实 happy path均不能 PASS。

### 5.2 10 事件、权限、abort 与 error

- 动态从 accepted `CoworkStreamRegistry` 取分母并证明 wire/AsyncAPI/generated/bridge/observed集合精确相等：`message/messageUpdate/sessionStatus/contextUsage/goal/contextMaintenance/permission/permissionDismiss/complete/error`；missing/extra/duplicate=0。
- 至少三个真实 turn：成功 terminal、user abort terminal、runtime/error terminal。`complete` 与 `error` 互斥；abort不新增 wire type、不伪造 complete；每 request/turn 仅一个有效 terminal。
- 通过预装且供应链固定的 deterministic test skill/真实 OpenClaw RPC 触发 tool permission 与 AskUser：allow/deny/updatedPermissions、回答、重复同答、异答 conflict、timeout、owner切换、terminal cleanup；每个 permission 恰一次 dismiss。
- goal、context usage/maintenance 必须由真实 service/runtime action触发，不以 WS 注入。若 accepted runtime无法真实触发，V204应重开，V208不得将该事件标 N/A。
- REST 401/403/404/409/413/415/422/429/500/502/503/504 与 WS protocol/ticket/gap/corruption错误按 accepted envelope/code 显示；原始 SQL/path/provider/secret不进 UI/log。

## 6. REST/WS、ticket 与流可靠性矩阵

### 6.1 契约与认证

- 全 Route/Bridge/Channel registry 动态 equality；每个 GA invoke走 generated REST，每个 stream row有 generated validator/dispatcher/disposition。Electron-only叶明确 unsupported/no-op/fallback，不返回 `undefined` 或假成功。
- ticket TTL/一次消费/high entropy，绑定 tenant/user/sub/sid/jti/origin/session/resource；复用、过期、撤销、换 Origin/user/scope均拒绝且不泄露资源存在性。
- REST Bearer/refresh Cookie、CSRF/CORS/Origin/Fetch Metadata按 V201；WS auth首帧前无订阅/业务帧，auth timeout/双auth/坏 schema fail closed。

### 6.2 `seq/sinceSeq/ack/gap`

- `seq` 使用 Redis Stream ID二元数值比较，不 `Number()`、字典序或与 DB message sequence混用；per-tab/per-scope checkpoint隔离。
- 只有 schema/scope/generation合法且 reducer 接纳并在 IndexedDB事务持久的最高连续 seq 才 ack；3/1/2、缺2、同seq同/异payload、ack丢失、page crash均不越 gap。
- 断线后取新 ticket并以 `sinceSeq` 重放；`seq > sinceSeq` 与 live 有序合流，交界 duplicate effective-once。retention刚好保留/刚trim/trim与live并发均测。
- gap时暂停该 scope增量，取权威 snapshot+watermark后原子恢复；snapshot慢/失败保持显式 degraded，不清空或用旧 cache伪绿。

### 6.3 chaos 与并发

必测：登录前/WS auth前、首 message前、messageUpdate中段、permission pending、complete前、complete已达但checkpoint/ack未落、complete后；offline/online、DNS/TLS reset、proxy idle close、半开、延迟/抖动/丢包、重复 close/error；Redis relay重启/trim、API replica滚动、OpenClaw reconnect/owner generation切换、PG/S3短故障。

- duplicate只去同 identity+payload；同seq/identity异payload=`STREAM_CORRUPTION`并resync。
- reorder/buffer/queue按 accepted item/byte/age界限；只可合并同 message 连续 update，permission/dismiss/status/error/complete不可丢。溢出必须 gap/snapshot/fail closed，禁止 OOM/无界暂停 ack。
- 多 tab各自 ticket/socket/generation/checkpoint；A unsubscribe/logout/close/revoke不取消B。StrictMode双mount、刷新/后退、快切session、unsubscribe与replay、cancel与terminal均不双连接/双副作用。

## 7. 五类 Artifact 与浏览器安全门

| 类别 | 正向 | 必拒/必隔离 |
|---|---|---|
| text | UTF-8/BOM/UTF-16、Unicode、byte/codepoint截断提示 | invalid encoding、NUL/binary、超限、旧version |
| markdown | heading/list/code/link、同租户安全图片 | raw HTML、javascript/file/localfile/remote embed、Mermaid执行 |
| code | Unicode/长行/只读/离线高亮 | execute/edit/save、运行时下载、半字符/无提示截断 |
| image | PNG/JPEG/WebP/GIF/AVIF、hash/dimension；SVG静态分支 | corrupt/bomb/frame/pixel超限；SVG DTD/entity/script/event/foreignObject/external/CSS URL |
| html | sanitized static layout | script/module/eval/worker/storage、fetch/XHR/WS/remote asset/DNS、meta/base/form/open/top-nav/download |

- 三浏览器均断言 DOM、CSP violation、network、opener/top URL/storage和server audit；截图只补视觉，不替代安全断言。
- preview始终绑定 immutable blob version/hash/size/MIME；incomplete/finalizing 409，expired/revoked/deleted 410/404，corrupt 422，S3失败503。新版本失败必须清旧画面/blob URL/cache，不回旧内容。
- GET/HEAD、single/open/suffix/invalid/multi Range、ETag/cache、issue/revoke/logout/two-tab、WS ready replay/gap 与 S3 timeout/TLS/DNS/429/5xx/partial/checksum全测。
- 浏览器不得直接拿 S3 key/prefix/公网 URL/signed PUT；HTML iframe `sandbox=""` 与 strict CSP不可被测试配置放宽。

## 8. 隔离、断公网、数据与 cleanup

### 8.1 隔离/安全

- tenant A/B使用相同 logical IDs/session/artifact/request名称；user A猜测B REST path、WS scope、ticket、Redis stream、S3 object、preview、workspace/agent均0泄漏、统一不可探测语义。
- body/query/header/cookie伪造 tenant/user/owner无效；关闭应用过滤的红队 lane仍由 non-owner PG FORCE RLS与复合FK阻断。
- secret/PII canary覆盖Authorization/Cookie/code/state/verifier/nonce/ticket/gateway/model key/S3 signature、prompt/tool/content/路径及 nested Error/JSON/url/base64变体；Git/DB/Redis/S3 metadata/browser storage/console/network/trace/video/screenshot/log/report扫描0命中。

### 8.2 生产完全断公网

在无公网 NAT/代理的同构 namespace 从 Web/browser、API、worker、runtime-orchestrator、OpenClaw分别探测：批准内网 OIDC/model/PG/Redis/S3/DNS/NTP/OTel可达；公共 DNS/IP、metadata、K8s API（无授权Pod）、未批准 RFC1918、CDN/font/Sentry/analytics/package registry与runtime npm/pip/git/curl下载全部不可达。任何公网 fallback或动态安装使整门 FAILED，不因主 journey成功豁免。

### 8.3 数据与资源清理

- setup/teardown都按精确 runId/tenant/schema或row label/Redis prefix+consumer/S3 prefix+version/PVC/browser profile/ticket/preview/process 清理；禁止 wildcard/shared bucket flush、admin清全库或复用上轮残留。
- 测试前断言 namespace clean，测试后两次幂等 cleanup并验证 PG/FK、Redis key/consumer、S3 multipart/object、PVC/temp/lease、OIDC fixture transaction、browser context/socket/timer/IndexedDB/blob URL、port-forward/process outstanding=0。
- 失败也必须先保存脱敏证据再清活资源；不能以“保留调试环境”为 PASS。cleanup失败使本 run `FAILED`，报告残留 exact IDs与owner。

## 9. 性能、flake 与 retry 禁止掩盖

### 9.1 固定质量阈值

在冻结资源规格、内网同地域、预热5次后采样；报告 p50/p95/p99与样本数，不删异常值：

| 指标 | 样本/硬门 |
|---|---|
| fixture IdP callback→authenticated shell | Chromium n≥20 p95≤5s；WebKit/Firefox各n≥10 p95≤7s |
| submit click→REST accepted | 每浏览器n≥20，p95≤1s、p99≤2s |
| accepted→首语义 stream | 每浏览器n≥20，p95≤5s；真实内网模型/skill |
| accepted→terminal complete/error | 每浏览器n≥20，p95≤30s、p99≤45s |
| retention内重连→恢复live | 每浏览器n≥20，p95≤3s；0丢/重terminal |
| gap→snapshot恢复 | 每浏览器n≥10，p95≤5s |
| ready小artifact click→可见 | 五类每浏览器n≥10，p95≤3s |
| update洪泛/慢consumer | 达accepted队列上限仍无OOM；heap/queue回落；terminal 0丢失 |

真实 IdP主链每浏览器至少连续3次0失败；fixture主 journey每浏览器连续10次、随机顺序3个seed均0失败。V208-specific branch/statement≥80%，auth/ticket/RLS/seq/gap/terminal/sanitizer/egress/cleanup critical branch接近100%；mutation score总体≥85%、critical集合≥90%，critical surviving mutant=0。

### 9.2 禁止伪稳定

- Playwright `retries: 0`；CI job、shell和测试框架不得自动 rerun failed test。产品协议内部有界 retry是被测行为，必须记录 attempt/backoff/idempotency，不能成为测试重跑。
- 禁 `test.skip/fixme/only/slow`、quarantine、soft assertion后PASS、`|| true`、`continue-on-error`、动态放宽timeout/queue/CSP/threshold、失败后更新snapshot或只上传成功 shard。
- 同一测试首次失败即 FAILED；后续诊断 rerun是新 runId，不能覆盖首次报告。flake候选必须有 owner、复现、修复 Red→Green与10次0失败后才能移除标签；没有“允许 flaky”名单。

## 10. Playwright 证据、同源 hash 与可观测

每个 scenario贯穿 `runId/traceId/testId/browser/project/shard/tenantHash/sessionHash/requestHash`，串联 browser action→REST→PG→owner→OpenClaw→outbox→Redis→WS→artifact。证据包至少含：

真实 IdP 的 credential 输入区使用合成测试账号别名、密码型遮罩且不截屏；Playwright trace 在 secret 输入/token exchange 前停止 chunk，callback 完成后以同 runId 续开，空档用 IdP 脱敏 audit event hash 与浏览器 redirect 时间线闭合。其余 trace 先落本 run 的隔离临时目录，上传前必须去除 Authorization/Cookie/body/input 等敏感字段并验证仍可打开；只对 sanitized trace计算证据 hash，raw trace随后 exact 删除。任何无法安全脱敏的 trace/video不得上传，也不能因此免除同场景的DOM、协议、服务端和hash证据。

- source SHA、plan SHA、V201～V207 accepted SHA/hash、contract source/output、migration、runtime/config revision、OCI imageID/digest、browser binary hash/version、namespace/Pod UID/replica/runtimeClass/NetworkPolicy。
- Playwright trace、主 journey全程 video、关键 checkpoint与失败 screenshot、DOM/accessibility snapshot、browser console/network summary；默认脱敏，真实 IdP密码/token输入不进 trace/video。
- API/proxy/OpenClaw/worker/OTel sanitized logs、PG audit/query summary、Redis Stream/replay/ack summary、S3 audit/checksum、K8s events/resource metrics、fault timeline与cleanup manifest。
- commands、exit、started/finished UTC+CST、shard/scenario counts、skipped/retry/flake/perf/mutation、PASS/FAILED/BLOCKED及原因。

strict `evidence-manifest.json` 为唯一索引：每文件记录 relative path/size/SHA-256/content class/redaction scan；再计算 packageHash。report、JUnit、trace/video/screenshot/log必须同 `sourceSha/runId/environmentId/contractHash`，缺文件、hash漂移、时间在run外、未知字段、旧report或source不等于checkout HEAD均 FAILED。只上传某个成功重跑覆盖失败证据禁止；所有 shard manifest必须汇总后才有总状态。

## 11. CI 分片、timeout 与汇总门

V208实施需把 `test:e2e` 从 scaffold 78 激活并建立 fail-closed jobs；建议分片如下，实际文件名可按实现一次冻结：

| job | shard | timeout | 依赖/输出 |
|---|---|---:|---|
| `v208-contract-unit` | contract、report policy、state machine、mutation 4 shards | 15 min/shard | 无集群；生成clean、coverage/mutation |
| `v208-integration-core` | auth/PG/Cowork、REST/WS/Redis、artifact/S3、chaos 4 shards | 35 min/shard | 真实服务/两replicas；JUnit+logs |
| `v208-e2e-journey` | Chromium/WebKit/Firefox × fixture/real IdP 6 shards | 40 min/shard | trace/video/screenshots；主链 |
| `v208-e2e-faults` | stream/auth/multitab/artifact-security 4 shards | 45 min/shard | tc/toxiproxy或等价故障；浏览器证据 |
| `v208-nonfunctional` | perf/flake/egress/isolation/rollback 4 shards | 60 min/shard | 样本与资源/cleanup报告 |
| `v208-aggregate` | 不分片 | 15 min | 校验所有预期 shard、hash、三态、0 skip/retry/cleanup |

- 单测试默认≤60s；真实 turn/chaos可显式≤120s，suite teardown≤5min。超时即 FAILED并仍执行有界cleanup，不自动增加timeout重跑。
- shard assignment由版本化manifest稳定决定；missing/duplicate scenario、空 shard、仅成功 shard上传、矩阵 fail-fast导致证据丢失都让 aggregate FAILED。jobs可 `fail-fast:false` 收齐证据，但最终 required check必须失败；这不等于容忍失败。
- 集群/Secret/IdP缺失让对应 shard写 BLOCKED/2并上传报告；aggregate取最坏状态：任一FAILED→FAILED/1，否则任一BLOCKED→BLOCKED/2，否则全PASS→PASS/0。GitHub required job对1/2均不通过。
- CI artifacts命名含 runId+source短SHA+project+shard，保留策略按安全批准；上传前redaction/hash，绝不上传Secret、kubeconfig、完整业务内容。

## 14. 发布、回滚、返工与四角色交接

- CI激活顺序：contract/unit→fixture integration→fixture IdP三浏览器→真实IdP/真实集群→chaos/nonfunctional→required aggregate。V208 gate未全PASS不得进入V209。
- 回滚测试/CI接线只能回到前一可运行版本，不能把required gate退为78/N/A或降低阈值来发布。测试环境先停新journey、收齐证据、exact cleanup；产品candidate保持V201～V207已接受feature flags，不由V208私自切生产。
- 若V208发现缺陷：归属V201 auth、V202/203 data、V204 runtime、V205 config、V206 bridge、V207 artifacts；原任务完成修复Red→Green→Reviewer→Tester→协调者重新ACCEPTED后，V208以新accepted SHA从AC-01全量重跑。不能只在E2E加等待/容错。
- rollback rehearsal验证关Web/bridge/artifact flags、停止ticket/preview、drain/abort active turn、保留forward schema/PG数据/Redis retention/S3 immutable blob、Electron路径不受影响；不得down/drop/反灌SQLite/放公网。
- Developer只追加`开发记录.md`，不写Reviewer/Tester结论；Reviewer必须独立checkout/env/runId和mutation；Reviewer `APPROVED_FOR_TEST` 后独立Tester新增corners并形成test commit；失败回Developer或重开上游，修复后必须重新Reviewer。
- Tester `PASSED` 只把candidate与packageHash交总体协调者。协调者核对V201～V207接受链、40 AC、strict evidence、cleanup后决定V208是否ACCEPTED及是否创建V209；任何V208角色不得自行勾总表或签署V209。
