## 2. 目标、源码边界与非范围

### 2.1 目标

交付与 Electron preload 同形的浏览器 `window.electron` 实现：全部 registry `invoke/request-response` 叶通过生成映射走 REST，全部 registry `on/send` 叶通过一条产品 WS（ASR 使用其独立 WS）进入桥本地回调；具备 OIDC access 注入、短期一次性 ticket、`seq/sinceSeq/ack`、断连补发、retention gap snapshot、背压、多 tab 与明确降级。Electron 继续使用既有 preload，Web 与 Electron 双跑但不双发副作用。

### 2.2 建议物理边界

## 3. 全 registry 的 invoke → REST

1. 接受后的 `BridgeRegistry`、`RouteRegistry`、`InventoryBridgePaths` 与 generated `ElectronBridgeMap` 是唯一映射；测试按集合相等枚举全部叶。当前 321/184/158 只作基线快照。
2. 每个 `disposition=ga && transport=rest` 叶必须解析到存在的 operationId，并从生成 metadata 映射 path/query/body/header、success status、response schema、error schema；不得使用任意 URL 的通用 `fetch` 绕过 allowlist，也不得按 IPC 字符串手写 switch 副本。
3. Web bridge 保留旧方法签名/返回形状，adapter 只做明确的兼容转换：例如分页、服务端 canonical `requestId` 与旧 `clientRequestId`、逻辑工作区路径。tenant/user 永远来自 AuthContext，不接收调用方覆盖。
4. REST trust boundary 对成功体和统一错误信封都做 generated runtime validation。未知错误码、HTML proxy error、空 body、错误 content-type/压缩损坏均转稳定 `CONTRACT_VIOLATION`/`TRANSPORT_ERROR`，不得按 `message` 分类。
5. access token 只在内存，通过 `Authorization` header 注入；`Accept-Language` 来自 Web i18n。请求日志不得含 Authorization、Cookie、prompt、附件、code、ticket 或 response body。

### 3.1 幂等、超时、取消、abort、retry

- GET/HEAD 等安全查询才可自动网络重试；mutation 仅当 route 契约声明 idempotency 且携 `Idempotency-Key`、canonical body hash 相同才可重试。相同 key+不同 payload 必须 409；未知结果先经 V204 receipt/session query 对账，绝不换 key 重发 `start/continue`。
- 401 只允许 Auth refresh single-flight 后重试一次；refresh 失败/撤销立即清内存并进入登录。403/404/409/413 不重试；429/502/503/504 仅对安全或幂等请求按 `Retry-After` + 有界指数抖动重试。
- 每个 operation 使用集中 timeout policy；caller signal、路由卸载、显式取消、全局 teardown 与 timeout 合并到一个 `AbortController`。HTTP abort 只停止本地等待，不得谎称服务端业务已取消。
- `api.cancelStream(requestId)` 固定 REST `POST /api/v1/model/stream/{requestId}/abort`；Cowork stop/permission/AskUser/compact 走各自 registry REST route。不得新增 WS cancel 帧。cancel/abort 与 complete/error 并发只能产生一个有效终态。
- 重试预算、attempt、backoff、timeout/abort reason 可观测；timer、controller、in-flight promise 在完成、logout、切租户、卸载后全部清理。

## 4. 全 ChannelRegistry 的 on/send → WS

验收必须枚举接受后的 **每个** `ChannelRegistry` row，不能只测 10 个 Cowork 事件。当前 32 项分组是：

- 产品 `/api/v1/stream`：1 个 `clientControl`、10 个 Cowork session 事件、4 个 `apiStream*`、13 个 user/resource 通知，共 28 项；
- 独立 ASR `/api/v1/asr/sessions/{asrSessionId}/stream`：`asrAudioChunk/asrPartial/asrFinal/asrError` 4 项。

若接受后的 registry 改数，equality gate 必须立即失败并要求实现或显式 disposition；不得以“未在本计划列出”跳过。

### 4.1 回调与订阅规则

- `cowork.onStream*` 只登记桥本地 callback。`activeSessionIds` 以引用计数产生 `subscribe/unsubscribe {sessionId,sinceSeq?}`；最后一个 session consumer 离开才 unsubscribe。
- 用户级事件随已认证连接恢复，只登记本地 callback；资源事件（至少 `filesChanged`）按规范化 params/refcount 产生 `subscribeEvent/unsubscribeEvent`。session/user/resource 不能串 registry。
- 4 个 `apiStream*` 按服务端 canonical `requestId` 本地多路分发，不发送订阅帧；旧 clientRequestId 只在桥内相关，不进账单、abort path 或审计主键。
- ASR 必须走独立 URL、ticket/scope 与双向 schema；V5 上游未具备时返回 accepted contract 的明确 `UNSUPPORTED_FEATURE`/`DEPENDENCY_NOT_READY`，不能把 ASR 混入产品 WS、伪造 partial/final 或静默成功。
- callback unsubscribe 必须幂等；一个 callback 抛错只隔离并记录该 callback，不得阻断其他订阅、改动其他 scope checkpoint 或造成全局 socket 重连。任意页面崩溃后的恢复正确性由权威 snapshot、幂等 reducer 与持久 checkpoint 保证，不把任意用户 callback 宣称为跨崩溃事务。

### 4.2 Cowork 10 事件与终态

精确 wire 集为 `message/messageUpdate/sessionStatus/contextUsage/goal/contextMaintenance/permission/permissionDismiss/complete/error`。runtime alias `contextUsageUpdate/goalUpdate` 只应在 V204 被归一；`delta/tool/thinking/done/abort` 不得进入 Web wire 集。

桥以 `(tenant,scope,seq,event identity)` 去重并保持 `message → messageUpdate* → complete|error`；`permission/permissionDismiss`、AskUser、context 与 goal payload 按 generated validator。`complete/error/abort` 竞态只生效一次；重复 terminal 可 ack/dedupe但不能重复 Redux finalize、toast 或 callback。

## 5. ticket、OIDC、CSRF/CORS/Origin

- ticket issuance 仅允许有效 V201 access bearer；body 严格只有 `sessions[]` 与 `resourceSubscriptions[]`，资源先逐项授权。不得传 tenantId/userId/userEvents/JWT/cursor/lastSeq。
- ticket TTL 使用 V105/V201 接受值（必须在短期 30～60 秒范围）、高熵 nonce、原子 single-use，并绑定 tenant、user/sub、auth `sid/jti`、session/resource scopes、**精确 Web app Origin**。多 tab 各取不同 ticket，绝不共享或缓存 ticket。
- REST ticket 请求与 WS Upgrade/首帧的 Origin 必须相同且命中精确 HTTPS allowlist；服务端在原子消费前重验 user/membership/sid/jti 撤销和 scope。ticket 不进 URL、subprotocol、localStorage、sessionStorage、日志、trace、错误或 analytics。
- WS open 后首帧只能 `{type:'auth',ticket}`；`authok` 前不得业务订阅/数据；过期、复用、scope 越权、origin 不符、认证超时使用 frozen protocol error/close code并 fail closed。
- Bearer REST 的 CORS 只放精确 app origin/header/method；不得 `* + credentials`。refresh/logout 继续使用 V201 HttpOnly Cookie，并强制 CSRF token、Origin/Fetch Metadata；bridge 不读取 refresh Cookie。OIDC state/nonce/PKCE 仍由 V201 管理，V206 不复制认证状态机。

## 6. seq / sinceSeq / ack / resume / retention / gap snapshot

### 6.1 连续 checkpoint 与 ack

- `seq` 为 Redis Stream ID 或等价单调字符串，按 `(milliseconds,sequence)` 数值二元组比较；不得字典序、`Number()`、`+1` 或与 DB message cursor/展示 sequence 混用。
- checkpoint key 以 `{auth subject/tenant, stream path, scope kind+id, tabStreamId}` 隔离；`tabStreamId` 是只含随机非秘密 ID、同一 tab 刷新可复用的 sessionStorage 标识。连接/subscription generation 是 checkpoint 记录中的 fencing 状态，不是换代后丢失旧 seq 的 key。只有 schema-valid、scope-valid、generation-current、已经过幂等 reducer 接纳的**最高连续 seq**才推进；乱序帧留在有界 reorder buffer，不能越 gap ack。
- checkpoint 与有界 event/terminal identity 去重摘要在 IndexedDB 同一事务持久化后才可 `ack`；ack 只发送最高连续 checkpoint，可批量/节流但在 terminal、pagehide 有界 flush。wire 是 at-least-once，UI effective-once 依赖 identity+seq 与幂等 reducer；任意页面 callback 不是 durability boundary，硬崩溃后先恢复权威 snapshot/state再 replay。ack 丢失允许重放，但不能重复 finalize、消息 append 或用户提示。
- checkpoint adapter 只存 seq/scope/version/time、generation fencing与有界identity摘要，不存 token、ticket、prompt/payload。账号/logout/tenant/contract major变化清理对应namespace；多tab不共享消费权、ticket或tabStreamId。

### 6.2 重连、retention 与 gap snapshot

- 每次断线都重新申请 ticket，认证后按 session/resource 最后连续 `sinceSeq` 重放；用户级事件自动恢复。单 tab 同时只允许一个 reconnect loop/socket generation。
- 必测断点：首帧前、message 中段、permission pending、complete 前、complete 收到但 ack/checkpoint 未持久、服务端已持久 terminal 后；每种均不得假 complete、丢 terminal或重副作用。
- retention 的时间/条数窗口由接受后的 V105/V204 服务端配置与响应定义，客户端不得假设“5 分钟/500 条”永远可用。边界测试覆盖刚好保留、刚过期、trim 与 replay/live 合流。
- `sinceSeq` 在保留窗口内：服务端按 `seq > sinceSeq` 有序补发，再接 live；客户端去重交界帧。窗口外/trim/corrupt gap：服务端发 frozen `STREAM_GAP`/等价控制结果和可验证 resume/snapshot metadata。
- 收到 gap 后，bridge 暂停该 scope 增量 dispatch，调用 V204 权威 session/message snapshot/query 对齐，原子替换/合并 reducer 状态，使用服务端返回的 snapshot watermark/resume seq 重新订阅；snapshot 失败保持显式 degraded/error并重试，不得越 gap、清空对话或拿缓存旧状态伪成功。
- 若 V105 接受契约没有 gap code、snapshot endpoint/DTO、watermark/resume 语义，V206 为 `BLOCKED_CONTRACT_CHANGE`，不能手写私有字段。

## 7. 重复、乱序、丢帧、弱网、背压、多 tab 与并发

- duplicate：同 `(scope,seq,event identity)` 同 payload静默去重；同 seq 不同 payload 视为 `STREAM_CORRUPTION`，停止 scope并 snapshot。
- out-of-order：有界 reorder；缺口超时间/条数/字节阈值进入 gap流程。未知非关键可选事件按 contract 忽略并诊断；未知关键 type或 envelope version=`UNSUPPORTED_EVENT_TYPE`，不得投错旧 callback。
- loss/weak network：offline/online、DNS失败、TLS断、代理 idle close、半开、延迟/抖动/packet loss、重复 close/error 只能触发一个带 jitter 的 reconnect loop；20s ping/45s pong 等具体值以 V105接受配置为准并用 fake clock测试。
- backpressure：队列同时受 item/byte/age 上限；只可合并同 session+message 的连续 `messageUpdate`（以及 accepted contract 明确允许的非关键采样），绝不丢 `permission/permissionDismiss/error/complete`。溢出进入 gap/snapshot；不能静默 drop、OOM或无限暂停 ack。
- 多 tab：每 tab独立 socket/ticket/subscription generation/checkpoint namespace；A tab unsubscribe/logout/close 不应取消 B tab订阅。跨 tab refresh 可由 V201 single-flight 协调，但 access/ticket 不经 BroadcastChannel 明文传播。
- 并发：StrictMode 双 mount、同 session 多 callback、两个 session、两个 requestId、用户/资源同名事件、unsubscribe 与 in-flight replay、active session 快切、logout 与 ticket/reconnect、cancel 与 complete 均有确定状态机和资源清理。

## 8. 降级矩阵与不可支持 Electron API

1. 以接受后的全部 `BridgeRegistry` leaves 生成机器矩阵：`ga-rest / ga-stream / browser-fallback / no-op-inapplicable / unsupported / deferred`；missing/extra/targetless-ga=0。当前 321 叶不能靠人工抽样验收。
2. `ga` 必须真实走 target；不能因为后端尚未实现而返回假 success。`deferred` 在 V2 未开放时走明确 `UNSUPPORTED_FEATURE`/`DEPENDENCY_NOT_READY`，不能 `undefined` 或空对象。
3. 有 Web 等价者（安全 `openExternal`、Clipboard、文件上传/虚拟工作区等）做 capability/permission 检测并保持旧签名；失败返回 frozen `browser_capability_missing`/fallback，不访问本机路径或 Node API。
4. 无 Web 等价者（指定本机 App、computer-use、VM/browser automation、local-service、桌面 OAuth/device integration 等）返回结构化 unsupported，UI不进入成功分支。只有 V105/`13` 明确标为“不适用”的窗口/托盘/更新器叶可做确定性 no-op；no-op 仍有可观测 capability 状态，不能假称实际执行。
5. 所有用户可见 unsupported/degraded 文案使用 zh/en i18n；正常降级可 resolve兼容 shape，不以未捕获 reject 崩溃，但必须包含稳定 code/messageKey/fallback。REST 501 仍用统一错误信封，不能把 bridge shape 当后端错误体。
6. V207 artifact 预览叶、V5 ASR/provider 叶只做 transport/disposition；不得在 V206 补业务实现或用 fixture 冒充支持。

## 9. schema、错误、可观测与 redaction

- REST request/response/error、WS control、全部 ChannelRegistry envelope/data 均在边界运行 generated Zod validator；验证失败不进入 callback/Redux，不推进 checkpoint，并按 fatality触发 scope resync或连接关闭。
- 错误分类只认 stable code：auth、validation、not found/permission、conflict/idempotency、rate/timeout、ticket/protocol、gap/corruption、unsupported、dependency、internal。HTTP status/message 不作为唯一分支。
- trace 贯穿 browser requestId/clientRequestId → REST operationId → V204 request/run → stream seq；指标至少含 request latency/retry/abort、socket state/reconnect cause、ticket result、subscription count、replay count/lag、ack lag、gap/snapshot、queue item/bytes/age、schema/unknown event、callback error。
- metric label 禁高基数原始 tenant/session/request/seq；日志/trace/report递归脱敏 Authorization/Cookie/token/code/verifier/state/nonce/ticket、URL query、prompt/tool args/附件、IdP/profile PII。ticket 仅可记录不可逆短 fingerprint；canary 扫描必须 0 命中。
- 客户端 telemetry 只发批准内网 OTel/日志 endpoint；无 Sentry/CDN/font/analytics/remote-config 公网 fallback。

## 13. 迁移、发布、回滚与风险升级

- **expand-first**：先部署兼容旧/当前 contract minor 的 REST/WS薄适配和 ticket consume → dark mode health/contract probe → Web bridge feature flag → 内部 canary → 全内部 Alpha。Electron preload 始终保持原路径。
- 双跑是“不同客户端各走自身桥”，不是同一 UI 请求同时发 Electron+REST；shadow 只允许无副作用 query，对 start/continue/permission/cancel 禁止双发。
- API/WS drain：停止签发新 ticket/接收新订阅，现有连接发 frozen reconnect/close 语义并有界等待 ack/outbox；客户端重连前取新 ticket。滚动期间兼容当前和上一个 contract minor。
- bridge teardown/flag off/logout 清理 socket、ticket promise、retry/heartbeat/timer、handlers、AbortController、queue和对应 IndexedDB namespace；测试 Redis keys/tickets/streams/prefix 全清，`cleanup=true` 才可 PASS。
- 回滚顺序：Ingress/Web flag → Web bootstrap install → bridge runtime → API REST/WS adapters；保留 V201/V204服务、接受契约和向前兼容数据。不得回滚已发布 migration或只手改生成物。
