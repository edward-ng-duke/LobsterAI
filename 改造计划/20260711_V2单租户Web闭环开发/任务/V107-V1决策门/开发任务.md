## 2. 结果三态、禁止 waiver 与 GO 公式

### 2.1 唯一叶子状态

- `PASS`：目标断言在声明环境真实执行，阈值全过，证据同源、新鲜、完整、hash 可重算且签署有效。
- `FAILED`：输入与环境完整，但功能、安全、兼容、性能、完整性、证据或 cleanup 任一断言失败。
- `BLOCKED`：缺前置接受、Day-0、工具、权限、服务、RuntimeClass/节点、审计、预算、签署或可验证证据。

所有必需叶子只允许这三态。`SKIPPED / NOT_APPLICABLE / DEFERRED / PARTIAL / CONDITIONALLY_PASS / ACCEPTED_WITH_RISK / TEMP_GREEN` 不得进入 V1 分母；空矩阵、0 case、旧报告、截图、mock、fixture 或 exit 0 无 artifact 都不是 PASS。

### 2.2 无 waiver、无临时绿灯

- V107 不接受 CVE/供应链 waiver、过期前暂准、break-glass 公网窗口、临时 root/runc/Kata 代替 V102 gVisor、宽 NetworkPolicy、临时公网 provider/S3/OIDC、未签名镜像、测试删除/skip、性能阈值事后调整或“已知问题不影响演示”。任一存在即对应门 `FAILED` 或 `BLOCKED`，最终 `NO_GO`。
- V106 仅允许 canonicalization 后已被机器证明**不改变语义**的格式差异；任何需要 owner/expiry/忽略规则才能通过的差异都不是 V1.5 PASS。通配 `approved-differences`、secret/security 差异或待 V205 解释项直接阻断。
- 风险只能通过证据关闭，不能通过改严重级、延期、口头接受或在 V107 新增例外关闭。

### 2.3 机械 GO 公式

```text
GO :=
  every(P00..P04, V101..V106).accepted
  AND every(cumulativeGate).PASS
  AND every(V1.0..V1.5).PASS
  AND V1ScopedOpenRiskCount = 0
  AND blockerCount = 0
  AND failedCount = 0
  AND waiverCount = 0
  AND staleEvidenceCount = 0
  AND sourceMismatchCount = 0
  AND environmentMismatchCount = 0
  AND checksumMismatchCount = 0
  AND missingSignatureCount = 0
  AND cleanupOutstandingCount = 0
```

任一项不成立即 `NO_GO`；尤其**任一 `BLOCKED` 都不得 GO**。没有人工 override 字段，不允许投票把 0 改 1，也不允许“先启动 V2、后补签名/证据”。

### 2.4 Phase A 授权与 Phase B requalification

- Phase A 机械公式通过后发布 `V2_DEVELOPMENT_AUTHORIZED`，绑定 `authorizationSourceSha`、`v1ControlledSurfaceHash`、environmentRevision、packageHash 与签名。该记录是不可变的阶段授权，不因 V201～V208 正常新增源码、测试、forward migration 或已通过的 C201/C207 additive checkpoint 自动变为 EXPIRED。
- 只有 V1 受控面变化才暂停新增 V2 工作：P00～P04/V101～V106 实现或接受被撤销，runtime/image/chart/harness/NetworkPolicy/registry trust root/批准 endpoint/集群环境漂移，或出现新的 V1 P0/P1/BLK。此时发布 Phase A revocation，并回对应 V1 owner。
- V208 产品/测试修复全部落定后冻结 `v2CandidateSourceSha`，执行 Phase B：在最终候选重放累计门、V1.0～V1.6 sentinel、风险/签名/cleanup，校验 Phase A pedigree 与 `C201/C207` checkpoint。通过时发布 `V1_REQUALIFIED_FOR_V2_CANDIDATE`；失败不追溯抹除历史授权，但阻止 V208 接受与 V209 开始。
- Phase B 的 24 小时 freshness 只约束最终候选 requalification/签署窗口；正常 V2 开发期不要求不断重签 Phase A。

## 4. same-source / same-environment / runId / freshness 规则

### 4.1 联结键与禁止拼证据

V107 的联结键冻结为：

```text
candidateSourceSha
+ environmentRevision（动态门）或 evidenceEnvironmentClass（V105/V106）
+ artifact tuple hashes/digests
+ upstreamTaskId
+ role
+ runId
+ startedAt/finishedAt
```

- 不得从不同 source SHA、cluster UID、CNI/RuntimeClass/StorageClass/endpoint registry/budget revision、镜像 digest 或 config/contracts hash 中各挑一项 PASS。
- 每个上游 task 的 Developer、Reviewer、Tester runId 必须互异；V107 Developer、Reviewer、Tester 的 decision runId 也必须互异。复制目录、改文件名或仅换 summary ID 不算新 run。
- Reviewer/Tester 必须亲自触发命令和动态 sentinel，不得把 Developer 原 runId 重签成独立重放。CI retry 需新 invocation/runId，并保留旧失败历史。
- 一个 gate 内需同 run 的原始事件、指标、manifest、审计与 cleanup 不得跨 run 补洞；若任一子产物缺失，整个该 run 非 PASS。

### 4.2 新鲜度与失效事件

- 协调者冻结本阶段 source SHA 后才开始 replay。Phase A authorization window 与 Phase B requalification window 各自从首个 replay `startedAt` 起最多 **24 小时**；24 小时是生成/签署窗口，不是 Phase A 对正常 V2 开发的自动失效时钟。
- P00～P04/V105/V106 静态/loopback 门、V101～V104 各任务最近一次完整 Developer/Reviewer/Tester 包、V107 Reviewer/Tester sentinel、风险快照和环境 attestation 均须在该 window 内新鲜生成；历史接受包只作血缘，不能作为 final claim 的唯一证据。
- 动态域的完整独立 Tester 包必须同 `candidateSourceSha + environmentRevision` 且在 24 小时 window 内；V107 sentinel 不能给陈旧完整矩阵“续期”。发生任何下列事件则 affected domain **立即过期，必须回原任务全矩阵新 runId 重跑**：源码/镜像/chart/contracts/config/harness 变化；集群/CNI/RuntimeClass/node image/StorageClass/CSI/S3/model/endpoint registry/NetworkPolicy/NAT/审计/预算变化；凭据或信任根轮换；cleanup 未完成；时钟不同步；发现 P0/P1/证据篡改。
- 过期证据状态是 `BLOCKED_STALE_EVIDENCE`，不是 PASS；重新 hash 旧文件不会刷新时间。

## 5. 证据包、hash 与签署契约

证据目录固定为 `docs/poc/20260712_V107_V1GoNoGo决策证据/<decisionRunId>/`；禁止 `outputs/`。先写同父目录临时目录，完整验证后原子 rename。至少包含：

```text

candidate-lock.json        # candidateSourceSha、image/chart/contracts/config/harness immutable tuple

```

- `checksums.sha256` 覆盖除签名容器自身外的全部声明 artifact；`packageHash` 是 canonical evidence manifest + checksums 的 SHA-256。子任务包 hash 必须与其 `checksums.sha256` 重算一致。
- 签署 payload 固定绑定 `schemaVersion/candidateSourceSha/environmentRevision/decisionRunId/packageHash/decision/signedAt/role/agentOrCIIdentity`。签名可以是批准的 CI OIDC/cosign/detached signature；仅在 Markdown 填名字或截图不算机器签名。
- GO 至少需要 V107 Developer evidence publisher、独立 Reviewer、独立 Tester、SEC（V1.0/V1.2）、PLAT（V1.0～V1.3）、DATA（V1.3）与总体协调者签署。一个身份不得代签两个独立角色；签署缺失/过期/撤销即 `BLOCKED`。
- 签名后任何 byte、索引、风险状态或 decision 变化都会改变 `packageHash`，旧签名全部失效，必须重新生成包并重走 Reviewer/Tester。禁止修改 package 后只补 `checksums.sha256`。
- 证据严禁 token/key/PEM/kubeconfig/Secret data、真实租户 prompt、未脱敏模型响应/流日志；secret scan 命中即 FAILED、隔离证据并轮换可能凭据。

## 10. 原子决策、回滚与过期规则

### 10.1 原子决策

- `decision.json` 正式状态只有 `GO` 或 `NO_GO`；准备中的 `PENDING/BLOCKED` 只存在临时目录，不得成为正式 GO 文件。
- 先锁依赖/候选/环境，重算三态与风险，校验 artifact/hash/cleanup，再收集绑定同一 `packageHash` 的签名；最后重新计算 GO 公式并将整个临时包原子 rename。
- 任一并发 writer、同 decisionRunId、签署后变化、rename 中断或公式/人工字段不一致都拒绝发布。已有正式决定保持不可变，后续决定用新 runId 追加，不覆盖历史。
- `NO_GO` 必须写清每个 failed/blocked claim、owner、返工任务、失效传播和下次最早重验条件；不能只有“暂缓”。

### 10.2 GO 后启用与回滚

- 只有总体协调者核对 Tester `PASSED`、签名和 packageHash 后，才能在 `00-总体完成表.md` 更新 V107，并允许 V201/V202 创建实现提交。Developer/Reviewer/Tester 不得自行勾表。
- GO 不授权发布、部署或外部试用，只解锁 V2 开发批次。V2 仍须自己的 Reviewer/Tester/阶段门。
- GO 后若需回滚，协调者发布新的 `decision=NO_GO, reasonCode=PREVIOUS_GO_REVOKED` 决策记录（保留旧包），停止尚未开始的 V2 工作；已产生的 V2 分支/数据由各任务按自身回滚，不由 V107 destructive reset。

### 10.3 授权撤销与最终候选过期

- Phase A 只因 V1 受控面或其环境/信任输入变化而撤销：image/chart/runtime/harness、P00～P04/V101～V106 接受、NetworkPolicy/NAT/RuntimeClass/StorageClass/CSI/S3/model/endpoint registry、registry trust root/凭据、预算、风险/BLK、签名、cleanup、公网/secret/数据损坏。正常 V201～V208 源码、测试、forward migration与已接受 C201/C207 additive delta不在此列。
- 发生 V1 受控面事件时，已开始的 V2 工作停在安全边界，affected V1 task 新 runId 完整重验并发布新的 Phase A。禁止只改 `expiresAt` 或重签旧包延寿。
- Phase B 绑定精确 `v2CandidateSourceSha`；任何产品/测试/contract/migration/image/config/environment 变化都会使该 requalification 过期，须在新最终候选重跑 Phase B。旧 Phase A 授权审计记录保持不可变，不因 Phase B 失败被改写。
