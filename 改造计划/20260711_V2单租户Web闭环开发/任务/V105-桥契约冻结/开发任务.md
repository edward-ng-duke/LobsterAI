## 3. Cowork 10 事件四方精确冻结

“四方”固定为：① contract registry 的 wire type、IPC topic、AsyncAPI channel、payload Zod；②生成 AsyncAPI message/envelope/extensions；③ `ElectronBridge`/bridge map 方法与精确 callback 签名；④现有 `IElectronAPI` + preload `satisfies ElectronBridge` + runtime forwarder 输入。四方集合、名称、optional、返回 unsubscribe、payload 字段/nullable 必须双向相等，不只检查字段名 substring。

| wire type | IPC topic | AsyncAPI channel | bridge method | 旧桥 callback data 的关键形状/归一 |
|---|---|---|---|---|
| `message` | `cowork:stream:message` | `cowork/stream/message` | `cowork.onStreamMessage` | `sessionId,message,beforeMessageId?` |
| `messageUpdate` | `cowork:stream:messageUpdate` | `cowork/stream/message-update` | `cowork.onStreamMessageUpdate` | `sessionId,messageId,content,metadata?` |
| `sessionStatus` | `cowork:stream:sessionStatus` | `cowork/stream/session-status` | `cowork.onStreamSessionStatus` | `sessionId,status` |
| `contextUsage` | `cowork:stream:contextUsage` | `cowork/stream/context-usage` | `cowork.onStreamContextUsage` | `sessionId,usage`；runtime `contextUsageUpdate` 只作输入 alias |
| `goal` | `cowork:stream:goal` | `cowork/stream/goal` | `cowork.onStreamGoal` | `sessionId,goal`；runtime `goalUpdate` 只作输入 alias |
| `contextMaintenance` | `cowork:stream:contextMaintenance` | `cowork/stream/context-maintenance` | `cowork.onStreamContextMaintenance` | `sessionId,active` |
| `permission` | `cowork:stream:permission` | `cowork/stream/permission` | `cowork.onStreamPermission` | `sessionId,request`；工具授权与 AskUser 共用 |
| `permissionDismiss` | `cowork:stream:permissionDismiss` | `cowork/stream/permission-dismiss` | `cowork.onStreamPermissionDismiss` | data 仅 `requestId`；tenant/session 在 envelope 顶层 |
| `complete` | `cowork:stream:complete` | `cowork/stream/complete` | `cowork.onStreamComplete` | 精确对齐接受 SHA 与真实旧桥的 `sessionId`、`claudeSessionId` nullable/optional 语义 |
| `error` | `cowork:stream:error` | `cowork/stream/error` | `cowork.onStreamError` | `sessionId,error`，不得以 message 文案分类 |

`StreamEnvelope` 顶层必须固定 `type/version:1/tenantId/sessionId?/requestId?/seq?/idempotencyKey?/emittedAt/data`；`seq` 是字符串补发/去重游标。旧 callback 为兼容可能在 `data` 重复携带 `sessionId`，wire 路由仍只信 envelope scope；测试须分别验证两层，不得把 `permissionDismiss.sessionId` 塞入 data。

## 4. Bridge transport 与 disposition 冻结

1. 每个 GA `invoke`/request-response leaf 必须指向至少一个真实 `RouteRegistry.operationId`，transport=`rest`；冻结 method/path/path-query-body/response/error/auth/status，禁止指向 WS message 或虚构 route。
2. 每个 GA `on/send` leaf 必须指向真实 `ChannelRegistry.messageType`，transport=`stream`；冻结 scope=`session|user|resource|asr`、replayable 与 unsubscribe 返回形状。参数化 `api.onStream*` 仍按 `requestId` 在桥本地分发。
3. session callback 注册本身不得发送无 scope channel 订阅。会话 wire 订阅只由 `activeSessionIds` 引用计数生成 `subscribe/unsubscribe`；资源事件只由 `resourceSubscriptions` 生成 `subscribeEvent/unsubscribeEvent`；用户级事件随 auth 自动恢复。
4. targetless leaf 不得仍为 GA。它必须显式为 `browser-fallback`、`deferred` 或 `unsupported`，并冻结 fallback/`UNSUPPORTED_FEATURE` 语义；不得返回 `undefined`、假 success 或静默 no-op。浏览器 fallback 不能伪装成 REST，unsupported 不能伪装成 stream。
5. `BridgeRegistry`、GA inventory、route/channel targets 必须双向相等；一个 leaf 多 target 可保留，但每个 target 都须存在且用途明确。`api.onStreamError` 与 `cowork.onStreamError` 等同名 leaf 不得折叠。

## 5. Ticket、控制帧与安全冻结

- `POST /api/v1/stream/ticket` 只能由有效 access bearer 申请；body 仅允许 `sessions[]` 与 `resourceSubscriptions[]`，后者 V1 至少为 `files:changed + {workspaceId,path?}`。拒绝 `tenantId/userId/JWT/userEvents/cursor/lastSeq`、重复 session、完全相同 resource scope、未知字段/频道；用户事件不得由客户端列 scope。
- ticket 内部绑定 tenant/user、已逐项授权的 session/resource scope、nonce、`expiresAt`；TTL 30～60 秒、原子一次性消费。过期=`TICKET_EXPIRED`、复用=`TICKET_ALREADY_USED`，跨 scope/不存在统一不泄露存在性的拒绝。
- WS URL 不得出现 JWT/access token/ticket query；不得用 `Sec-WebSocket-Protocol: bearer.*` 或长效 bearer 首帧。首帧只允许 `{type:'auth',ticket}`，未认证超时/业务帧先到须 `protocolError` 并关闭。
- ticket REST 使用显式 `Authorization` header，不能把 refresh/session Cookie 当 ticket 授权；因此跨站自动 Cookie 请求无权签发。冻结 Origin/CORS allowlist；若未来允许 Cookie ticket issuance，必须另行加入 CSRF header/双提交契约，不能静默改变本基线。
- client control 精确为 `auth/subscribe/unsubscribe/subscribeEvent/unsubscribeEvent/ping/ack`；server control 至少冻结 `authok/pong/protocolError` 及 close code `4401/4408`。只有 `sinceSeq`，不得出现 `cursor/lastSeq`。

## 6. `sinceSeq`、终态与并发语义

- replay 读取严格为同一 scope 中 `seq > sinceSeq`，按 Redis Stream id 二元组顺序比较，不能字符串排序或“整数 +1”；checkpoint 只推进到连续、已处理帧。
- live/replay 合流需按 `(scope,seq)` 去重；重复帧不得重复 append/finalize。乱序或 gap 不得越过缺口推进 checkpoint，应触发 bounded reorder 或 `STREAM_GAP`/全量对齐。
- 断在首帧、中段、`complete` 前、收到 `complete` 后尚未持久化 checkpoint 四种情形都要测试；最终 `complete` 恰好生效一次。未知非关键事件忽略并诊断，未知必需/不兼容版本报 `UNSUPPORTED_EVENT_TYPE`，不得投递错误旧 callback。
- 单 tab 同 session 多 callback 使用引用计数，最后一个取消才发 unsubscribe；两个 tab/两个 socket 各取不同一次性 ticket，任一 tab 取消不得影响另一 tab。session 与 resource registry 不得混淆。
- unsubscribe 与 replay/live in-flight 竞态以订阅 generation/barrier 丢弃旧 generation 帧；`api.cancelStream` 仍走 REST `/api/v1/model/stream/{requestId}/abort`，不得新增 WS cancel 分支。
- 背压必须有有限队列/阈值和可观测结果；只可合并同 session+message 的连续 `messageUpdate`，不得丢 `permission/error/complete`。溢出需 `STREAM_GAP`/重同步，不能静默丢帧。心跳超时后重连并用最后连续 `seq` 续传。
