## 3. PVC workspace 与 artifact S3 职责边界

| 数据/行为 | 权威落点 | V104 硬性语义 |
|---|---|---|
| `/workspace/project` 项目文件、agent 产物 | PVC `tenants/{tenantId}/ws-{workspaceId}/project` | POSIX 真源；文件服务与单个持锁 Pod 看到同一子路径 |
| `/workspace/state` 的 AGENTS/MEMORY/SOUL/IDENTITY | 同一 workspace PVC 的受控子树 | 不向普通文件 API/用户列表暴露；只有单一持有 fencing token 的 Pod/Config Sync 可写 |
| OpenClaw `openclaw.json`、gateway token/port、临时 transcript/log/cache | Pod `/state`、Secret、stdout、emptyDir/另行冻结的运行时恢复面 | **不属于 workspace PVC/S3 镜像**；不可把完整 HOME 挂载来绕过归属 |
| `files/{relPath}` 热文件镜像 | S3 `tenants/{tenantId}/ws-{workspaceId}/files/...` | PVC 仍权威；S3 最终一致，预览/下载前须做新鲜度+内容 hash 校验 |
| artifact/缩略图/分享快照 | S3 `.../artifacts/{artifactId}` / `.../thumbnails/...` | 对象级不可变快照；只使用具体 key，禁止桶/前缀级 URL |
| 大文件上传暂存 | S3 `tenants/{tenantId}/uploads/{uploadId}/...` | S3 分片直传→checksum/尺寸校验→PVC 同目录临时文件→fsync+rename→`landed`；未 landed 不可见 |
| 备份/冷归档 | S3 `tenants/{tenantId}/backups/ws-{workspaceId}/...` | 带 manifest/checksum/版本/保留策略；恢复先进新 PVC/prefix 验证，不就地覆盖源 |

边界规则：

- workspace 挂载/租约粒度是 `workspaceId`，不是 session/agent/单文件。文件服务可与一个 RW OpenClaw Pod 使用 RWX，但第二个 OpenClaw Pod 只能排队或只读。
- 普通路径从 `/workspace/project` 的相对路径开始；客户端不能传宿主绝对路径、PVC subPath 或完整 S3 key。
- 不做 PVC↔S3 “同步双权威”。镜像失败不回滚已 fsync 的 PVC 正常写，而是记录待同步 mutation 并有界重试；S3 旧版绝不能在断连后自动覆盖较新 PVC。
- 生产 runtime 只访问批准内网 object-store endpoint；无公网 NAT/代理兜底，不运行时安装/下载依赖。V104 记录这条内网 allow，但不代替 V103 全套 deny 证据。

## 4. 试验协议与一致性语义

### 4.1 单写者租约与 fencing

- 使用一套 `rt:workspace-lease:{tenantId}:{workspaceId}` Redis CAS/TTL 与 PVC `.lobsterai/lease.json` 恢复自证；lease file 不得单独授予 RW，不再建第二套锁 key。
- 每次授予生成单调可比的 fencing token/version；每个写入、上传落地、镜像/删除 mutation 都携带 token。旧 holder 在 TTL 后恢复时必须被拒绝并退出/降只读。
- 两 Pod 同时 acquire 只有一个 RW；另一个按 FIFO 排队、超时明确失败或降级 readonly，不允许无限卡住。连续续租失败进入 drain，禁止新写。
- 文件服务与 OpenClaw Pod 的写权归属要在矩阵中明示；不得一边声称单写者，一边让两个不校验 token 的进程并发覆盖同一文件。

### 4.2 POSIX 写、`rename` 与 `fsync`

受测安全更新路径为：同目录临时文件→完整写入→`fdatasync/fsync(file)`→同文件系统 `rename`→`fsync(parent directory)`→发布变更事件/同步 mutation。

- 读者只能看到旧完整版或新完整版，不能看到部分长度/串行内容；跨文件系统 `rename` 必须拒绝或转显式 copy+verify 流程，不冒充原子。
- `fsync(file)+fsync(dir)` 成功返回后立即 kill Pod/重建并重挂，最终名称与 checksum 必须保留；在 fsync 返回前故障，不承诺最后一次写入必定保留，但必须保证不暴露损坏终态，且临时文件可审计/清理。
- 直接 overwrite、append、并发 create/delete/list/stat、目录项和权限/mtime 语义分别量测；不用文件数量相等代替内容 hash 对账。
- 必须记录 CSI/backend 对 close/fsync/rename 的实际观测，未能验证持久化语义时是 BLOCKED/FAILED，不从“POSIX 兼容”标签推导 PASS。

### 4.3 PVC→S3 镜像、大文件落地与幂等

- 每次镜像 mutation 有唯一 `mutationId`、workspace/fencing token、relPath、generation、size、SHA-256（可增补供应商支持的 checksum）与状态 `pending -> uploading -> verified -> committed | failed`。重试以 `mutationId` 幂等，同一 mutation 不产生重复对象、重复事件或重复配额结算。
- 只有 PUT/multipart complete 成功、对象精确 key HEAD/GET 的 size/checksum/version 与源一致时才 `committed`；不用 bucket listing 或 ETag 无条件代替 checksum。
- timeout/429/5xx/连接断开使用预冻结的有界指数退避+抖动和整体 deadline；不可重试的 4xx fail closed，可重试错误到上限后明确 FAILED。不得无限重试、吞错或把业务断言失败降为 BLOCKED。
- multipart 在任一 part 失败、complete 响应丢失、重复 complete 或 client 中断时，必须查询精确 upload ID 确认终态；无法验证完整性不得 landed。过期上传只按 upload ID/run prefix abort，lifecycle 作兜底，不删已提交对象。
- 大文件一律 S3 暂存→分片 checksum/完整尺寸校验→流式下载到 PVC 同目录临时文件→file+dir fsync/rename→`landed`。不把全文件读入内存，不让部分文件占用最终名称。
- 删除使用带 generation/fencing 的 tombstone，镜像 worker 不得让旧 PUT 在新 DELETE 后“复活”文件；PVC 断连导致空 listing/错误时禁止生成批量 S3 delete。

### 4.4 文件事件、配额、回收与恢复

- 事件主路径优先使用 OpenClaw 文件工具成功事件；inotify/fsnotify 跨 Pod 传播可作快路径，不可靠它唯一保证。丢事件时按 generation/mtime/hash 扫描补偿，重复/乱序事件幂等；观测 `files:changed` p95 使用 `08` 的 3s 目标，不可靠 watcher 失效后的无限轮询补绿。
- 配额先在合成 Redis 计数器按 tenant/workspace 原子预留，成功 landed/写入后结算，失败/超时/取消幂等释放；精确边界允许，超 1 byte 拒绝，并发不得超发。实际 `du`/S3 与计数对账，漂移超用量 1% **或**绝对值超 100MB 即触发纠偏/告警，不静默修成 PASS。
- 回收按软删 `.trash` →保留期→精确 PVC 路径+S3 object version/prefix 删除；冷归档必须先生成可恢复备份 manifest/checksum，再释放 PVC。任务 cleanup 不产品化注销流程，只能清本 run 合成数据。
- PVC 断连/节点迁移时，当前 writer 停止新写并显式 FAILED/drain；恢复后先按 lease/fencing、PVC/PV UID、workspace manifest/hash 对账，再允许写。不越权重挂、不因空目录重建 S3。
- 跨 Pod 重建：删除/强杀 Pod A，Pod B 以新 lease/fencing token 挂载同一 workspace；校验 fsync 已确认数据、`/workspace/state` 和 `/workspace/project` manifest 零差异，旧 A 的迟到写失败，S3 镜像不退版。只证明工作区重建，不声称当前 turn/会话 transcript 无损。
- 备份恢复：将某一已验证 backup 恢复到**新的短命 PVC 和新 run prefix**，校验 path/type/mode/size/checksum/manifest 后再宣布可切换；不得为测试就地覆盖原 workspace。

## 5. 真实候选矩阵与故障注入

### 5.1 矩阵主键和重复规则

矩阵主键至少为 `sourceSha + runId + StorageClass UID/provisioner version + PV/PVC UID + nodeA UID + nodeB UID + probe imageDigest + objectEndpointId + bucket/prefix + caseId`。每个必需候选单独运行，不得拼接不同候选/日期的最优结果。

- 功能/P0 case 必需项 100% 通过；延迟与吞吐重复次数在 Day-0 预冻结，报告 p50/p95/p99/max 与失败率。
- 任一 P0 失败则该 StorageClass 候选 no-go；所有候选失败则 V104 FAILED/V1 no-go，触发 workspace-service/FUSE/对象存储权威备选评审，不得降低安全/一致性标准。
- 某候选缺专用节点/故障注入能力是该单元 BLOCKED；若它是唯一必需候选，V104 总结论 BLOCKED。

### 5.2 必跑真实场景

| 类别 | 真实执行 | 硬断言 |
|---|---|---|
| 双向可见 | Pod A 写/Pod B 读，Pod B 写/Pod A 读；不同节点 | p95 < 500ms、p99 < 2s；size/SHA-256 一致 |
| 小文件并发 | 同 workspace 20 并发 create/overwrite/delete/read/list，多轮确定 seed | 无丢文件/脏读/部分内容/残留目录项；manifest/hash 零差异 |
| 单写者 | 两 Pod 竞争 RW，同文件和 `MEMORY.md` 并发；TTL 过期、续租失败、旧 Pod 恢复 | 唯一 RW；另一排队/只读；过期 token 无一次迟到写成功 |
| POSIX 原子性 | 跨 Pod 高频 read 期间 temp-write/fsync/rename；fsync 前/后 kill | 只见旧/新完整版；fsync 后重挂 hash 保留；fsync 前不暴露损坏终态 |
| 大目录 | 单目录 10k 文件 listing/分页/stat，同时有受控写 | p95 listing < 2s；无漏/重；不一次返全量 |
| watch/补偿 | 跨 Pod 写，丢/重/乱序 watch 事件，watcher 重启 | 支持则实测；不可靠时显式切补偿，最终 manifest 一致；`files:changed` p95 < 3s |
| PVC 断连 | 受控阻断 CSI/backend、I/O 延迟/超时、重连/重挂；不影响共享资源 | 写入有界失败、不假成功；无越权重挂；空 listing 不删 S3；恢复先对账 |
| S3 故障 | 内网 endpoint 断连、DNS/TLS 失败、延迟、timeout、429/5xx、响应丢失 | 有界 retry/幂等；PVC 真源不回滚；无重复结算/重复对象；失败可观测 |
| 部分/大上传 | 分片中断、缺/错 part、complete 响应丢失/重放；边界和大文件 | 未验证不 landed；重放唯一终态；过期 multipart 精确 abort；流式处理内存受控 |
| checksum/旧版 | 注入 bit flip、错 size/checksum、旧 mirror 延迟到达、delete/put 重排 | 不 committed/landed；新 generation/tombstone 不被旧操作覆盖 |
| 配额 | 精确边界、+1 byte、20 并发预留、超时/取消释放、对账漂移 | 不超发；释放一次；超额新写失败但读仍可用；漂移超门告警 |
| 路径/链接 | `..`/多层编码/绝对路径/盘符/NUL/前缀混淆/case+Unicode/symlink/hardlink/FIFO/device/socket | project root 外和特殊文件全拒绝；`state/.trash/.lobsterai-tasks/lease` 不泄露 |
| 跨 Pod 重建 | Pod A 写并 fsync，kill/删除，Pod B 新 token 挂同 PVC，A 迟到写 | 已确认 workspace manifest 零差异；A 被 fencing；S3 不退版；只丢未 fsync 的当前操作 |
| 备份/回收/恢复 | 软删、GC 重放、冷归档，恢复到新 PVC/prefix，再清本 run | 保留策略生效；无越界删除；恢复 manifest/checksum 零差异；cleanup 幂等 |

### 5.3 故障注入安全约束

- 优先使用集群已批准的 Chaos/CSI fault API、仅本 run NetworkPolicy overlay、内网 Toxiproxy/故障端或短命测试后端。工具/权限/故障半径未签署时 BLOCKED，不自行阻断共享 CSI/S3/节点网络。
- 每个注入都先证明基线健康，记录开始/结束、作用对象 UID、预期与实际故障，有独立存储/网络侧证，不把目标本身宕机当阻断证据。
- 注入超过总 deadline、发现跨 run/跨 namespace 影响、数据泄露或无法恢复时立即中止，执行精确隔离和数据保护 runbook。

## 9. 数据安全、精确回滚与清理

### 9.1 数据保护红线

- 只用合成数据、专用短命 PVC/PV 和唯一 S3 run prefix；不挂生产/共享租户卷，不复用别人的 bucket prefix，不在故障注入时触及共享 CSI/S3 控制面。
- 删除/恢复/回滚前先对当前 PV/PVC/Pod/object/upload 的 UID、runId label、prefix 和 checksum manifest 做二次校验；任一不匹配即停止，不“尽量删”。
- 发现越前缀读写、旧 holder 迟到写、数据损坏、空目录覆盖或 Secret 泄露时立即停止后续 case，将资源降只读/隔离，保留脱敏失败证据，通知 PLAT/DATA/SEC 并轮换可能泄漏凭据。

### 9.2 精确回滚

- 代码按 V104 分 commit 倒序回滚；Helm fixture 回滚后重跑 P04/P03 相关门。不为 CI 变绿把 `poc:v1:check` 改回 skip/exit 78，不删失败测试，不放宽 root/路径/IAM/NetworkPolicy。
- 候选回退只到上一个已验收的 `StorageClass UID + CSI version + mount options + object endpoint/policy version` 精确 tuple，不用同名 StorageClass/浮动 S3 配置推导等价。
- 数据恢复采用“备份→新 PVC/新 prefix→全量 manifest/checksum 验证→显式切换”；不就地覆盖受损源，不删原卷/原 prefix 直到协调者与 DATA 签收。
- 若所有 RWX 候选 P0 失败，回滚结论是 V1 no-go 与 workspace-service/FUSE/对象存储权威备选评审，不把未验证的 local-path/RWO 当临时生产方案。

### 9.3 精确 cleanup

1. 先停 probe/mirror/upload worker，禁止新 mutation；等待已知有界任务终止，记录未完成项。
2. 只释放本 run lease/Redis quota 预留，使用精确 key/version 与幂等 CAS；不清空共享 Redis 前缀。
3. 只 abort 证据中声明的 multipart upload IDs，只删本 run 精确 object versions/prefix；禁止 bucket-wide delete、未过滤递归删。
4. 对 PVC 生成最终 manifest/hash/可选 snapshot，然后仅按预期 PVC UID+label 删短命 PVC；`Delete` reclaim 验证 PV 消失，`Retain` 则由 PLAT 按精确 PV/backend ID 另行签收，不由通用 cleanup 强删。
5. 只删 `runId + managed-by + UID` 匹配的 Pod/Job/NetworkPolicy/SecretRef/ConfigMap；禁止 namespace-wide delete、删 namespace、共享 StorageClass/Secret/NetworkPolicy。
6. cleanup 立即重放一次必须幂等；任一未清项/越界尝试记录在 `cleanup.json` 并阻断 PASS。历史脱敏成功/失败证据按 retention 保留，不随集群资源清理删除。

## 10. Developer → Reviewer → Tester 独立交接

- Developer 按 `Round N · YYYY-MM-DD HH:mm · Agent/commit` 追加 Red→Green→Refactor、命令/退出码/环境/证据，使用唯一 Developer runId，对 AC01～AC28 定位；不自审、不写 Reviewer/Tester 结论、不勾总表。
- 独立 Reviewer 必须读计划、全 diff、Red/Green 和原始存储/故障证据，且用**不同于 Developer 的新 runId**至少独立重放：preflight、双向可见、20 并发、租约+fencing、rename/fsync 后 crash、PVC 断连不删 S3、S3 timeout/partial/complete 重放/checksum、配额边界、路径/symlink、Pod 重建、恢复到新 PVC/prefix 和 cleanup 二次执行。无独立集群/故障权限时写 `BLOCKED`，不得 `APPROVED_FOR_TEST`。
- 任一 P0/P1、缺有效 Red、本地/mock 冒充 RWX/S3、无 fsync 或故障侧证、双写/越界/空目录覆盖风险、证据可伪绿或 cleanup 不精确，Reviewer 必须 `CHANGES_REQUESTED`。
- 仅 Reviewer 使用新 runId 得出 `APPROVED_FOR_TEST` 后，独立 Tester 才开始。Tester 必须用**不同于 Developer/Reviewer 的第三个全新 runId**重放完整候选+P0+故障+安全+恢复矩阵，并新增至少一个前两者未覆盖角度：symlink swap TOCTOU、fsync 后 node drain、complete 响应丢失、旧 tombstone 重排、Unicode 归一化冲突、配额释放竞态、watcher 重启、证据原子中断或 Retain PV 清理逃逸。
- Reviewer/Tester 的环境、StorageClass/S3、节点、故障权限不可用时结论是 `BLOCKED`；环境完整但数据/性能/安全/恢复/清理断言失败是 `FAILED`。不得复用旧 runId 写通过。
- Tester `FAILED` 退同一 Developer，修复后必须重新独立 Reviewer 且再用新 runId。只有 Tester `PASSED` 且协调者核对三套独立证据、候选 go/no-go、数据安全和 cleanup 后，协调者才可更新总体表。
