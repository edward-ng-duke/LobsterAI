## 3. Cowork application service：command / query / CRUD

### 3.1 唯一公共上下文

每个 public 方法首参为不可伪造的 `TenantRequestContext`，至少含 `tenantId/userId/traceId/authz`，由上层受信认证组件构造；body/query/sessionKey 中的 tenant/user 字段一律拒绝或忽略。context 必须传入 V203 transaction、Redis key、ownership route、event envelope 与 trace。缺失/过期/跨租户 context fail closed，不允许 admin/default tenant fallback。

### 3.2 Commands（有副作用）

| command | 原子语义 | runtime / owner side effect | 返回 |
|---|---|---|---|
| `startSession` | 校验 tenant 内 agent/workspace；同一 transaction 建 session、首 user message、server requestId/idempotency receipt、running 状态 | commit 后以稳定 requestId 调 `chat.send` | accepted `{sessionId,requestId}`，不等待 complete |
| `continueTurn` | tenant/session/agent 校验；会话锁；同 key 重放返回原 requestId，不重复消息/turn | 选定 active/primary sessionKey 后 `chat.send` | accepted `{requestId}` |
| `patchSession` | title/pin/goal 等 domain patch；runtime model/config patch 只使用 accepted contract 支持项 | 需要时 `sessions.patch`，DB/runtime 失败补偿有记录 | accepted domain result |
| `stopSession` | 幂等标记 cancelling；路由 owning replica | 精确 `chat.abort {sessionKey,runId}`；终态后重复 stop 可读成功 | 不伪造 complete |
| `respondPermission` | 校验 tenant+session+requestId+kind；原子 claim pending request | owner route 后真实 approval/AskUser resolve RPC；重复同答幂等，异答 conflict | applied/terminal result |
| `compactContext` | 与 active turn/锁策略一致，记录 request correlation | `sessions.compact`，不能用本地摘要冒充 | usage/result |
| `deleteSession` | 先拒绝/停止 active turn；删除策略消费 V203 | 对该 session 的全部 gateway keys 调 `sessions.delete`，失败可重试/补偿 | deleted/idempotent |

`startSession` 固定为“创建 + 首 prompt + 立即开跑”，不创建空会话；`continueTurn` 是独立 command。command 不返回 raw gateway payload、Prisma row、token 或 sessionKey。所有服务端 `requestId` 在首次执行前生成并稳定复用；客户端 idempotency key 只作 tenant+operation scoped 去重输入，不能直接充当全局 requestId。

### 3.3 Queries（无业务副作用）

- `listSessions/getSession/getMessages` 只读 V203 facade，使用其 opaque `(created_at,id)` cursor；禁止 offset/sequence 作为新 API 游标。
- `getContextUsage/getGatewayHistory/reconcileSession` 可调用 `sessions.list/chat.history`，但不得创建 turn、改变 ownership 或隐式发 prompt；允许更新明确标注的 cache/observation，不改变业务记录。
- session 不存在与跨租户不可见统一映射 `NOT_FOUND`；query 不泄露另一个 tenant 的 agent、sessionKey、runId、message count 或 timing 差异。
- query 返回 accepted contract/domain DTO；V206 负责 HTTP envelope/status，V204 不自行发送 response。

### 3.4 transaction、outbox 与锁

- session/message/status/idempotency receipt/outbox intent 在同一 PG transaction 内写入；Redis publish 不能成为 PG commit 的前置原子假设。outbox relay 至少一次投递，消费者按 event identity 幂等。
- 若接受后的 V202/V203 schema 没有 V204 所需 `command_receipt/runtime_turn/pending_permission/event_outbox` 支撑，V204 可新增**向前兼容、tenant_id 非空**的 V204 migration；不得修改 V202/V203 已发布 migration。若公共 schema 所有权未决，记 `BLOCKED_SCHEMA_CONTRACT_GAP`。
- 每 tenant+session 同时仅一个生成 command。锁 key 使用已冻结 tenant 前缀，token 校验释放并有看门狗；进程 crash 后 lease/owner TTL 可回收。锁失败映射 accepted `SESSION_BUSY/IN_PROGRESS`。
- PG commit 后、`chat.send` 前 crash 的 receipt 必须可恢复；不能留下 running session 永久无 turn，也不能重启后重复首 user message。

## 4. 真实 OpenClaw RPC、correlation 与多 key

### 4.1 仅允许的真实 RPC

| RPC | V204 用途 | correlation / retry 规则 |
|---|---|---|
| `chat.send` | start/continue/受限续接 | `sessionKey` + 稳定 `idempotencyKey=requestId`；ack 返回 runId 后原子绑定 |
| `chat.abort` | stop/timeout/cancel | 精确 sessionKey+runId；重复 abort 幂等，终态后不改写终态 |
| `chat.history` | history/reconnect/reconcile | 只读，有界 timeout/retry；校验 key 所属 tenant/session |
| `sessions.list` | usage/key/run reconciliation | 只读、有界；不能用全局 list 泄露他 tenant row |
| `sessions.patch` | accepted session runtime patch | 仅幂等 patch 可重试；timeout 后先 read-after-write 确认 |
| `sessions.compact` | manual compact | command receipt 去重；timeout 后查询 checkpoint/usage再决议 |
| `sessions.goal` | goal command/snapshot | action+requestId 关联；accepted payload 为准 |
| `sessions.delete` | session cleanup | 对全部 scoped key；重复不存在视为幂等，但越权不视为不存在成功 |

`session.start/session.continue/session.history/session.delete` 一律禁止；测试须 mutation 替换成这些假 RPC 并失败。

### 4.2 correlation 状态机

每个 turn 持久/可恢复关联至少包含：

```text
tenantId + sessionId + requestId + idempotencyKey
  -> selected sessionKey + gateway rpc request id + runId?
  -> owningReplicaId + ownerGeneration
  -> state(pending|sending|running|cancelling|completed|errored)
  -> raw chat/agent sequence checkpoints + terminal event id
```

- ack 前事件可按 sessionKey+idempotencyKey 暂存到有限 orphan buffer；ack 后绑定 runId 并重放。无法唯一归属的帧不得猜 session。
- 同 runId 映射到不同 tenant/sessionKey、同 requestId 映射到不同 payload hash、sessionKey 所属关系变化均 fail closed 并发安全告警。
- timeout 分连接、RPC ack、首语义事件、turn terminal、permission 和 drain；阈值从配置注入并有上下界，不允许测试看结果后修改。
- retry 仅限明确幂等/只读操作。`chat.send` ack timeout 属结果未知：先用同 key 的 `sessions.list/chat.history` reconciliation；只有能证明未接受时才以**同 idempotency key**重发。禁止生成新 key 后盲重试。
- cancel 与 retry 竞态：一旦进入 cancelling，不再启动新 send；late ack 立即绑定后 abort；late complete/error 通过 terminal CAS 恰一次生效。

### 4.3 sessionKey[] 与隔离

- 1 个 Cowork session 可有多个 OpenClaw key；mapping 以 `(tenantId,sessionId,sessionKey,agentId,kind,status)` 为准。tenant 权限来自可信 mapping，不从可伪造 key 字符串推断。
- start 使用 accepted V102/V203 规则建立 primary managed key；continue 优先 active turn key，否则选择显式 primary。存在两个可写候选且无唯一 primary 时返回 conflict，不“取数组第一个”。
- history/usage 逐 key 合并时保留来源并确定性排序；delete 枚举全部 key；patch/abort/permission 必须命中明确 key/run/request。
- agent 必须属于同 tenant 且 enabled；`logical_id=main` 在不同 tenant 可同时存在，内部 UUID 不得泄露或串用。workspace/context/capsule 同样按 tenant+session 取，不得由全局 cache 只按 sessionId/key 命名。

## 5. 10 事件归一与关键时序

V105 接受 SHA 的 `CoworkStreamRegistry` 是唯一输出集合。runtime alias 只在 adapter 内存在，wire 不出现 `delta/tool/thinking/done/abort/contextUsageUpdate/goalUpdate`。

| wire type | 真实输入/领域触发 | V204 硬语义 |
|---|---|---|
| `message` | chat delta 首实体、agent assistant/tool event、已持久 user message | 稳定 messageId；先落 PG/intent，再出 event；同消息只 create 一次 |
| `messageUpdate` | chat delta/snapshot、assistant stream、tool result/backfill | 必须已有/同时补齐 message；仅同 message 连续 update 可 coalesce；最终 metadata 不丢 |
| `sessionStatus` | command 状态、agent lifecycle、reconcile | 状态机合法迁移；running/idle/completed/error 不倒退 |
| `contextUsage` | `sessions.list`/history/compaction observation | runtime `contextUsageUpdate` 归一；usage 与 tenant/session/key 对齐 |
| `goal` | `sessions.goal`/goal snapshot | runtime `goalUpdate` 归一；null/optional 按 accepted schema |
| `contextMaintenance` | compaction stream/tool lifecycle | start/end 配对；timeout/crash 要补显式 end/error，不永久 active |
| `permission` | exec/plugin approval、真实 AskUser/MCP callback | 精确 `kind:'tool'|'ask'` 语义；pending request 持久且 owner 可路由 |
| `permissionDismiss` | approval resolved、answer applied、timeout/cancel/terminal cleanup | `data` 仅 `{requestId}`；tenant/session 在 envelope；恰一次 |
| `complete` | chat final/lifecycle end 经 history/DB 对账 | 最后 messageUpdate 已持久且 outbox append 后才 terminal；每 request/turn 恰一次 |
| `error` | chat/lifecycle/gateway/timeout/validation terminal failure | 稳定 code 分类；不能与 complete 双终止；不含 secret/raw provider 错误 |

### 5.1 `message → messageUpdate → complete` 规则

1. start/continue 的 user message 与 running 状态先 commit；接受响应不等待流。
2. 同 assistant/tool message 的首 update 若先到，进入有限 reorder buffer；补出 `message` 后才发 update，不能让 V206 reducer接到悬空 update。
3. final content/history reconciliation、PG final metadata、所有 terminal 前必需 outbox 均完成后，才 CAS 写 `complete`；`complete` 后内容帧为 late duplicate，记录诊断但不发布。
4. chat final 与 lifecycle end 同时到只产一个 complete；chat error/lifecycle error/timeout/abort 与 final 竞态只允许一个 terminal outcome。
5. user-requested abort 产生明确 status/permissionDismiss；是否输出 error/complete 严格按 V105 accepted contract，禁止新增 `abort` wire type或伪造成功 complete。

### 5.2 permission / question / abort / error

- 工具授权使用 accepted `PermissionResult` 判别联合；“本会话始终允许”只通过 `updatedPermissions`，不自创 `allow_always`。
- AskUser 使用同一 permission respond application port 的 `kind:'ask'` 分支；answer key 集合、重复回答、超时、session terminal、owner crash 均测试。若 pinned OpenClaw 无真实 AskUser callback/resolve 能力，报告 `BLOCKED_RUNTIME_CAPABILITY_GAP`，不能以本地 callback 假通过。
- 未知/过期 requestId、requestId 属于别 tenant/session、owner generation 陈旧必须返回可路由错误或总线转发，禁止现状 controller 的静默 return。
- permission resolved/timeout/abort/complete/error 都清理 pending 并出恰一次 dismiss；重复同 result 返回原结果，冲突 result 返回 conflict，不二次调用 gateway。

## 6. sequence、duplicate、out-of-order、backpressure、reconnect

### 6.1 两层 sequence 不混用

- **raw gateway seq**：按 `(tenantId,sessionId,runId,source=chat|agent,seq)` 去重/重排。chat 与 agent 的 seq 域分开，不能用一个 lastSeq 相互覆盖。
- **Cowork stream seq**：由 tenant+session Redis Stream `XADD` 返回的字符串 ID；写入 `StreamEnvelope.seq`，不使用 number、DB message sequence 或 raw gateway seq。
- raw frame 无 seq 时使用稳定 event identity（runtime event id；若无则受控 hash+run+type）并标记低置信；不能只按 payload 文案去重。

### 6.2 有界重排与 gap

- duplicate 只忽略完全相同 identity；同 identity 不同 payload 是 corruption，`FAILED`/安全告警。
- out-of-order 在冻结的 frame 数/毫秒窗口内排序；不能像现状 `seq <= lastSeq` 那样把晚到但必需帧静默丢弃。窗口到期仍缺口则进入 reconciliation，并输出 accepted `STREAM_GAP`/error 语义，checkpoint 不越 gap。
- pending/orphan/reorder/permission/update/outbox 队列均有 item/byte/age 上限与 metrics；溢出不能静默 drop。

### 6.3 backpressure

- 只允许合并同 tenant+session+messageId、同 generation 的连续 `messageUpdate`，必须 leading+latest trailing；`message/permission/permissionDismiss/error/complete/sessionStatus` 不可丢。
- Redis/PG relay 慢时暂停/限速 gateway reader或进入 bounded spill/outbox，不无限堆内存。达到硬限后停止接受新 turn、标记 gap/error并保持已有 terminal 可恢复。
- 30,000,000-byte gateway frame 硬限和 29.5MB send 安全线不得放宽；超限 fail before send，附件交后续已冻结引用路径，不塞 WS。

### 6.4 backend gateway reconnect

- backend WS 心跳断开后指数退避+jitter，有最大次数/总时长；单实例只允许一个 reconnect flight。drain/配置重建时禁止自动重连抢 ownership。
- 重连后校验 gateway identity/version/attestation与 owner generation，再通过 `sessions.list/chat.history` 对 active turn 对账；不能假设 raw gateway 完整 replay，也不能凭 `Pod Running` 判 ready。
- 若 final 已在 history 但事件未处理，按 persisted correlation 补 final message/complete；若状态不可证明，发明确 retryable error/中断提示，不制造 complete。历史与工作区零丢失，in-flight turn 最多按 D9 丢当前 turn。
- Redis Stream 的产品补发、ticket 和浏览器 `sinceSeq` 由 V206/V208 验收；V204 只保证 durable、连续、tenant-scoped producer 与 replay/read port。

## 7. 多副本 ownership 与 command bus

- 唯一 registry 复用 `rt:session:{tenantId}:{sessionId}`，记录 owningReplicaId、commandChannel、generation、sessionKey/runId、lease expiry；禁止另建无 tenant 前缀的第二套 owner key。
- owner claim/renew/release 使用 CAS/lease token；旧 generation 的 event/command/result 全拒绝。实例 crash 后新 owner必须先 reconcile，不能双 owner 同时 `chat.send`。
- permission/stop/compact/patch/send 等回指 command envelope 带 tenantId/sessionId/requestId/commandId/ownerGeneration/payloadHash/trace context；每副本 command inbox 幂等。
- 未命中 owner：若 turn 终态则返回终态；若 lease 可恢复则 bounded claim+reconcile；否则 accepted error。禁止本地 Map 未命中直接 return。
- 生产 Redis 故障时 fail closed：不接受新副作用 command；已持有 gateway 的实例进入有界 drain并保存 PG 状态，不能偷偷退进程内模式。

## 8. 错误映射、可观测与 secret redaction

### 8.1 错误映射

- 内部分层为 validation/authz/not-found/conflict/busy/database/runtime-unavailable/gateway-auth/gateway-timeout/gateway-protocol/stream-gap/cancelled/internal；外部只映射 V105 accepted `ErrorRegistry`。
- repository error 按 V203 accepted mapping转换；OpenClaw 原始 message 只作脱敏 cause，不以人类文案决定错误码，不向客户端透传 stack/SQL/path/provider body。
- 若 `SESSION_NOT_OWNED`、gateway timeout 等缺少可表达的 accepted code，立即 `BLOCKED_CONTRACT_CHANGE` 并重开 V105；不得塞入 `INTERNAL_ERROR` 掩盖可恢复语义或私加 enum。
- REST status/envelope 序列化由 V206 做；V204 返回 typed domain error，保留 retryable/userVisible/i18nKey 元数据。

### 8.2 observability

- trace：`api handoff → Cowork command → PG → owner bus → OpenClaw RPC → raw event → normalize → outbox → Redis`，关联 traceId/tenantId/sessionId/requestId/runId 的**受控/哈希维度**。
- metrics 至少覆盖 command latency/result、active turns、RPC latency/timeout/retry、owner miss/generation reject、raw duplicate/out-of-order/gap、buffer bytes/overflow、outbox lag/retry、Redis publish、terminal duplicate prevented、reconnect、redaction violations。
- 日志用英文模块 tag、结构化字段和等级；热路径逐帧不打 info。正常生命周期 log，recoverable warn，失败 error 并把 error 对象放最后。
- 证据报告可按 tenant/session/trace 检索，但 Prometheus label 禁原始高基数 requestId/runId/sessionKey；trace/log 中按策略采样。

### 8.3 redaction 与证据

禁止记录/持久到证据：Authorization/JWT/refresh token/WS ticket/gateway token/provider key/Secret/kubeconfig/PEM、URL query credential、完整 prompt/system prompt/model response/tool input/output、附件 base64、绝对用户路径。仅允许长度、类型、批准的 model ID、状态、latency、usage、不可逆 hash和合成 nonce。

redaction 必须递归处理 Error cause、header、URL、nested payload、数组、JSON string、日志格式化失败和第三方 gateway error；敏感 canary 注入日志/trace/report 后扫描为 0。用户可见错误另走 accepted i18n key，不能把脱敏后的内部 debug 文案直接展示。

## 9. 生产完全断公网

- V204 production Pod 只连接批准内网 PG/PgBouncer、Redis、OpenClaw gateway/Cowork callback、OTel collector 和 DNS；endpoint 由 allowlist/Service identity 配置，不接受请求体覆盖。
- 禁止 provider 公网、公共 DNS fallback、开发机 port-forward/localhost、`0.0.0.0/0` 代理兜底；禁止运行期 npm/pnpm/yarn/npx/pip/git/curl 下载依赖、插件或 skill。
- 所有 Node/OpenClaw/Redis/OTel 依赖在测试/镜像构建阶段锁定并进入内部镜像、lockfile、SBOM/digest；生产启动缺包/证书即 fail readiness，不动态联网修复。
- V204 集成验收在 V103 已通过的网络基线上重复最小探针：批准内网 gateway/PG/Redis 可达，公网/metadata/未批 RFC1918 不可达。真实内网模型或 gateway 缺失是 `BLOCKED`，不能临时放公网。

## 13. 迁移、发布、回滚与风险升级

### 13.1 向前兼容迁移

- V204 支撑表/索引/constraint 采用 expand-first，全部 tenant-scoped 表 `tenant_id NOT NULL`，遵循 V202/V203 RLS/复合 FK/双键规则；migration 固定 hash，真实 PG17 升级/重复 deploy/旧 app 兼容测试。
- 新列先 nullable/backfill/验证再收紧（若涉及历史数据）；不修改已发布 migration、不手工改 `_prisma_migrations`。Redis key/version 新旧 relay 可短期双读，但**不得双发 chat.send**。
- rollout 顺序：schema expand → disabled service → shadow query/真实 gateway smoke（不发用户 turn）→ 小流量单 owner → full V204 producer → V206 后续接 transport。feature flag 默认关，readiness 在依赖缺失时 fail closed。

### 13.2 回滚

1. 停止接受新 Cowork commands，返回维护/可重试 typed error；不让请求落到旧/新两套 runtime同时执行。
2. 对 active turns 按预算 drain；到期精确 abort，flush final/outbox，释放 owner/lock，保留 PG session/message/receipt/outbox和 Redis retention。
3. 关闭 V204 compose/feature flag，回滚 app/library release；Electron 旧路径从未被切改。V206 尚未实现时无浏览器 route 需要回滚。
4. 保留 expand schema和已提交数据，不 down/drop、不反灌 SQLite；停止 relay 前记录 checkpoint，恢复时幂等重放。
5. 回滚后重放 contracts/V203/桌面 build、owner 清理、无双发/无 orphan/无 secret 证据。cleanup 失败为 FAILED。

### 13.3 风险升级/阻断

以下任一触发立即阻断依赖 V206/V208：真实 gateway raw 语义与 V105 不兼容；AskUser 无真实能力；V203 facade/schema 无法原子承载 receipt/outbox/pending permission；同 session 多 key 无确定选主；unknown result 无法安全 reconcile；终态重复/丢失；跨 tenant 泄漏；必须放公网；backpressure 无界；R1 迫使修改巨大 legacy adapter。需要改大文件时先按根 AGENTS 提精确抽取职责/模块/API/迁移/测试/风险方案并获确认，不在 V204 drive-by refactor。
