## 2. 源码审计事实、任务边界与禁止项

### 2.1 当前实现事实

- `src/main/libs/openclawConfigSync.ts` 为 3502 行桌面生产实现：直接 import `electron`/`fs`，一次 `sync()` 内读取约 25 个 live getter，混合 providers/models、agents、IM/bindings、MCP、plugins/skills、sandbox/session/browser/cron、workspace 文件、secret env、existing config、session store 与落盘副作用。
- 当前单文件写用 `tmp-${Date.now()} + rename`，没有多文件/双卷事务、版本/CAS、分布式 writer lease；`previousBindingsJson` 是进程内状态。损坏 existing config 多处被忽略/覆盖，不能直接成为服务端并发协议。
- `src/main/main.ts` 在 `getOpenClawConfigSync()` 组装 SQLite/KV/IM/MCP/Skill live getters，并用进程内 Promise queue 串行化；它比较 secret env、bindings 和 `OpenClawConfigImpact` 后 hot reload 或停止/重启桌面 gateway。其诊断会打印 secret 值前缀，服务端严禁照搬。
- `src/main/libs/openclawEngineManager.ts` 负责本机 state/token/port、spawn 与 invalid-config restart suppression。V205 只把其真实 config/gateway 行为当 oracle，不把 Electron manager 移入容器。
- `apps/runtime-orchestrator` 当前仅有 `/healthz` 壳；`charts/lobsterai/templates/sandbox-pod.yaml` 尚无 config-sync initContainer，workspace 仍是 fixture `emptyDir`。V205 在既有 app/image 内增加 entrypoint，不创建第五个 deployable。

### 2.2 本任务做什么

1. **原样复用 V106 纯 renderer**：服务端与桌面 shadow 必须 import 同一实现、同一 snapshot schema、同一 canonical/secret/AGENTS renderer；允许为 workspace package 可见性做一次机械搬迁/重导出，但须保持 V106 golden 字节与 import audit 全绿，禁止 server fork。
2. 新增 server snapshot provider：在 PostgreSQL tenant transaction 内一次性构造公开 snapshot 与 `SecretRef` 计划，禁止 renderer 回调 DB/live getter。
3. 新增 V205-owned revision/apply ledger、CAS 与审计 migration；不替代 provider/IM/MCP/skill/plugin 等权威领域表。
4. 在既有 `apps/runtime-orchestrator` 镜像增加 `config-sync render|validate|apply` 子命令/initContainer entrypoint；HTTP `/healthz` 壳与子命令共镜像，但 config sync 不是独立服务。
5. 增加受限双落点 effects adapter：动态 config → `/state`；托管 workspace state → `/workspace/state`；普通项目文件仍仅 `/workspace/project`。
6. 交付 diff/no-op/hot-reload/rebuild **决策结果与可执行入口**。V205 不实现 V206 WebBridge/controller；活跃 turn 的上层 drain/续接只调用 V204 已接受契约，不在 V205 发明第二套 Cowork 控制器。

## 4. PostgreSQL snapshot、SecretRef 与 revision ledger

### 4.1 一致性与 tenant 边界

- provider 使用 V204/V203 接受的非特权 app role 与 `withTenantTransaction`，在 `REPEATABLE READ READ ONLY` 事务中设置 `app.tenant_id/app.user_id`；禁止 admin URL、事务外延迟 query、raw client 泄漏、SQLite fallback。
- request 只接受已验证的 `tenantId + sessionId + workspaceId + agent logical_id/UUID`。session、workspace、agent、config profile 必须在同 tenant 且双键往返；不存在与跨租户统一 fail closed，不泄露对象存在性。
- 一次事务读取所有领域行、排序并深复制，计算 `sourceHash`；读取期间另一事务修改任一域，当前 snapshot 保持一致，下一 revision 才可见。renderer 只接 plain frozen data。
- `capturedAt/nowIso`、path flavor、runtime version/digest、approved internal service URLs、mount roots、plugin/skill inventory 均显式注入，不读宿主 `process.env/cwd/platform/home`。

### 4.2 snapshot 分母

公开 snapshot 必须覆盖并由 completeness checker 与 PG source registry 双向核对：

- providers/models：只指向批准内网模型 proxy，含模型能力/参数/default/agent override；凭据为会话短期 `SecretRef`，不含 upstream key。
- agents：PG UUID + tenant-local `logical_id`、main、enabled、model/identity/system prompt/skills、workspace 关联；稳定 main-first，拒绝重复/非法 ID。
- IM/bindings：全部平台与 multi-instance metadata、agent binding、account key；credential 只存 ref，重复 prefix/规范化碰撞失败。
- MCP：stdio/http/SSE、预装 command/args、批准内网 URL、canonical headers/env refs；禁止运行时下载与任意公网 endpoint。
- skills/plugins：供应链 allowlist、固定 digest/version、tenant enable/config、mount/load path；未知 plugin config secret 字段 fail closed。
- sandbox/session/browser/tools：服务端强制 policy、resource/runtime class、SSRF、heartbeat/memory/dreaming/embedding；用户输入不能放宽集群策略。
- workspace：`/workspace/project` 与 `/workspace/state` 分离；main/non-main managed AGENTS、SOUL/IDENTITY/USER/MEMORY/HEARTBEAT 的 owner/effect inventory；existing user section 由持锁 initContainer 的 local-state provider 读取并带 hash。
- cron/commands/callback：server scheduler 归属、内网 AskUser/media/MCP callback、gateway CLI bind 与固定 port。与桌面差异必须是 exact pointer/file 的 approved difference，不得 wildcard。

若任一域在 accepted PG schema/repository 中没有权威 owner，provider completeness gate 必须 `BLOCKED_SERVER_SOURCE_SCHEMA`；V205 不能用单个无约束 JSON blob掩盖领域缺口。

### 4.3 SecretRef 规则

- PG、公开 snapshot、ConfigMap、golden、plan、manifest 与 audit 只保存 opaque `secretRefId/kind/version/envName`；secret 值只从批准的 K8s Secret/CSI/内网 secret backend 注入 init/main container 内存或 env，永不进入 argv、文件、日志或 diff。
- ref ID、envName、kind 有 allowlist 与 tenant ownership；ref version 参与有效 secret fingerprint，值本身不参与公开 hash。引用集合与渲染的 `${ENV_NAME}`、实际注入 key 三向相等；多/少任何一项均失败。
- provider、embedding、IM、MCP headers/env、plugin config、callback/gateway token 全覆盖。未知疑似 secret key（token/key/password/secret/cookie/auth 等）默认阻断。
- secret 轮换形成新 desired revision；日志只记 ref 的不可逆短 hash、kind 和变更类别，绝不打印值、长度、前后缀或可逆编码。

### 4.4 V205-owned forward migration

V205 可追加而不得改写已发布 migration：

- `runtime_config_snapshots`：immutable `tenant_id/target_id/revision/schema_version/renderer_version/runtime_version/runtime_digest/public_snapshot_hash/secret_ref_set_hash/created_at` 及受控 public snapshot/ref metadata；禁止 secret value。
- `runtime_config_targets`：`tenant_id/target_id/workspace_id/desired_revision/applied_revision/applied_plan_hash/last_good_revision/last_good_plan_hash/fencing_token/status/error_code/updated_at`，带 tenant FK、RLS ENABLE+FORCE、唯一键与非负/单调 checks。
- 状态写采用 CAS：`WHERE desired_revision=:expected AND fencing_token=:token`；same revision+same hash 幂等，same revision+different hash 为 corruption，旧 revision 永不覆盖新 revision。
- migration 须做空库、accepted V202/V203 upgrade、重复 deploy、真实 catalog/RLS/drift、锁预算和中断恢复；生产无 destructive down。若 source domain schema 缺失，先 BLOCKED，不在此 migration 偷建替身领域表。

## 5. Orchestrator/initContainer 与 apply 协议

### 5.1 入口与输入

- 同一 `lobster-runtime-orchestrator` 镜像增加显式子命令；无参数仍运行现有 service shell，`config-sync render|validate|apply --request-file ...` 运行一次后按三态退出，不监听第二端口。
- request/public snapshot 从只读投影/ConfigMap 文件读取；secret 从只读 Secret volume/env 读取；禁止把大 JSON/secret 放 argv。输入 schema、size、owner/mode、duplicate key、NUL、symlink 均在读取前校验。
- initContainer 使用与主容器兼容的 UID/GID/fsGroup（当前镜像基线 `10001`），只获两受限根写权限；不持 K8s API token、PG admin secret 或公网能力。若 leader/materializer 需要 K8s/PG 权限，留在 runtime-orchestrator 主进程，不下放 sandbox initContainer。

### 5.2 双落点与 owner

- `/state/openclaw.json`、commit manifest、仅会话动态状态写 Pod `emptyDir`；gateway token/port 由 Secret/env，不写 workspace。
- `/workspace/state/<agent-scope>/AGENTS.md|SOUL.md|IDENTITY.md|USER.md|MEMORY.md|HEARTBEAT.md` 写 workspace PVC 的托管区；`/workspace/project` 永不被 config sync 写。
- AGENTS managed marker 精确保留 V106 规则：最多一个 marker、用户区不损坏、managed 区重建幂等、LF/单尾换行；main/non-main 不串写。MEMORY/USER 等 ownership 依 V106 inventory，未被 renderer 管理的文件不得顺手覆盖。
- 目录默认 `0700`、config/manifest/temp/backup `0600`；workspace 文本按已冻结共享组策略 `0600/0640`，owner 必须是 runtime UID/GID，禁止 world-writable/setuid/device/FIFO/socket。

### 5.3 staging、验证、提交与恢复

双卷不能一次 rename，冻结协议如下：

1. `PREPARE`：在每个目标文件同一文件系统、受限隐藏 staging 目录写完整 bytes；`fsync(file)`、校验 next hash/size/mode/owner，再 `fsync(dir)`。写入前验证 allowed root、realpath、relative path、expected previous hash、lease/fencing/revision。
2. `VALIDATE`：在 staging root 上运行 schema/secret/reference scan，并用 pinned OpenClaw runtime 对真实 `openclaw.json` 做启动/读取验证；未通过时目标 rename 数必须为 0。
3. `COMMIT_WORKSPACE`：持 workspace writer lease，以固定路径排序逐文件原子 rename；每步 journal/preimage hash。任一步失败立即按 preimage/LKG 恢复；恢复不完整为 `FAILED_PARTIAL_APPLY` 并 quarantine，绝不启动 gateway。
4. `COMMIT_STATE`：在 `/state` 同法提交 config；最后写 state generation manifest。
5. `BARRIER`：所有目标重新 readback hash、fencing/revision 后，最后原子写 `/state/.lobsterai-config-commit.json`。主容器 entrypoint 在启动 gateway 前复核 manifest、workspace generation/lease、plan hash 与 runtime digest；不匹配即退出。
6. `FINALIZE`：CAS 更新 PG applied/LKG ledger 后才清本 invocation temp/旧 backup；cleanup 只能按 runId 删除自身内容，重放幂等。

准备阶段失败不得改目标；commit 中断必须有精确 changed/restored/unrestored 清单。不得先删旧文件再写，也不得在 rename 失败后返回 `changed=true, success=true`。

### 5.4 锁、leader election 与多 Pod 单写者

- runtime-orchestrator 多副本使用 Kubernetes Lease（或已接受等价 leader elector）选出 materializer/reconciler leader；只 leader 创建 revision ConfigMap/Secret 引用和推进 desired state。lease identity/renew/expiry 必须可审计。
- 每个 `tenantId/workspaceId` 另有 D5 writer lease + 单调 fencing token；粒度是 workspace，不是 agent/session。只有持最新 token 的 Pod/initContainer 可 RW 挂载并写 `/workspace/state`，其他请求排队或只读，不能靠 leader 单点假定互斥。
- PG CAS 是第二道门；leader 丢租约、网络分区、暂停后恢复、旧 initContainer 晚到、两个 Pod 同 revision、同一 session 重试，均不能越过新 fencing token。
- 锁获取顺序固定：leader ownership → target revision row → workspace lease → state/workspace file locks；超时有界且返回 BLOCKED/FAILED，不无界等待。SIGTERM 在 prepare/commit 任一点均有恢复测试。
- 同 revision+same plan 仅一方应用，其余 `NO_OP_ALREADY_APPLIED`；同 revision+不同 plan/hash 立即 corruption FAILED；更旧 revision `STALE_REJECTED`。

## 6. diff、no-op、reload/rebuild 与 LKG

### 6.1 canonical diff 与动作矩阵

- diff 基于 V106 canonical plan/manifest，不比较注入时钟或 secret 值；输出 exact JSON Pointer/file、old/new hash、impact reason。无字段/文件/有效 secret ref version 差异为 `NO_OP`，不写文件、不更新 meta、不 reload/rebuild。
- 初始安全矩阵：provider/model/有效 secret rotation、MCP、plugin/load path、agent/workspace/sandbox/session/gateway auth、runtime/schema version → `POD_REBUILD_REQUIRED`；IM binding 若仅 V204 控制面消费 → `CONTROL_PLANE_ONLY`，若进入 gateway config 则按真实验证结论 reload/rebuild。
- skills enable/managed AGENTS 等候选 `HOT_RELOAD` 只有在 pinned OpenClaw 真实 watcher 测试证明 PID 不变、config 已采纳、行为在冻结 SLA 内生效且不丢状态后才允许；未证明/未知字段一律升级为 rebuild，禁止乐观 reload。
- V205 输出决策并提供 init/apply入口；运行中 rebuild 调用 V204 接受的 workload/drain/capsule/sessionKey 契约。不得在本任务修改 V206 controller，也不得宣称当前 turn 零丢失；D9 口径是至多丢当前 turn、历史/PVC 零丢失。

### 6.2 失败、回滚与 last-known-good

- 每个 successful generation 记录 public/plan/file hashes、renderer/schema/runtime version/digest、secret ref versions、fencing token、gateway validation result；不保存 secret value。
- render/validate/prepare 失败：目标与 LKG 不变；已运行健康 Pod 继续使用 LKG且标记 desired drift，不触发 restart storm。新 Pod 默认不以旧 revision 冒充成功启动。
- commit 失败：按 journal 恢复 LKG/preimages并验证 bytes；成功恢复返回 FAILED（非 PASS），当前健康 Pod可继续。恢复失败 quarantine target、阻止 main container、告警人工恢复。
- gateway 在新 generation 启动/真实 reload 验证失败：停止新实例，恢复兼容的 LKG并重新验证。仅当 `schema/runtime/secret refs/lease` 兼容且 policy 明确时可标 `LKG_FALLBACK_DEGRADED`；绝不返回正常 PASS或把旧 config 当新 revision applied。
- 防 restart loop：同 plan hash 的失败计数/退避有界；invalid config 不自动无限重启。回滚 action、operator、reason、from/to revision 可审计。

## 7. 真实 OpenClaw、golden/shadow、安全与兼容验证

### 7.1 pinned OpenClaw gateway 真实 config 验证

- 使用 `package.json` pinned `v2026.6.1` 且绑定实际 runtime image digest，不用自制 JSON schema/mock parser代替。
- initContainer 生成 staging config 后，以成品镜像内真实 `/opt/openclaw/openclaw.mjs gateway --allow-unconfigured --bind lan --port <test-port>`（或 pinned runtime实际等价入口）加载它；注入 test-only SecretRef、批准内网/保留域 endpoint，断言 gateway ready/health、config 无 validation error、插件/skills/MCP/agent路径可解析，再有界停止。
- 对未知 top-level、非法 provider/plugin/MCP/binding/path/placeholder mutation，真实 gateway 必须拒绝且主容器未启动；捕获 invalid-config restart suppression，不能只断言进程存在。
- HOT_RELOAD 候选必须在真实 gateway 运行中修改 generation，断言 config/行为已采纳、PID未变、重复帧/turn状态不丢；rebuild 类必须证明旧进程 drain/退出、新 digest/revision ready。无批准内网模型时 config boot 可测，真实 turn 项仍 BLOCKED，不能外连供应商。

### 7.2 golden 与 shadow 双跑

- 直接消费 V106 sanitized fixture/manifest。PG fixture → server snapshot provider → **同一个 V106 renderer** → plan，与 V106 public snapshot/golden 逐字节比对；server intentional differences 只能 exact pointer/file + DEC/owner/expiry 批准。
- scenario 分母含 providers/agents/全部 IM/MCP/skills/plugins/sandbox/workspace、minimal config、secret rotations、Windows/POSIX path flavor、managed AGENTS、existing state与全 effect inventory。
- shadow 在隔离 PG tenant、只读桌面 oracle/temp roots运行；同 source 同时产 desktop canonical、V106 renderer、V205 provider/apply dry-run 三份 hash。任何 unexplained diff、缺 scenario、fixture manifest 篡改均失败。
- 状态严格区分：`V106_FIXTURE_PASS ≠ V106_SHADOW_PASS ≠ V205_SERVER_GOLDEN_PASS ≠ V205_POD_APPLY_PASS ≠ V205_GATEWAY_BOOT_PASS ≠ V2_PASS`。

### 7.3 path、平台、secret、audit 与断公网

- pure renderer 继续在 Linux/Windows CI按显式 pathFlavor跑 drive/UNC/Unicode/空格/CRLF fixture；生产 effects adapter 是 Linux POSIX，遇 Windows drive/UNC/device path、绝对路径注入、`..`、双编码、NUL、symlink/hardlink/大小写/Unicode normalization collision 均 fail closed。
- 审计只记录 tenant/target/revision/run/plan/source hash、动作、changed pointer名、secret ref kind/hash、lease/fence、结果/error code/耗时；禁止 prompt、system prompt正文、secret/env值、header、token前缀、原 config、租户路径。异常对象统一 sanitizer 后再 log/metric/trace。
- 生产镜像静态与运行时测试禁止公网 DNS/IP、metadata、K8s API（initContainer）、非批准 RFC1918、package registry；批准 PostgreSQL/Secret/内网模型及 callback 用精确 allowlist。运行时安装命令/动态 plugin download 必须失败。
- fixture/evidence/image layer/SBOM/log/stdout/stderr/temp/backup/LKG 全面 secret sentinel 扫描，含 base64/URL/JSON escaped 变体；扫描报告只写类别与位置 hash。
