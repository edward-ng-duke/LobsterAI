## 4. 核心 repository 语义

### 4.1 sessions

- create 在 tenant transaction 内验证 owner/agent/workspace 归属并生成内部 UUID；默认值由 schema 与 mapper 双向断言。
- find/detail 不存在与跨租户目标都返回同一 `NOT_FOUND`，不暴露内部 UUID/存在性。消息窗口与 total 不允许 per-message 查询。
- patch 区分 omitted、`null`、空字符串和默认。非 status patch 默认不重排；status 只在值变化时触碰 `updated_at`；显式 touch/suppress 是 facade option。
- session list 的 keyset 编码完整排序 tuple：`pinned DESC`，pinned 区 `pin_order ASC`，unpinned 区 `updated_at DESC`，最后 `created_at,id` 确定性终结。cursor 含 version/direction/filter fingerprint；篡改、错 filter/tenant 都 `VALIDATION_FAILED`。
- search 冻结 trim、空 query、LIKE 通配符、ASCII case、Unicode normalization 和 agent filter。PG collation 与 SQLite 不同须进精确 approved difference，不能依赖机器默认 locale。
- pin order 在同 tenant+agent 原子分配，不使用无锁 `MAX+1`；unpin 幂等且清 `pin_order`。
- delete/batch delete 事务内处理四核心表，重复 ID 去重、空集合 no-op；软删/物理删以 V202 为准，对外不出现半可见。
- fork 禁止 running；锁 source，验证 boundary，批量复制消息并重映射 references，清瞬时 metadata，复制胶囊并重置 revision；任一失败全回滚。

### 4.2 messages

- cursor 权威是 `(created_at,id)`，不是 `sequence`。`before/after`、空页、页尾、相同时间、新插入、删除和反向读须有无重无漏定义；不返回 offset。
- `sequence` 是可空展示序，不作 key/unique/cursor/idempotency。append 通过 session allocator/等价原子方案分配；100+ 并发写不重序/丢写。
- insert-before 固定锁序，留间隙或 shift，但不改 `(created_at,id)`；missing target 的 append 行为只存在显式 compat policy。目标 message 的 legacy `sequence=NULL` 必须有单独 corner，不得让 SQL 与 NULL 比较后静默插入错误位置。
- replace 只替换 user/assistant，保留 tool/system 行本身；legacy 实现重插后的会话消息会排在当前剩余最大 sequence 之后，并不真正保留 tool/system 的原位置，这一事实必须由 golden 冻结并由 approved difference 明确目标取舍。重复 role+content 按旧顺序消费时间；timestamp 优先级为 explicit > old > now；metadata 默认 final/non-streaming；只有更新的 user message 单调推进 session `updated_at`。
- update 限定 `session_id+message_id+tenant_id`，流式更新不碰 session；delete 幂等且不能影响他会话/他租户同 ID。
- mapper 区分 SQL NULL、JSON null、空对象/数组；metadata 必须可安全序列化。超大内容限额以 contracts/V202 为准；V203 不自建 S3 外移。
- rail index 仅 user/assistant 非空内容，排除 `isThinking=true` assistant，offset 指全消息序列位置，预览与旧 helper golden 对齐且无 N+1。

### 4.3 capsules

- get 只读同 tenant session；SQL NULL 返回 null，JSONB 经 schema 验证，缺 `completedFacts` 映射 `[]`。
- create/update 使用 expected revision/CAS：新建从 1 开始，只有 current=expected 才更新；stale writer 返回 conflict。legacy 无条件 upsert 仅用于 parity harness。
- delete-many 去重、空集合 no-op、返回真实变更数；跨租户 ID 不影响。

### 4.4 agents

- 在 accepted V202 字段上扩展 P02 最小 agent facade，不导出 raw client，不破坏 `AgentDto.id=logical_id`。
- A/B 都可有 `main` 且内部 UUID 不同；session 关系用内部 FK+同 tenant 复合外键，对外只暴露 logical ID。
- 显式 logical ID 作为幂等键：同 key+同 canonical payload 重放返回原行；同 key+不同 payload 返回 conflict。旧 slug+时间戳递归仅供 compat 比较。
- `main`/`is_default` 不可删；删自定义 agent 与其 session/message/capsule 在一个事务内，并依 V202 软删/FK 决议执行。memory 副作用只交接。
- update 区分 omitted/null/empty，非法 skillIds/icon/source/preset 返回 validation；仅实际变化时更新 `updated_at`。pin order 原子。
- list 固定为 default 优先、再 created/id，一次读完必要列，不 per-agent 再查 session/skills。

## 5. 事务、并发、幂等、retry 与错误映射

### 5.1 事务和锁序

- 每个 public 调用进入 P02 `withTenantTransaction`，参数化 `set_config(...,true)` 同时设置 tenant/user；禁止 session `SET`、unsafe raw 和事务外延迟查询。
- 固定锁序：agent（若有）→ sessions（UUID 升序）→ capsules（session UUID 升序）→ messages（UUID 升序）。fork/delete/replace/insert-before 均同序。
- 不向上层暴露 raw transaction；内部组合复用同一 scoped tx，不在 tx 中再开新 tx/client。
- 多步骤操作都设 fault point：首写后、子行中途、最终父更新前、commit 前；失败后 row count/hash/FK/sequence/revision 与事务前相同。

### 5.2 retry 和幂等

- 仅对 SQLSTATE `40001`、`40P01` 和 Prisma `P2034` 重试整个 transaction closure。
- 最多 3 次，指数 backoff + bounded jitter，时间/随机可注入；报告 attempts/SQLSTATE/operation，不记录内容或 secret。
- 仅重试 idempotent closure。create/add/fork 在首次 tx 前生成稳定 ID，重放使用同 ID；唯一冲突区分同请求重放和真冲突。
- `P2002/23505`、`P2025`、FK/check/RLS/validation 不盲目 retry；连接失败不在 repository 内无界循环。

### 5.3 错误映射

V203 集中定义 repository error code；V204 将来映射 HTTP，V203 不实现 controller。

| 底层条件 | repository code | 预期 contract code/属性 |
|---|---|---|
| 输入/cursor/JSON/值域错 | `INVALID_ARGUMENT` | `VALIDATION_FAILED`，不可重试 |
| 不存在或跨 tenant 不可见 | `RESOURCE_NOT_FOUND` | `NOT_FOUND`，不泄露存在性 |
| `P2002/23505` 且非重放 | `CONFLICT` | 由 V204 按 route 决议，禁止伪装 500 |
| capsule/stale 并发写 | `STALE_WRITE` | retryable conflict，不覆盖新值 |
| `40001/40P01/P2034` 耗尽 | `RETRY_EXHAUSTED` | `INTERNAL_ERROR` retryable，不含 SQL |
| context 缺失/RLS 越权 | `TENANT_CONTEXT_INVALID` / `ISOLATION_VIOLATION` | fail closed；不自动改用 admin |
| 连接/未知 Prisma 错误 | `DATABASE_UNAVAILABLE` / `DATABASE_ERROR` | `INTERNAL_ERROR` |

## 6. tenant/RLS、非特权连接与 N+1/性能

### 6.1 隔离硬断言

- 四表在 RLS manifest、migration `ENABLE+FORCE`、catalog、Prisma extension 四集合中双向一致，`missing=extra=0`。
- app role 必须 `rolsuper=false`、`rolbypassrls=false`、非 owner，且不经 membership 继承越权。admin URL 只做 migrate/catalog。
- 关闭 Prisma extension 的红队 lane 中，A 对 B 的 select/insert/update/delete/fork/pin/capsule/agent 仍被 RLS/constraint 拒绝；伪 tenant、缺 context、session `SET`、commit/rollback/throw/cancel 后池复用均 fail closed。
- 子行复合 FK/CHECK 阻止跨 tenant 挂接，即使用管理连接绕 RLS 也失败。

### 6.2 查询轮次与性能门

| 场景 | SQL round-trip 上限 | PG17 暖机性能门 |
|---|---:|---|
| list/search 50 sessions | 1 | p95 ≤100ms，命中 tenant/order 索引 |
| session detail + last 50 + total | ≤2 | p95 ≤150ms，查询数不随消息数增长 |
| message page 100 | 1 | p95 ≤100ms，命中 cursor 索引 |
| rail index 1,000 messages | 1 | p95 ≤250ms |
| list 100 agents | 1 | p95 ≤100ms |
| fork 1,000 messages + capsule | ≤6 | p95 ≤1s，批量写 |

fixture 至少含两租户、1,000 sessions、100,000 messages、100 agents、相同时间/NULL sequence/大 JSON。报告记录 CPU/内存/PG config/冷暖次数。绝对阈值与同环境 baseline 都须满足；环境不可比则 `BLOCKED`，不删性能门。

## 7. shadow、受控 dual write、reconcile、cutover 与 SQLite 退场

### 7.1 原则

- SaaS 是全新 PG 后端，**禁止生产服务引入 SQLite、禁止生产写双写用户 SQLite、禁止 PG 反灌 SQLite**。
- 桌面 `lobsterai.sqlite` 继续由 Electron 权威使用，V203 不删表/禁用。parity 对真实/脱敏 snapshot 只能 read-only/immutable 打开，运行前后 SHA-256 必须一致。
- dual write 只在 **CI/本地/受控 staging parity lane**：同一份脱敏 operation journal 执行到 PG 与从 snapshot 新建的临时可写 SQLite oracle copy。源 snapshot 永远只读；临时 copy 必须 runId 隔离并 cleanup。
- V5 可选桌面导入不属于 V203。V203 只提供四域 canonicalizer/reconcile 原语，不声称导入工具完成。

### 7.2 分阶段准入

| 阶段 | 流量/权威 | 进入门 | 退出/回滚 |
|---|---|---|---|
| S0 行为冻结 | read-only legacy oracle | inventory/golden 有分母和 hash，V202 硬门 PASS | 重建 fixture，不动业务数据 |
| S1 shadow read | PG 测试写；两边读 canonical diff | 核心 CRUD Green；差异有精确 policy | 关 shadow flag |
| S2 controlled dual write | journal 写 PG + ephemeral SQLite copy | idempotency/rollback/retry Green；0 未解释 diff | 销毁 runId PG fixture/temp copy |
| S3 reconcile | 停止新 journal，重算 | row count/hash/FK/null/default/order/page/revision/dual-key 均 0 unexplained diff | 任一差异退 S1/S2 |
| S4 PG cutover-ready | server factory 仅产 PG facade | 28 AC、Reviewer、Tester 通过；无 SQLite import graph | V203 只交 readiness；V204 接线另验收 |
| S5 rollback rehearsal | 回滚 library/repository flag/release，保留 expand schema | 中止重试/并发期不丢已承诺写 | 不回 SQLite、不 down migration；必要时停 V204 路由 |

reconcile 每操作记录 `operationId/idempotencyKey/tenantFixture/inputHash/legacyResultHash/pgResultHash/approvedDifferenceIds/rowCountDelta/referenceDelta/orderDelta/errorClass/latency`，不记录原始 prompt/tool/system/capsule 内容。

## 11. 回滚、风险升档与四角色交接

### 11.1 回滚

- 代码回滚顺序：CI/validate → performance/reconcile harness → repositories → mapper/cursor/error/retry。V202 expand schema/migration 保留，不 down/drop，不改 history。
- 测试数据按 runId/tenant/schema 精确清理；cleanup 失败则本 run `FAILED`，报告残留 ID。
- cutover 后只回滚 server release/repository flag/路由，暂停新写并保留 PG 数据；不启 SQLite 生产备路，不反灌。
- 任一 tenant leak、丢/重写、半事务、游标漏页、未解释 diff、retry 非幂等或性能越门，停止 V204 依赖任务。

### 11.2 R1 高风险升档

P50 20 pd 不是承诺。若 inventory 未归类语义 >5%、golden 未解释差异 >0、核心操作无法按固定锁序原子实现、查询轮次/性能不达标，立即切 30～40 pd 高风险档并阻断 V204；不删表、不减四核心域。需改大文件架构时先按根 AGENTS 提精确抽取方案。

### 11.3 四角色

- Planner 只冻结四文件，不写实现/总表。
- Developer 解锁后只追加 `开发记录.md`，逐 AC 写 Red→Green→Refactor、mutation、命令/退出码/环境/证据；不写 Reviewer/Tester 结论。
- Reviewer 使用独立 context/runId/PG container，审 diff、Red、28 AC、范围边界，并独立做至少 12 个 mutation。任一 P0/P1、缺 Red、skip、admin 冒充、未解释 diff、越界或 stale 证据必须 `CHANGES_REQUESTED`。
- Reviewer `APPROVED_FOR_TEST` 后，独立 Tester 用新 runId/container/fixture 新增未覆盖 corner/security/concurrency/performance 测试。`FAILED` 退 Developer，修复后必须重新 Reviewer。
- Tester `PASSED` 后，协调者核对 V202 硬门、28 AC、同 SHA 证据、SQLite/PG 边界和 V204 未触碰，才可更新总表。
