## 3. 冻结模型与物理约束

Prisma 字段用 camelCase，物理表/列用 snake_case `@map/@@map`。状态/模式用共享 `as const` 来源；数据库用 `CHECK`，不使用难以扩展的 PostgreSQL enum。

Prisma 不能完整表达 RLS、partial unique、JSON/check、不可变 trigger 与 `ON DELETE SET NULL(column-list)`；这些约束必须进入**新** migration raw SQL，并由 checker 把 Prisma 模型、migration 声明和真实 catalog 作为一个整体核对。禁止用 `prisma db push` 取代 migration，也禁止因 Prisma schema 未显示某 raw 约束就把它从 drift 分母删掉。

### 3.1 `agents`（扩展 P02 现表，不重建）

- 保留：`id UUID PK`、`tenant_id UUID NOT NULL`、`logical_id TEXT NOT NULL`、`name`、`model`、`skill_ids JSONB`、物理 `working_directory`、时间列及 P02 tenant FK/RLS。
- 新增兼容字段：`owner_user_id UUID NULL`、`description/system_prompt/identity/icon TEXT NOT NULL DEFAULT ''`、`enabled/pinned/is_default BOOLEAN`、`pin_order INTEGER NULL`、`source TEXT`、`preset_id TEXT NULL`、`workspace_id UUID NULL`、`deleted_at TIMESTAMPTZ(6) NULL`。
- `working_directory` 物理列与 P02 Prisma `workingDirectory` 字段不 rename/drop；另加 `workspace_id`。旧绝对路径仅作兼容输入，生产不得当沙箱路径直接使用。
- 约束：`UNIQUE(tenant_id,logical_id)` 保持；另建 `UNIQUE(tenant_id,id,logical_id)` 供 session 三列 FK；`logical_id/name` trim 后非空且拒 NUL；`source IN ('custom','preset')`；`skill_ids` 必须是 JSON array；`pin_order` 仅 pinned 时可非空；每租户至多一个未软删 default agent 用 partial unique index；`main` 仍是 logical ID，不是共享 UUID。
- 索引：`(tenant_id,enabled,pinned,pin_order)`、partial unique `(tenant_id) WHERE is_default=true AND deleted_at IS NULL`、`(tenant_id,updated_at DESC,id)`；不重复创建 P02 已有等价索引。

### 3.2 `cowork_sessions`

- 字段：`id UUID PK`、`tenant_id UUID NOT NULL`、`owner_user_id UUID NOT NULL`、`title`、历史物理列 `claude_session_id`（Prisma `runtimeSessionId`）、`status`、`pinned/pin_order`、`workspace_id/rel_root/legacy_cwd`、`system_prompt/model_override/execution_mode/active_skill_ids/goal_json`、`next_message_sequence INTEGER NOT NULL DEFAULT 100`、fork 全字段、`agent_pk UUID NULL`、兼容 `agent_id TEXT NOT NULL DEFAULT 'main'`、可空 `idempotency_key/idempotency_request_hash`、`deleted_at/created_at/updated_at TIMESTAMPTZ(6)`。
- `cwd` 不进入目标新列；历史 SQLite `cwd` 的导入映射固定为 `legacy_cwd`，目标工作区是 `workspace_id + rel_root`。`fork_workspace_path/fork_git_branch/fork_git_base_ref`、`goal_json`、`claude_session_id` 物理名原样保留。
- checks：`status IN ('idle','running','completed','error')`；`execution_mode IS NULL OR IN ('auto','local','sandbox')`；`fork_mode IN ('none','conversation','worktree')`；JSON 数组/对象类型正确；title/rel_root 无 NUL；`next_message_sequence >= 0`；未 pinned 时 `pin_order IS NULL`；无 parent 时 fork message 也必须为空；idempotency key/hash 必须同时 NULL 或同时非空，hash 为 lowercase SHA-256 hex。
- 关联：`tenant_id→tenants ON DELETE CASCADE`；`(tenant_id,agent_pk,agent_id)→agents(tenant_id,id,logical_id) ON DELETE SET NULL(agent_pk)`，从数据库保证内部 UUID 与兼容 logical ID 指向同一 agent；`(tenant_id,parent_session_id)→cowork_sessions(tenant_id,id) ON DELETE SET NULL(parent_session_id)`；禁止自 parent。`agent_pk` 为 NULL 时保留历史 `agent_id`，由 V203 在新写事务中解析。
- 唯一/索引：`UNIQUE(tenant_id,id)`；partial `UNIQUE(tenant_id,idempotency_key) WHERE idempotency_key IS NOT NULL`；列表 cursor `(tenant_id,deleted_at,updated_at DESC,id)`；owner、parent、pinned、agent 访问索引均以 `tenant_id` 开头。

### 3.3 `cowork_messages`

- 字段：`id UUID PK`、`tenant_id UUID NOT NULL`、`session_id UUID NOT NULL`、历史物理 `type TEXT`（Prisma `role`）、`content TEXT NOT NULL DEFAULT ''`、`metadata JSONB NULL`、`sequence INTEGER NULL`、`idempotency_key/idempotency_request_hash TEXT NULL`、`created_at TIMESTAMPTZ(6) NOT NULL DEFAULT now()`。
- canonical `type` 为 `user|assistant|tool|system`。未来桌面导入把 `tool_use/tool_result` 映射为 `tool` 并在 metadata 保留 `legacyType`；未知值在 dry-run 阻断，不静默归类。V202 不执行该桌面导入。
- `id`、`tenant_id`、`session_id`、`created_at` 是 cursor/归属不可变列；migration 用 DB trigger 或等价列权限阻止应用角色更新。`sequence` 可空、可在 insert-before 时调整，**不 unique、不作 cursor/幂等键**。
- 约束：role check；`sequence IS NULL OR sequence >= 0`；metadata 为 object 或 NULL；idempotency key trim 后非空且仅允许 user message，key/hash 同空同非空，hash 为 lowercase SHA-256 hex。`UNIQUE(tenant_id,session_id,id)` 供关联；partial `UNIQUE(tenant_id,session_id,idempotency_key) WHERE idempotency_key IS NOT NULL` 防同会话用户写重放。
- 关联：`(tenant_id,session_id)→cowork_sessions(tenant_id,id) ON DELETE CASCADE`，数据库禁止跨租户挂消息。若保留 `forked_from_message_id`，再以 `(tenant_id,parent_session_id,forked_from_message_id)` 关联消息三元组并在删除源消息时只清 fork message 引用。
- 索引：权威 keyset `(tenant_id,session_id,created_at,id)`；展示 `(tenant_id,session_id,sequence,id)`；是否加 metadata GIN 必须以真实查询/容量证据决定，V202 不因旧文档示例默认增加高写放大索引。

### 3.4 `cowork_session_capsules`

- 一会话一胶囊：`session_id UUID PK`、`tenant_id UUID NOT NULL`、`version/revision INTEGER`、`capsule_json JSONB`、`updated_at TIMESTAMPTZ(6)`、`last_source TEXT`、`last_compacted_at TIMESTAMPTZ(6) NULL`。
- checks：version/revision 非负；capsule 为 JSON object；`last_source` 使用现有 continuity source 常量集合且非空；`last_compacted_at IS NULL OR last_compacted_at <= updated_at`。
- `(tenant_id,session_id)→cowork_sessions(tenant_id,id) ON DELETE CASCADE`；索引 `(tenant_id,updated_at DESC,session_id)`。session 软删时胶囊保留，最终硬 purge 才 cascade。

### 3.5 软删、级联与 RLS

- `agents`、`cowork_sessions` 是资源型表，使用 `deleted_at` 软删。`cowork_messages/capsules` 不单独软删：reconciliation 可硬删消息，session 的最终合规 purge 才由 FK 级联清子表。
- soft delete 不触发 FK cascade；默认 tenant-scoped 读路径还必须过滤 `deleted_at IS NULL`，该应用层行为由 V203 实现，V202 只测试 schema/extension 注册表已标记资源软删语义。
- tenant 删除最终 cascade agents/sessions，再 cascade messages/capsules；生产执行前仍遵守合规顺序，不以数据库 cascade 代替 token/Redis/object/PVC 清理。
- RLS registry 必须把 `Agent/CoworkSession/CoworkMessage/CoworkSessionCapsule` 与物理表双向列全；四表均有 ENABLE+FORCE、相同 tenant `USING/WITH CHECK`。无上下文、非法 UUID、跨租户父子、伪造 body tenant 均 fail closed。

## 4. 顺序、cursor、幂等与事务协议

1. **cursor**：API cursor 保持 P01 opaque string；内部载荷版本化为 `{v:1,createdAt:<UTC ISO>,id:<uuid>}`。查询按 `(created_at,id)` 严格 ASC/ DESC keyset，比较使用 `(created_at,id) >/< ($ts,$id)`，取 `limit+1` 决定 `hasMore`。同时间、删除边界、前后翻页不得重复/漏项。cursor 不含 `sequence`，不得暴露 offset。
2. **不可变**：消息 `created_at/id` 一旦 insert 不可 update；测试直接以 app role mutation，必须失败。时钟以 DB `timestamptz(6)` 为准，禁止把 epoch 毫秒当秒；所有报告同时记录 UTC 与 CST。
3. **展示 sequence**：默认 append 在同一显式事务中原子执行 `UPDATE cowork_sessions SET next_message_sequence=next_message_sequence+1 ... RETURNING next_message_sequence-1` 后 insert。该行锁协议在真实 PG 并发测试中证明唯一、单调且回滚不产生重复；gap 允许，sequence shift 允许。具体 repository/retry 实现在 V203。
4. **幂等**：创建 session 使用 `(tenant_id,idempotency_key)`，用户消息使用 `(tenant_id,session_id,idempotency_key)` partial unique；两者保存 canonical request SHA-256。相同 key+相同 hash 由 V203 回放原资源；相同 key+不同 hash 应 409。V202 只提供约束与冲突/并发测试，不伪造 handler。
5. **事务与 retry**：session+首 user message、fork+消息/capsule copy、capsule CAS revision 更新必须全部成功或全部回滚。并发冲突分类冻结：unique/idempotency 命中为确定性读取/409；serialization/deadlock 最多按 V203 的有界 jitter retry；FK/check/权限错误不 retry。V202 用 SQL/Prisma 事务 fixture证明 rollback、并发和错误码，不实现业务循环。
6. **capsule CAS**：update 使用旧 `revision` 条件并写 `revision+1`，并发仅一方成功；禁止无条件 last-write-wins。V203 才封装 API。

## 6. Forward migration、backfill、锁与回滚

### 6.1 migration 顺序

1. preflight：锁定 P02 migration checksum、schema/generated hash、PG17 server version、extensions、角色与当前 catalog；检测 drift/失败 migration/脏 `_prisma_migrations` 立即阻断。
2. expand：为 `agents` 加 nullable/有安全常量默认的新列；新建 sessions/messages/capsules、复合键、checks/FK/RLS。禁止修改 P02 已发布 migration。
3. backfill：按 UUID keyset 小批次更新既有 agents（logical main/default、JSON/boolean/时间兼容）；每批 commit、可续跑、记录行数/hash，不用 OFFSET，不持有全表长事务。
4. contract：先 `CHECK ... NOT VALID`/FK `NOT VALID` 再 `VALIDATE CONSTRAINT`；backfill 完成才 `SET NOT NULL`。大表索引使用受控的 concurrent/non-transactional forward 步骤并确保 migration history 可审计，不用未记录的生产 ad-hoc SQL。
5. generate/validate：Prisma generate 两次确定，schema↔migration↔generated↔RLS registry↔真实 catalog 双向一致；空库 deploy、P02→V202 deploy、重复 deploy三条路径全绿。

### 6.2 大数据与锁门

- Testcontainers 至少构造大批 P02 agents 与大批 session/message fixture，记录行数、数据库大小、各 migration step、最长锁等待、statement duration、阻塞 PID、索引有效性和前后 checksum。
- `lock_timeout`/`statement_timeout` 必须显式且失败即 `FAILED/BLOCKED`；并发读写探针在 expand/backfill/validate 期间持续运行。超冻结锁预算时停止发布，拆 migration 或采用 expand/contract，不临时调大到无限。
- 任一失败不得留下 invalid index、半 backfill、RLS 未 FORCE 或 migration marked applied 但 catalog 不完整；测试需注入中断后验证可安全续跑/forward-fix。

### 6.3 回滚

- 生产/共享库没有 destructive down migration。应用回滚到 P02/V201 兼容版本时忽略新增表列，保留数据；关闭 V202 写路径，不 drop table/column/policy。
- migration 未开始可回滚发布；测试库可删库重建。已部分执行或已发布时先停写并恢复快照/PITR，或追加 forward-fix migration；禁止手改 `_prisma_migrations`、重写旧 SQL、`migrate resolve` 假绿。
- 回滚演练必须验证 P02 旧 agent 读写仍可、V202 数据保留、RLS 未被放开。V203 尚未开始，因此不得以 repository rollback 证明 V202 rollback。
