## 4. PostgreSQL、扩展、Prisma 与角色决策

- PostgreSQL：CI/Testcontainers 兼容基线为 17 major；`postgres-image.json` 固定 exact patch+digest，不使用 `latest`/浮动 major。报告需记录 image ref/digest、架构和 `SHOW server_version`。
- `citext`：首 migration 显式 `CREATE EXTENSION IF NOT EXISTS citext`，preflight 查询 `pg_available_extensions` 与已安装版本。权限不足或扩展不可用时失败，不降级成普通 `text` 假绿。
- UUID：preflight 实际执行 `SELECT gen_random_uuid()`。基线 PostgreSQL 可用内建函数；若批准目标库没有，则同一变更选择并记录 `pgcrypto` 或应用侧 UUID，连同 schema/migration/tests 一起更新，不能静默依赖隐式扩展。
- Prisma：CLI、client、Testcontainers 依赖必须锁入对应 workspace/根 lockfile；版本与 Node 24/PostgreSQL 17 兼容。生产运行时不得动态 `npm install`，所有工具仅在构建/CI 或迁移镜像中使用。
- 迁移连接使用 admin/migration role；应用/RLS 测试使用独立 `NOLOGIN/LOGIN` 测试角色，断言 `rolsuper=false`、`rolbypassrls=false` 且不是表 owner。测试不能用 migration role 查询后宣称 RLS 已验证。
- 请求连接语义固定为 PgBouncer transaction mode：每个逻辑请求开启显式事务，用参数化 ``SELECT set_config('app.tenant_id', ${tenantId}, true)`` 与 `app.user_id`；禁止 `SET`、`$executeRawUnsafe` 和字符串拼接。事务成功、抛错、回滚后均不得在复用连接中残留上下文。

## 8. 回滚、恢复与角色交接

### 8.1 回滚/恢复

- 数据库只做 expand-first 向前 migration。测试库可删库重建；真实/共享数据库不得手工执行 down SQL、改 `_prisma_migrations` 或覆盖已提交 migration。
- migration 应用前保留快照/空库基线；失败时停止应用发布，恢复快照或创建新的 forward-fix migration。应用代码回滚不得删除新表/列/policy。
- 整体代码反向回滚：CI/stage gate → Testcontainers/tests/report → facade/context/extension → RLS/migration/schema → Prisma deps/lockfile。schema、migration、generated consumer 必须同批一致，不能只退其中一个。
- 测试结束清理 container/volume/network/test role/runId seed；清理失败必须在报告列残留资源，不得仍标 PASS。

### 8.2 Planner → Developer → Reviewer → Tester

### 9.2 manifest v2 与 selector 契约

锁定以下 TypeScript 语义；JSON 使用等价字段：

```ts
type ApprovedPostgresPlatform = 'linux/amd64' | 'linux/arm64';

interface ApprovedPostgresImage {
  platform: ApprovedPostgresPlatform;
  digest: string; // 必须匹配 /^sha256:[a-f0-9]{64}$/
  immutableReference: string;
}

interface PostgresImageManifestV2 {
  schemaVersion: 2;
  image: 'postgres:17.10-bookworm';
  serverMajor: 17;
  serverVersion: '17.10';
  distribution: 'Debian bookworm';
  platforms: ApprovedPostgresImage[];
}
```

`platforms` 的序列化顺序固定为 `linux/amd64`、`linux/arm64`，且必须恰好两项。根对象和每个 platform 对象都使用字段白名单；重复 JSON key、重复 platform、缺少 platform、额外字段、大写 digest、非 Linux 平台均拒绝。`immutableReference` 必须字节级等于 ``${image}@${digest}``，不接受等价别名。

`scripts/db/postgres-image-policy.mjs` 固定导出：

- `normalizeDockerPlatform(value)`：接受 Docker 原生值 `linux/amd64`、`linux/arm64`、`linux/x86_64`、`linux/aarch64` 以及 Node arch `x64`、`amd64`、`x86_64`、`arm64`、`aarch64`，只返回 `linux/amd64 | linux/arm64 | null`。不 trim、不忽略大小写；空串、带空白、`darwin/*`、`windows/*`、`linux/ppc64le`等全部返回 `null`。
- `loadPostgresImageManifest(root)`：严格拒绝 unknown root/nested fields、非 v2、重复/缺失平台、非 exact tag、坏 digest 和 reference 不等式。
- `selectApprovedPostgresImage(manifest, dockerPlatform, nodeArch)`：Docker 与 Node 归一平台必须相同，并精确返回唯一 entry；未知平台抛出带 `blocked=true` 的错误。

错误分类不得交给调用者猜测：manifest/schema/digest/reference 不合法和 inspect 后完整性不符为 `FAILED/1`；Docker provider 不可用、registry 无法拉取已批准 immutable ref、未知平台或 Node/Docker 架构漂移为 `BLOCKED/2`。`preflight.mjs` 和 `run-integration.mjs` 必须保持该映射一致。

arm64 保留当前已验证 digest `sha256:17b6c778de50f4bb9a878c36e736110fbcd9b7020377d6fdfdf20f7c0347e40a`。amd64 digest 只能由当前 Docker/OCI registry manifest 实际解析并经 pull+inspect 复核后写入，禁止从网页、日志或浮动 tag 猜值。
