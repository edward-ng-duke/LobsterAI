### [P2] FIFO 回归无条件依赖 POSIX `mkfifo`，Windows 官方 `npm test` 必然失败

- 位置：`tests/db/evidence-bootstrap.test.ts:574-580`。
- 测试直接执行 `spawnSync('mkfifo', [protectedPath])`，随即断言 `status === 0`；没有 platform guard、capability detection 或 `skipIf`。Windows 不提供 `mkfifo` 可执行文件，Node `spawnSync` 在该路径返回 `status=null` 与 `error.code=ENOENT`，因而正式 `npm test` 会在进入被测 launcher 前直接失败。
- 这不是 Windows 上无法触发的产品行为，而是 test fixture 的平台依赖；项目声明支持 Windows，且该文件被默认 Vitest `npm test` 收集，不能用当前 Linux CI 的 102/102 Green 覆盖。
- 要求：保留 POSIX FIFO 角度，但必须通过明确的 OS/capability 条件使非 POSIX 环境 skip，并为“不支持 FIFO fixture”的路径保留可审计断言；不得删除 Linux/macOS FIFO nonblocking 回归，不得放宽 runtime 的 regular-file 检查。

### 运行时修复与其他边界复核

- launcher 先将请求 root `realpath` 为真实目录，再对 bootstrap 执行 `lstat`、regular/non-symlink 与 `realpath === lexical path` 检查，然后才 `readFileSync` 并 `--import`；同字节 bootstrap symlink、FIFO 与外部 parent symlink 不能在身份检查前被读取。
- bootstrap 在 manifest read、Git history/hash 和 protected-file read 之前，对 required、存在的 optional 与自身执行同等 `lstat/realpath` 检查。same-byte validator/parser/manifest symlink、broken optional symlink、FIFO、root/parent escape 均 fail closed；真正 absent optional 不入 protected set，整个 worktree root 的合法目录 symlink 在 root 规范化后保持 PASS。
