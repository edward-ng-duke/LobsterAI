### 1.2 Day-0 与 approved internal endpoints

真实运行前由 PLAT/SEC/BE 共同冻结、preflight 实测并版本化：

- K8s context、cluster UID/minor、专用 namespace、CNI 名称/版本/NetworkPolicy 支持矩阵、Pod Security/admission、节点池/RuntimeClass、双栈模式、清理和只读审计 RBAC；
- 节点/子网的默认路由、NAT、互联网网关、防火墙/安全组、企业代理和 service-mesh/egress-gateway 拓扑；生产同构环境必须没有公网 NAT/代理兜底；
- 集群 DNS Service/Pod selector、IP、UDP/TCP 53、递归/转发策略、公共域名响应策略、查询日志与 correlation 能力；
- 云/平台 metadata 地址全集、K8s API Service IP/DNS/port、node-local DNS/代理/agent 地址，以及实际 IPv4/IPv6 CIDR；不得只测 `169.254.169.254`；
- 网络审计事实源：至少 CNI flow/drop log 与节点/边界防火墙日志；若启用内网 egress gateway/sidecar，再加 gateway access/deny log。记录查询入口、保留期、时钟同步、字段、owner 与脱敏规则；
- 在首次运行前冻结 timeout、重试次数、允许的 DNS/连接时延、审计日志到达窗口和证据保留期，不得看结果后改阈值。

批准内网端点使用单一、可机读且经 SEC/服务 owner 签署的 registry。每项至少含：

`endpointId/purpose/requiredForProfile/owner/environment/namespace/serviceName/podSelector/ports/protocol/FQDN/IPv4/IPv6/TLS SNI/CA/expected redirects/auth SecretRef/audit correlation/expiry/change ticket`。

约束如下：

- 最小 allow 按 workload profile 分账，禁止把某生产组件的批准项继承给 sandbox。V103 sandbox profile 只允许 kube-dns、已批准内网模型和确属 turn 所需的 Cowork/API 精确 Service/port；K8s API、Postgres、Redis 和任意横向 Pod 对 sandbox 始终是 deny，即使它们对 API/worker 是批准依赖。OIDC/S3 等其它生产端点另属对应 workload profile，不能用于放宽 V103 sandbox PASS 路径。
- Service selector 优先于 CIDR；只能由外部内网 VIP 表达的端点必须列精确 IP/port、变更机制和到期时间。禁止整 namespace、整 RFC1918、`0.0.0.0/0`、`::/0`、任意端口或仅凭域名字符串放行。
- registry 中 FQDN、Service、实际 Endpoints/EndpointSlice、解析 A/AAAA、TLS SNI 和服务端 receipt 必须同一 run 对账。漂移、过期、空 allowlist 或 owner 未签署均为 `BLOCKED`，不是 PASS。
- 当前 BLK-01/02/05/06 未解除时不能产真实 V103 PASS。缺批准内网模型/测试 namespace/CNI 权限时，静态开发可继续，但动态门为 `BLOCKED`，不得临时直连公网供应商。

### 1.3 `PASS / FAILED / BLOCKED`

- `PASS`：前置和 Day-0 完整，在生产同构真实集群执行；所有批准 allow 有客户端、服务端和网络审计三方关联证据，所有 deny/旁路在工作负载层与节点/网关层均被阻断，artifact 可重算。只有 PASS 可返回 0。
- `FAILED`：环境完整，但任一公网/metadata/K8s API/未批准内网/横向目标可达，批准目标不可达，策略/准入/审计/证据/cleanup 断言失败，或实际配置比 registry 更宽。必须非零并退 Developer，不能降级为 BLOCKED。
- `BLOCKED`：缺前置验收、Day-0 决策、批准端点、真实集群/CNI/双栈节点、审计权限/日志、RuntimeClass/digest、测试 fixture 或工具。使用专用非零码并穷举 `blocked: missing <decision>`；不得 apply 未决策略，也不得交 Tester。

`SKIPPED`、`NOT_APPLICABLE`、空目标集、只有 Helm render、kind/runc、本机防火墙、连接 timeout 无 drop 侧证、环境不可用、审计查不到日志或 exit 0 无 artifact 均不得汇总 PASS。

## 3. 分层强制边界与最小 allow

### 3.1 四层 fail-closed

1. **Pod/CNI**：namespace 与 sandbox label 均有 `policyTypes: [Ingress, Egress]` default-deny；allow 只按受控 namespace label + service pod selector + protocol/port，DNS 仅到实际 DNS Pod/IP 的 TCP/UDP 53。
2. **Pod 准入**：sandbox 禁 `hostNetwork/hostPID/hostIPC/hostPort`、privileged、额外 capability、未批准 sidecar/init/ephemeral container、hostPath 和 service-account token；RuntimeClass/image digest/securityContext 与 V101 tuple 一致。
3. **节点/子网**：无公网 route/NAT/通用代理，边界防火墙默认拒绝 IPv4/IPv6 公网、link-local metadata 和非批准内网；NetworkPolicy 漏洞不能变成公网可达。
4. **内网网关/审计**：如存在 egress gateway/mesh sidecar，只接受 registry 中精确目标/SNI/port，禁止 CONNECT/SOCKS/通配域名和公网转发；所有 allow/deny 可按 runId/correlation 查询。

任一层未部署、实际状态不可读取或 CNI 对某地址族/工作负载类型不执行策略时，不得用其它层“看起来拒绝”补 PASS；应 BLOCKED 或 FAILED，并在 V107 给出 no-go。

### 3.2 成对探针与判定

- 每项 allow 必须证明：DNS（若使用）解析符合 registry、TLS/协议握手正确、应用层 nonce response 成功、服务端收到同 correlation、CNI/网关日志显示命中精确 allow；只有 TCP connect 或 HTTP 任意响应不够。
- 每类 deny 至少同时用 FQDN（适用时）与 IP literal 探针；记录 DNS 结果、连接阶段、错误类别和独立 drop/deny 日志。单纯超时、NXDOMAIN 或目标自身宕机不能证明策略生效。
- 为避免把不可用目标误记为安全拒绝，deny corpus 使用受控、在边界另一侧可证明健康的 canary listener，或由防火墙/CNI 明确记录本 run 的 drop。公网无任何 route 时仍须用节点/边界路由和 drop 证据，而不是依赖某公共网站在线。
- 同一 run 先执行负控：临时、仅本 run 的批准内网 canary 在无 allow 时应被拒；随后应用精确 allow 后应成功；再删除/改端口/SNI 应重新失败。该 Red→Green→Red mutation 证明命中的是目标策略而非目标不可用。

## 4. 旁路与攻击面矩阵

### 4.1 DNS、地址族与名称技巧

- IPv4 与 IPv6 分别覆盖：公共单播、RFC1918/ULA 未批准地址、link-local、loopback、node/pod/service CIDR、IPv4-mapped IPv6；未启用 IPv6 但 kernel/CNI 可发 IPv6 时必须显式拒绝，不能标 N/A。
- DNS 覆盖 UDP/TCP 53、A/AAAA、CNAME 多跳、search domain、绝对域名尾点、大小写/IDN/punycode、低 TTL、同名多 A/AAAA、解析结果漂移和 DNS rebinding。每次 connect 前后记录答案与 TTL；批准 FQDN 若解析到未批准 IP，fail closed。
- 阻断任意外部 DNS、DoT 853、已知/自定义 DoH HTTPS、把 DoH 藏在批准 SNI/Host 下、IP literal 绕 DNS。集群 DNS 自身必须拒绝/不转发公共域，或由上层边界阻断其外连并有日志。
- 公网 deny 不只测 HTTP/TCP：覆盖 UDP/QUIC 443、任意 UDP、高风险 SCTP（若 CNI/kernel 支持）、ICMP/raw socket（预期因 drop capability/边界拒绝）和非常用端口；不支持某协议须由 Day-0 能力矩阵给出 BLOCKED/明确不适用依据，不能默认为已拒绝。

### 4.2 HTTP/TLS/代理旁路

- 3xx 覆盖单跳/多跳、相对/绝对 Location、内网→公网、内网→metadata、协议降级和重定向循环；client 不得因初始目标批准而自动访问未批准下一跳。
- TLS 覆盖 SNI 与 Host 不一致、无 SNI、IP+伪 Host、通配证书、错误 CA、批准 IP 上未批准 SNI、批准 SNI 解析到未批准 IP；身份任一维度不匹配均失败。
- 清空或拒绝注入 `HTTP_PROXY/HTTPS_PROXY/ALL_PROXY/NO_PROXY`；显式测试 HTTP CONNECT、SOCKS4/5、企业代理、localhost/node-local proxy、透明 sidecar/mesh 网关和端口复用。代理本身即使在内网，也不得成为任意目标隧道。

### 4.3 平台与容器旁路

- metadata 至少覆盖环境实际提供者和通用地址：`169.254.169.254`、`100.100.100.200`、`169.254.170.2`、相应 DNS 名/IPv6 link-local/provider-specific 地址；IMDS token/header 变体与 redirect 到 metadata 均拒绝。
- K8s API 覆盖 `kubernetes.default.svc`、Service IP、Endpoint IP、node control-plane 地址；Pod 必须 `automountServiceAccountToken: false`，无 token 并不替代网络拒绝证据。
- 横向覆盖同 namespace/跨 namespace Pod IP、Service/Endpoint、headless Service、node IP/NodePort、DB/Redis/S3 未批准 port、node-local daemon/Unix-to-TCP bridge；批准某服务不代表其邻接端口或后端任意 Pod 可达。
- `hostNetwork/hostPID/hostIPC/hostPort` 由静态 schema、render checker、admission dry-run 和真实拒绝事件共同证明；若 privileged 用户能自行创建此 Pod，RBAC 也必须收敛。
- sidecar、init container、lifecycle hook、debug/ephemeral container 与 mutating webhook 注入都共享 Pod 网络命名空间。所有镜像/命令必须入 allowlist；未批准注入在 admission 阶段拒绝。独立测试 `pods/ephemeralcontainers` RBAC，不能只看主 container。
- hostNetwork/ephemeral 探针被准入拒绝是该旁路用例的期望 PASS；若集群不允许执行准入 dry-run或读取拒绝事件，则用例 BLOCKED，不把“未尝试”记 PASS。

## 6. Red → Green 最小切片与 commit 边界

- `V103-AC02`：Day-0 context/cluster/CNI/namespace/双栈/节点边界/DNS/审计/预算完整；缺失时 BLOCKED、apply/probe=0。
- `V103-AC03`：approved endpoint registry 逐项签署、未过期、可机读，Service/Endpoint/A/AAAA/IP/port/SNI/CA/redirect/owner 对账；无宽 allow。
- `V103-AC04`：范围只落 PoC egress、Helm 策略、测试与证据，无公网出口、第二 runtime 或 V102/V104/V2/V4 越界。
- `V103-AC05`：applied default-deny 确实选中 sandbox，Ingress/Egress 都生效；render、server dry-run、live object、CNI observed state hash 一致。
- `V103-AC06`：DNS 仅到批准 cluster DNS TCP/UDP 53；外部 DNS、DoT、DoH 和公共递归/转发失败并有日志。
- `V103-AC07`：每个必需批准内网 endpoint 在 IPv4/IPv6 实际适用族上成功，且 client nonce、服务端 receipt、CNI/网关 allow 三方同 correlation。
- `V103-AC08`：公共 FQDN/IP、未批准 RFC1918/ULA、link-local、node/pod/service CIDR 邻接目标全部拒绝；目标健康或明确 drop 侧证排除“目标宕机”。
- `V103-AC09`：metadata 通用及实际 provider 地址、K8s API 各寻址方式、DB/Redis/S3 非批准端点、同/跨 namespace Pod/headless/NodePort 全拒绝。
- `V103-AC10`：A/AAAA/CNAME/search/尾点/IDN/低 TTL/多地址/rebinding 逐次重判，任一解析到未批准地址 fail closed。
- `V103-AC11`：内网→公网/metadata/未批准内网的 HTTP redirect 每跳拒绝；协议降级和循环不绕过。
- `V103-AC12`：TLS SNI/Host/IP/CA 全维度匹配 registry；无 SNI、伪 Host、批准 IP+未批准 SNI 均失败。
- `V103-AC13`：HTTP(S)/ALL_PROXY、NO_PROXY、CONNECT、SOCKS、node-local/企业代理和 mesh/egress gateway 无公网或通配隧道能力。
- `V103-AC14`：sandbox 禁 hostNetwork/hostPID/hostIPC/hostPort/privileged/hostPath/额外 capability/SA token，真实 admission 拒绝证据完整。
- `V103-AC15`：未批准 sidecar/init/lifecycle/webhook/ephemeral/debug container 在 admission/RBAC 阶段拒绝；已批准容器也受相同网络策略与审计。
- `V103-AC16`：节点/子网无公网 route/NAT/代理兜底，IPv4/IPv6 边界防火墙 drop 与 Pod 探针同 run 关联；仅 NetworkPolicy render 不合格。
- `V103-AC17`：allow→删/错 port/SNI→deny 及 deny→精确 allow→allow mutation 在隔离 run 中符合预期，证明不是服务不可用或宽放行。
- `V103-AC18`：CNI flow/drop、DNS、节点/边界、可选内网 gateway 审计有同步时间、5-tuple/endpoint/run correlation 和明确 rule/action；查无日志不算 PASS。
- `V103-AC19`：任何 deny 意外成功立即 FAILED；任何 allow 因环境/服务不可用无法验证则 BLOCKED/FAILED，不被错误计为安全 PASS。
- `V103-AC20`：T01～T30 均有有效 Red/Green 映射，纯逻辑分支覆盖 ≥80%，触碰 TS/TSX changed-file ESLint 0 warning，相关与累计回归全绿。
- `V103-AC21`：证据绑定 source SHA/runId/operator/cluster/namespace/CNI/RuntimeClass/node/Pod UID/imageID/endpoint registry/policy hash/审计源/tool versions，artifact 非空且 SHA-256 全覆盖。
- `V103-AC22`：summary 从子结果重算，原子发布；空/陈旧/冲突/篡改/中断/敏感信息均阻断 PASS，失败证据保留并脱敏。
- `V103-AC23`：cleanup 仅按 `runId + managed-by` 幂等清理临时 canary/probe；不删共享策略/namespace，回滚后 default-deny 与节点公网拒绝仍在。
- `V103-AC24`：文档明确 `HELM_PASS ≠ NETWORKPOLICY_PASS ≠ POD_PROBE_PASS ≠ NODE_EGRESS_PASS ≠ V1_PASS`；Reviewer 与 Tester 各用独立 Agent 和全新 runId 复核，环境不可用只能 BLOCKED。

```bash
git diff --check
git status --short
```

- 专用测试名以 P04 验收后的实际结构为准，但须有单一可发现入口和 T01～T30 映射；不存在的命令/环境报告 BLOCKED，不得用 `|| true`、`--if-present` 或 stub 伪绿。
- 触碰 TS/TSX 另跑 `npx eslint --ext ts,tsx --report-unused-disable-directives --max-warnings 0 <files>`。全量 lint 的既有债务与 changed-file 结果分开记录，已激活的 V103 门仍必须绿。
- 本 Planner 交付是 docs-only，不运行实现/集群测试；Developer、Reviewer、Tester 必须记录真实命令、退出码、CST/UTC 时间、环境、runId 和报告路径。

### 8.2 动态执行顺序

1. preflight 读取 Day-0/registry/live cluster/CNI/节点边界/审计能力；故意给错 context、过期 endpoint、宽 CIDR，证明 apply/probe=0。
2. render + server-side dry-run + apply default-deny/最小 allow/probe/canary；归档脱敏 live object 和 Pod/RuntimeClass/node/imageID attestation。
3. 对每个 approved endpoint 执行 allow、删 allow/错 port/SNI 后 deny、恢复精确 allow 后 allow；收集 client/server/CNI/gateway correlation。
4. 执行 DNS/IPv4/IPv6/redirect/SNI/DoH/代理/metadata/K8s API/DB/Redis/横向矩阵；受控 canary 或明确 drop log 证明目标健康，拒绝不只靠 timeout。
5. 对 hostNetwork/hostPort/sidecar/init/lifecycle/webhook/ephemeral/debug fixture 做 admission dry-run 和隔离真实拒绝；验证普通批准 sidecar 仍受同一策略。
6. 从 Pod 和受控节点审计 probe 双侧证明无公网 route/NAT/代理；删除 artifact、改 hash、延迟/删 audit log、注入编码敏感串、中断发布与重复 cleanup 各重放一次。

### 8.3 证据包契约

`evidence-manifest.json` 至少包含：schema/harness 版本、source SHA、runId、时间/operator/CI run、context/cluster UID/minor/namespace/CNI/Pod Security、RuntimeClass/handler、node UID/kernel/architecture/runtime、Pod UID/platform/actual imageID digest、IP family/CIDR、endpoint registry hash/signatures、DNS/NetworkPolicy/admission/node-edge/gateway 配置 hash、预算和工具版本。

子产物至少包含：

- `preflight/`：前置验收、Day-0、approved registry、Service/Endpoint/DNS/SNI 对账、网络/审计能力；
- `manifests/`：render/server dry-run/live applied 的脱敏 NetworkPolicy/Pod/admission/RBAC 引用；
- `allow/`：每 endpoint 的 client/server/CNI/gateway correlation、nonce/hash 和时延；
- `deny/`：按 public/metadata/kube-api/data-plane/lateral/DNS/redirect/proxy/container-bypass 分类的结果和独立 drop 侧证；
- `audit/`：脱敏 DNS/CNI/node-edge/gateway 规则与事件摘要，不含 token/credential/完整 payload；
- `mutations/`、`cleanup.json`、`summary.json`、`checksums.sha256`。

证据先写同父目录临时目录，schema/hash/secret scan/summary 重算通过后原子 rename；同 runId 冲突、中断、任一声明 artifact 缺失/为空/hash 错或 cleanup 未完成均不能覆盖历史成功包。

## 9. 回滚、故障处置与独立角色交接

### 9.1 回滚必须保持 fail-closed

- 代码按 V103 分 commit 回滚；策略只可回到上一个已验收的 default-deny + 更小 allowlist。禁止删除 default-deny、增加公网/NAT/代理、整 namespace/RFC1918 放行来恢复业务。
- endpoint 变更按 registry 版本和精确 Service/IP/port/SNI 回滚；先撤应用对端点的依赖，再收窄 allow。旧 endpoint 到期/失效时保持拒绝，服务不可用按 BLOCKED/事故处理。
- 集群 cleanup 仅删除本 runId 临时 probe/canary/policy overlay；共享 default-deny、审计、节点防火墙和历史脱敏失败证据保留。任何 namespace-wide delete 被工具拒绝。
- 意外公网、metadata、K8s API 或横向可达时立即停止后续探针，隔离 workload，保留脱敏流量证据，轮换可能暴露的凭据并通知 SEC/PLAT；修复后必须新 runId 从 Reviewer 重走。
- break-glass 若业务另行授权，只能作为任务外事故流程；存在 break-glass 公网窗口时 V103 结论仍 FAILED/BLOCKED，不能 PASSED。

### 9.2 Developer → Reviewer → Tester

- Developer 按 `Round N · YYYY-MM-DD HH:mm · Agent/commit` 追加 Red→Green→Refactor、命令/退出码/环境/证据，对 AC01～AC24 定位；使用唯一 Developer runId，不自审、不写 Reviewer/Tester 结论、不勾总表。
- 独立 Reviewer 读计划、全部 diff、Red/Green 和原始证据，并必须用**不同于 Developer 的新 runId**至少重放 preflight、完整 public/metadata/K8s API/lateral deny、一个 approved allow、一个删 allow mutation和审计重算。无独立集群/审计权限时写 `BLOCKED`，不得 `APPROVED_FOR_TEST`。任一 P0/P1、缺 Red、宽 allow、仅 timeout/render、旁路未覆盖或证据可伪绿均 `CHANGES_REQUESTED`。
- 仅 Reviewer 以新 runId 得出 `APPROVED_FOR_TEST` 后，独立 Tester 才开始。Tester 必须再用**不同于 Developer/Reviewer 的全新 runId**重放完整矩阵，并新增至少一个 Developer/Reviewer 未覆盖角度：IPv4-mapped IPv6、CNAME/低 TTL rebinding、多跳 redirect、SNI/Host 错配、DoH/CONNECT/SOCKS、provider metadata 变体、hostNetwork、mutating sidecar、ephemeral RBAC、审计延迟或原子中断。
- Tester 环境/端点/审计不可用时结论为 `BLOCKED`；deny 意外可达或 allow 无法验证为 `FAILED`。不得复用任何旧 runId 写 `PASSED`。Tester FAILED 后退同一 Developer，修复后必须重新独立 Reviewer 新 runId；只有 Tester PASSED 且协调者核对三套独立证据后，协调者才可更新总体表。
