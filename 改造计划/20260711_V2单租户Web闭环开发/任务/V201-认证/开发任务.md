## 1. 开工硬门与三态

Developer 在写任何 V2 认证业务实现前必须同时满足：

1. V107 Phase A 的开发记录、审核意见、测试报告和协调者结论在同一稳定 SHA 上明确 `V2_DEVELOPMENT_AUTHORIZED`；no-go、授权被 V1 受控面撤销、缺签署或只有 V101～V106 局部 Green 都不能解锁。正常 V2 源码推进不要求 Phase A 与当前 HEAD 相同。
2. P01、P02 在总体完成表中均由协调者接受；记录 P01 contract version/source hash 与 P02 migration/stage SHA。P02 的 REVIEW_PENDING、APPROVED_FOR_TEST 或 Developer 自测 Green 都不等于 ACCEPTED。
3. 下节 Day-0 表所有 REQUIRED 项均由 SEC/BE/PLAT 批准，且只记录非秘密值或 secref；不得用 localhost、示例域名、临时公网 IdP 或空值补齐。
4. 真实内网 IdP 测试账号、禁用账号、固定单租户映射和密钥轮换窗口可用；账号凭据只能由 CI/Secret 管理注入。
5. 生产同构网络具备精确的 IdP/DNS/时间源 allowlist，公网、metadata 和任意未批准 endpoint 均不可达。

所有 runner 和报告只允许三种结果：

| 状态 | 语义 | 退出码 | 能否形成完成证据 |
|---|---|---:|---|
| PASS | 目标行为在指定 SHA/环境真实通过 | 0 | 可以 |
| FAIL | 环境已具备，但产品断言、mutation、安全或回归失败 | 1 | 不可以，退 Developer |
| BLOCKED | 缺硬门、真实 IdP、Secret 引用、网络/时钟/PG/Redis 等外部输入，测试未实际完成 | 2 | 不可以，任务保持 TODO |

## 3. 冻结架构与范围

### 3.1 唯一登录链路

V201 将 LobsterAI Auth 作为批准内网 IdP 的单 issuer relying party，同时作为浏览器与内部 API 的会话 broker：

1. SPA 生成高熵 state、nonce、PKCE verifier/challenge(S256)；verifier/state 只在当前 tab 的 sessionStorage 中短暂保存，回调成功或失败后一次性删除。
2. SPA 进入 LobsterAI /oauth/authorize；Auth 服务把内部 client_id、精确 app redirect、challenge、state、return_to 与固定 upstream issuer/client 绑定到一次性短期 transaction，并为上游另生成 state/nonce/PKCE。
3. Auth 服务只向 Day-0 authorization endpoint 重定向；上游回调只落批准的后端 callback。后端校验 state 后用固定 token endpoint 换码，验证 ID token 的签名、kid、alg、iss、aud/azp、sub、nonce、时间与 tenant claim。
4. 身份只以规范化后的 upstream iss+sub 关联内部 user；email/display name 只是属性，不能自动覆盖既有 identity 或作为唯一登录键。active user、active membership 与唯一默认 tenant 均通过后才签发内部一次性 authorization code。
5. 后端精确重定向到 app callback，仅携带内部 code、原始 state 和必要的错误码；绝不在 URL 放 access、refresh、ID token、client secret 或 WS ticket。
6. SPA 比对并消费 state，再以内部 code+verifier+同一 redirect URI 调 /oauth/token。Auth 服务以原子更新消费 code，签发内部短寿命 access JWT 和 opaque refresh；access 仅存内存，refresh 只进 HttpOnly Cookie。
7. /auth/refresh 只读 Cookie 且强制 CSRF/Origin；/oauth/token 的 refresh_token body 分支只为 Day-0 明确批准的非浏览器 client，V2 Web 不使用该分支。

这条链路保留附录 C D11 的“内部授权码再以 PKCE 换内部 token”语义，并补齐外部 OIDC 的 state/nonce/mix-up 防护。P01 当前 Auth DTO 缺 client/nonce/transaction、路由方法和 OIDC 错误细节；V201 必须先通过下述 C201 checkpoint 修改唯一事实源与生成物，禁止在 controller 另造 DTO。

### 3.1.1 C201 additive contract checkpoint

C201 属 V201 内部 Phase 0，不新增总体任务行。它的目的不是重开已接受 V105，而是把认证 additive delta 形成可审计契约血缘：`effectiveContractHash = V105AcceptedContractHash + C201CheckpointHash`。

1. 只允许新增 V201 Auth route/DTO/error/security metadata，或为既有 auth/ticket schema 增加向后兼容的 optional 字段；不得删除/重命名既有 operation/channel/error，不得改变 Cowork 10、WS control、bridge leaf target、既有 required/nullable/status 语义。
2. 先写 mutation Red：删除旧 route、把 optional 改 required、改既有 ticket/bridge/Cowork 语义时 checkpoint 必须非零；再最小更新 canonical Zod/registry，原子 generate，禁止手改 YAML/generated。
3. 固定执行 `CONTRACT_BASE_REF=<V105接受SHA> npm run contracts:check -- --breaking`、两次 `contracts:generate` + clean diff、`test:contract`、V105 全 bridge/10 事件/ticket/loopback conformance 回归。输出 base hash、delta manifest、derived hash、missing/extra/breaking 分类。
4. 独立 Contract Reviewer 只在 `breaking=0`、既有 V105 projection 全兼容时写 `C201_APPROVED_FOR_TEST`；独立 Contract Tester 再做 required-field、route removal、error reuse、ticket/Cowork mutation。协调者在 V201 记录中写 `C201_ACCEPTED_FOR_V201` 后才可进入认证 controller/service Green；这不勾 V201 或 V105 总表。
5. 若 delta 改变既有 V105 语义、无法以 additive minor 表达或 V105 conformance 失败，状态为 `BLOCKED_C201_BREAKING_CHANGE`，必须重开 V105/P01 对应角色链。不得通过改 hash 白名单保留旧接受。

### 3.2 会话、刷新与撤销

- 内部 access JWT 必须至少含 kid/alg、iss、aud、sub、tid、sid、jti、iat、nbf、exp、scope/role；sub/tid/sid 使用内部稳定 ID。不得把整份上游 token、group 列表或不受控 profile 塞入 JWT。
- refresh 为高熵 opaque 值，数据库只存带 pepper 的不可逆 hash、family/version/replacedBy/过期/撤销元数据；pepper/signing/private key 均只存 secref。
- 同 family 刷新用跨副本锁或数据库原子 CAS 串行；grace window 内合法并发必须得到同一有效 successor 且不能被迟到响应覆盖。若需暂存 successor 明文，只能放 secret-class、加密、短 TTL 的内存/Redis 记录；不得落 PG 或日志。也可用经安全评审的确定性 KDF，但必须有独立 mutation。
- 超 grace 且旧 version 重放时原子撤销整个 family；登出、禁用用户、停用 membership、back-channel logout 也撤销 family，并把 sid/jti 撤销状态维持到最大 access/ticket TTL。
- JWKS 轮换：固定 alg；正常缓存遵从有界 TTL；遇未知 kid 最多进行一次受控同步刷新，仍未知则 fail closed；并发未知 kid 合并为 single-flight，禁止请求风暴；保留旧 key 到所有旧 token 自然过期。
- clock skew 只接受 Day-0 小窗口，不能让已经明显过期、未来 iat/nbf 或超最大 token age 的 token复活。

### 3.3 API、WS 与单租户身份一致

- API Guard 只从有效内部 Bearer 构造唯一 AuthContext{sub,userId,tid,role,scope,sid,jti}；忽略并拒绝 body/query/header 中试图覆盖 tenant/user 的字段。inactive user/membership、撤销 sid 和错误 scope fail closed。
- 数据库请求使用 P02 安全 facade，在显式事务中把已验证 tid/userId 注入 transaction-local context；不得让 apps/api 导入裸 Prisma client。
- POST /api/v1/stream/ticket 只能使用同一 AuthContext。ticket 短期、一次性，内部绑定 userId/tid/sid/access jti、允许的 sessions/resourceSubscriptions、nonce/expiresAt；跨租户或不可见资源返回不泄露存在性的 404。
- WS 首帧必须原子消费 ticket，并重建与 REST 完全相同的 AuthContext；不得接受 URL JWT、refresh Cookie、长效 bearer 或客户端提交的 tid/userId。登出/撤销发生在签发后、消费前时必须拒绝；多 tab 各自 ticket，不共享消费状态。

### 3.4 logout 与 back-channel

- POST /auth/logout 先做 CSRF/Origin 校验，再原子撤销当前 family/sid并清 Cookie；即使上游 end-session 不可用，本地登出也必须成功且旧 access/refresh/ticket 均失效。
- 若 Day-0 支持 RP-initiated logout，只向批准 end_session_endpoint 发送允许字段，post_logout_redirect_uri 仍精确白名单；失败不回滚本地撤销。
- 若支持 back-channel logout，专用端点验证 logout_token 的固定 alg/kid/iss/aud/iat/jti、events claim，以及 sid 或 sub 绑定；logout_token 不能含 nonce，jti 一次性，重放幂等且不能误撤销其他 issuer/sub。若 IdP 不支持，本项明确 N/A，不伪造 callback。

### 3.5 明确不做

- 不实现多 issuer、多 tenant 切换、自助注册、email/password、忘记密码、MFA、社交 IdP 聚合、账户自动合并或外部 Beta。
- 不复用桌面 loopback callback、src/main/main.ts 的明文 auth_tokens、Electron SQLite 或供应商公网 OAuth；桌面认证保持兼容但不作为 Web 完成证据。
- 不实现 V204 Cowork 业务、V206 完整重连桥或 V208 全产品 E2E；本任务只冻结并验证它们依赖的 AuthContext 与 ticket 身份边界。
- 不把 ID token 当 API access token，不将 refresh/ID token存 localStorage/sessionStorage，不用 introspection 公网调用代替本地 JWT 验证。

## 4. 源码与数据边界

预期修改范围（以 Developer 实际 diff 为准，超出需先更新计划）：

- libs/shared/contracts/src：Auth route/DTO/error/security scheme、claim/ticket metadata 的唯一事实源；同步生成 OpenAPI/types/validators，不手改 generated。
- libs/server/auth/src：配置解析、OIDC transaction/client、JWT/JWKS、session/refresh、AuthContext、CSRF/logout；保持 server-only，apps/web 禁止 import。
- apps/api/src：薄路由适配、Guard、readiness 与统一错误信封；业务逻辑不堆入入口大文件，若会显著放大现有文件先提取小模块。
- apps/web/src：登录状态机、callback、内存 access、sessionStorage 一次性 transaction、silent refresh、logout 和安全错误页；所有可见文本必须 zh/en i18n。
- prisma/schema.prisma + 新的 forward migration：在 P02 接受版本上扩展 users/memberships/identities/auth_codes/refresh_tokens/oauth_clients 或等价最小模型及 RLS/索引；禁止修改 P02 已发布 migration。
- tests/auth、tests/integration/auth、tests/e2e/auth、tests/fixtures/auth 与 scripts/auth：可控 IdP、mutation、真实 IdP runner、三态报告和 secret scan。
- deploy/chart/config 仅增加非秘密配置和 ExternalSecret/Vault 引用；不得放 client secret、签名私钥、测试密码或 token。

数据硬约束：

- identities 以 issuer+subject 唯一；subject 非空且不接受可变 email 代替。
- auth transaction/code/refresh 均短寿命、单用；code 和 refresh 只存 hash，唯一索引与原子 used/revoked 更新防并发重放。
- tenant-scoped 身份/session 表保持 tenant_id NOT NULL（仅确有 pre-auth 全局表可例外并由 Reviewer确认），ENABLE+FORCE RLS 与 app.user_id/app.tenant_id 场景均有真实 PG 测试。
- 单租户也必须从 membership/default tenant 得到 tid；禁止硬编码跳过 claim/membership 校验。

## 8. 迁移、发布与回滚

### 8.1 数据与部署迁移

- 只新增 forward migration；不修改P02已发布 migration，不写down SQL，不手改 _prisma_migrations。
- expand-first：先建表/列/索引/RLS和兼容nullable字段→回填单tenant identity/membership映射→加唯一/NOT NULL约束→发布auth服务dark mode→真实IdP smoke→开启Web auth flag/Ingress。

- key rotation采用prepublish新公钥→双key签验→切active kid→等待旧token最大寿命+skew→移除旧key；每步可回退且证据绑定kid。

### 8.2 回滚

- 应用回滚：先关闭Web auth入口和新session签发，保留验证/撤销端点到旧token自然过期；撤销本轮测试family和ticket；不删除身份表、code/token审计元数据。
