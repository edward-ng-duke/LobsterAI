# 文件工作区与对象存储

> 本文档面向后端/平台工程师与全栈开发者，阐述如何把 LobsterAI 桌面端“本地文件系统工作目录 + 本地文件预览”的模型，改造成多租户 SaaS 下的“每租户持久工作区 + 对象存储 + 文件服务 API”。它是 07（OpenClaw 运行时编排与沙箱隔离）与 12（Artifacts 与预览改造）之间的“存储底座”章节：07 决定 Pod 与卷的编排、12 决定 Artifact 如何在前端渲染，而本文档定义两者共享的**工作区数据面**。

## 1. 现状：本地文件系统即工作区

桌面端的“工作区”本质是**用户本机的一个真实目录**，OpenClaw agent 与前端预览都直接读写本地路径。没有任何存储抽象层。

### 1.1 工作目录（cwd）概念

- 新用户默认工作目录：`~/lobsterai/project`，见 `src/main/coworkStore.ts:37-39`（`getDefaultWorkingDirectory`）。
- 会话 cwd 存在 `cowork_sessions.cwd`、全局默认存在 `cowork_config`（见 `src/main/sqliteStore.ts`），每个 agent 还可有独立 `agents.working_directory`。
- 任务型会话有一个容器目录 `.lobsterai-tasks`，`normalizeRecentWorkspacePath` 会把 `.../.lobsterai-tasks/...` 归一化回父目录，见 `src/main/coworkStore.ts:41-51`。
- “最近工作目录”通过 `get-recent-cwds` IPC 返回，实现落在 `getCoworkStore().listRecentCwds()`，见 `src/main/main.ts:8478-8481` 与 `src/main/coworkStore.ts:1536`。

### 1.2 用户如何“选目录”——原生对话框

| IPC 通道 | 实现位置 | 语义 | 是否可移植 |
|---------|---------|------|-----------|
| `dialog:selectDirectory` | `src/main/main.ts:8522-8534` | 弹原生目录选择器，返回绝对路径 | ❌ Electron-only |
| `dialog:selectFile` / `dialog:selectFiles` | `src/main/main.ts:8536+` | 弹原生文件选择器 | ❌ Electron-only |
| `get-recent-cwds` | `src/main/main.ts:8478-8481` | 最近使用的本地目录列表 | ⚠️ 语义需重定义 |

这三者共同构成“**本地路径心智模型**”：用户脑中的工作区 = 磁盘上一个能用 Finder/资源管理器打开的文件夹。SaaS 化后这一心智必须被**服务端虚拟工作区**替换（见 §7）。

### 1.3 文件读写与缩略图（renderer 借道 main 做 fs）

renderer 无 Node fs 能力，一切文件 IO 都经 IPC 落到 main 进程直接操作本地磁盘：

| IPC 通道 | 实现位置 | 语义 |
|---------|---------|------|
| `dialog:readFileAsDataUrl` | `src/main/main.ts:8663` | 读本地文件 → base64 data URL（图片/附件预览）|
| `dialog:saveInlineFile` | `src/main/main.ts:8602` | 把内联内容写到本地文件 |
| `dialog:statFile` | `src/main/main.ts`（`statFile` 组） | `fs.stat` 元信息 |
| `dialog:readTextFile` | `src/main/main.ts` | 读文本文件 |
| `dialog:generateThumbnail` | `src/main/main.ts:8760` | 生成缩略图 |
| `shell:openPath` / `shell:showItemInFolder` | `src/main/main.ts`（`shell:*` 组）| 在本机打开文件/定位到资源管理器 |

### 1.4 Artifact 文件：本地路径 + 文件监听 + 本地预览 server

Artifact 的“文件型”产物是 agent 在 cwd 里落盘的真实文件。前端拿到的是**本地绝对路径**，预览与刷新链路完全绑定本地 fs：

- `artifact:watchFile` / `artifact:unwatchFile`：用 `fs.watch` 监听本地文件变更，debounce 300ms 后 `webContents.send('artifact:file:changed', { filePath })`，见 `src/main/main.ts:8985-9024`。
- HTML/PPTX 高保真预览走本地 HTTP server，见 `src/main/libs/htmlPreviewServer.ts`：
  - `createPreviewSession(filePath)`：以文件所在目录为 `rootDir`，生成 `sessionId`（16B hex）与 `token`（24B hex），监听 `127.0.0.1:{随机端口}`，返回 `http://127.0.0.1:{port}/{sessionId}/{file}?token=...`，见 `htmlPreviewServer.ts:345-361`。
  - 请求校验：token 比对 + **目录穿越防护**——`path.resolve(session.rootDir, relativePath)` 后必须 `startsWith(session.rootDir)`，否则拒绝，见 `htmlPreviewServer.ts:282-285`。
  - 这套“token + rootDir 隔离 + 路径归一化校验”的思路，正是 SaaS 侧签名 URL 与路径安全的直接前身（见 §5）。

### 1.5 现状总结（问题清单）

```mermaid
flowchart TB
  R[Renderer] -->|dialog:*/artifact:*| M[Main 进程]
  M -->|fs.read/write/watch| D[(本机磁盘 ~/lobsterai/project)]
  M -->|127.0.0.1 preview server| D
  OC[OpenClaw gateway 本地进程] -->|同一磁盘 cwd| D
```

| 现状特征 | SaaS 化障碍 |
|---------|-----------|
| 工作区 = 本机磁盘目录 | 云端无“本机”；多租户需隔离与持久化 |
| renderer 靠 main 做 fs | Web 无 main 进程，需 REST/WS 文件服务 |
| Artifact = 本地绝对路径 | 浏览器不能访问服务端路径，需签名 URL |
| 预览 server 监听 loopback | 多租户需网关 + 鉴权 + 租户边界 |
| `fs.watch` 直连本地 | 需改为服务端事件 → WS 推送 |
| 无 tenant 维度 | 所有路径无 `tenant_id`，无法隔离/计费/清理 |

## 2. 目标：每租户持久工作区 + 文件服务

### 2.1 核心概念替换表

| 桌面端概念 | SaaS 目标概念 | 说明 |
|-----------|-------------|------|
| 本机目录 `~/lobsterai/project` | **工作区内用户可见项目子树** | 对应目标工作区的 `/workspace/project` 子目录，不等同于整个 OpenClaw state-dir |
| 会话 cwd（绝对路径） | 工作区内相对路径 `ws://{workspaceId}/project/{relPath}` | 对外只暴露 `/workspace/project` 下的相对路径，绝不暴露宿主绝对路径 |
| `dialog:selectDirectory` | 工作区选择/创建 API + 目录树浏览 | 见 §7 |
| `get-recent-cwds` | 最近工作区/最近文件 API（按 tenant） | 见 §7 |
| `fs.watch` | 服务端文件事件 → WS `files:changed` | 见 §4.5 |
| 本地 preview server | 对象存储签名 URL + 预览网关 | 见 §5 |

> **挂载单位 = 工作区（对齐 07 并锁定为附录 C D5）**：本文的「虚拟工作区」源自现状 OpenClaw state-dir 单元 `workspace-{agentId}`（main 为 `workspace-main`），但目标 PVC 物理子路径统一为 `tenants/{tenantId}/ws-{workspaceId}` 并挂载为 `/workspace`。目标布局把同一工作区拆成两个子树：`/workspace/state` 存 `AGENTS.md`、`MEMORY.md`、`SOUL.md` 等 OpenClaw/LobsterAI 托管状态；`/workspace/project` 是用户可见 cwd 根。挂载 / 隔离 / 锁的粒度仍统一为**整个工作区**，不是只锁 `/workspace/project`。`session : workspace` 为**多对一**（一个工作区可服务同租户多会话，见 §7.3）。此前三处口径打架（07 曾按会话 `sessions/{id}`、本文早期按 `ws-{id}`、源码按 agent `workspace-{agentId}`）已由附录 C D5 拍死为「按工作区」；`workspace-*` 只作迁移来源名称，`ws-{workspaceId}` 是目标物理子路径，不得再写成互为别名的两套落点。
>
> **同一工作区同一时刻只允许一个 RW 挂载（OpenClaw Pod）**，并发访问经分布式租约串行化，其余请求排队或降级只读（机制见 §6.3）。`MEMORY.md`、dreaming 产物、`AGENTS.md` 的 LobsterAI 托管段落只由**单一持锁 Pod** 写入，杜绝两个 Pod 以 RWX 并发写同一批文件/记忆造成损坏（见附录 C D5）。

### 2.2 两条持久化路线（决策）

工作区数据既要被 **OpenClaw Pod 以文件系统方式读写**（agent 必须能 `fs.writeFile`），又要被 **文件服务与前端以对象方式访问**。这产生两种落地形态：

| 方案 | 描述 | 优点 | 缺点 | 采用场景 |
|------|------|------|------|---------|
| **A. PVC 权威 + 对象存储镜像** | 工作区真身是 K8s PVC（每租户一个 PVC，见 07）；文件服务读写 PVC；对象存储作为**导出/分享/预览/备份**的镜像 | agent 直接 POSIX 读写，语义与桌面端一致；低延迟 | 需 PVC↔对象存储同步逻辑；PVC 扩缩容运维 | GA 主线推荐，但必须先通过 V1 存储 PoC |
| **B. 对象存储权威 + 同步进 Pod** | 工作区真身在对象存储；Pod 启动时把前缀同步进 emptyDir，退出时回写 | 存储弹性强、天然多副本、成本可控 | agent 读写需同步窗口，一致性复杂；大工作区拉起慢 | GA 后续 / 冷工作区归档 |

**GA 主线默认：采用方案 A（PVC 权威 + 对象存储镜像）**，但它必须先通过 V1 存储 PoC，不能把 RWX/PVC 一致性当作默认成立。理由：

1. OpenClaw agent 需要真实 POSIX 文件系统语义（`fs.watch`、追加写、目录遍历），PVC 最贴近桌面端行为，迁移成本最低。
2. 对象存储承担“**对外**”职责：Artifact 签名 URL 预览、HTML 分享、大文件下载、跨会话/跨设备访问、备份归档，避免让 PVC 直接对公网暴露。
3. 方案 B 的“同步窗口一致性”在 GA 主线初期引入不必要的复杂度，留给后续做冷热分层。

> 与 07 的接口约定：07 负责“每租户一个 PVC、如何在 OpenClaw Pod 里 `volumeMount` 到 `/workspace`”；本文档只要求 07 保证 **文件服务 Pod 与 OpenClaw Pod 挂载同一 PVC 的同一子路径**（见 §6）。

### 2.3 从旧容器 HOME 卷到 SaaS 工作区的拆分

原容器改造计划使用“每容器独立 `HOME` + 一个持久卷”保存所有状态，这在桌面托管 PoC 中很有效：`~/.config/LobsterAI`、SQLite、OpenClaw state、默认工作目录和缓存都随卷保留。但 SaaS GA 不能把整份 `HOME` 作为租户工作区直接暴露或共享，必须拆成三类：

| 旧容器卷里的内容 | SaaS 归属 | 说明 |
|---|---|---|
| SQLite 主库、kv、会话、agent、技能启用态 | PostgreSQL / 服务端 DB | 由 `06` 迁移；不挂入 Sandbox Pod |
| OpenClaw 运行时动态 state：`openclaw.json`、gateway token、port、临时日志 | Pod `/state` 运行时动态 state 卷 / Secret / stdout | 由 `07` config sync 和 K8s Secret 管理，不进用户可浏览工作区；不同于本章 `/workspace/state` 子树里的 AGENTS/MEMORY/SOUL 等托管工作区状态 |
| 用户项目文件、agent 产物、可见工作目录 | PVC `/workspace/project` + S3 镜像 | 本文档主责，向前端和 File Service 暴露逻辑路径 |
| 编译缓存、浏览器缓存、临时文件 | emptyDir / 只读缓存 / 丢弃 | 可用于预热容量，但必须不含租户数据 |

桌面/旧容器导入时可以先把 `HOME` 打包作为输入，但导入器必须按上述归属拆分，不能把 `~/.config/LobsterAI` 原样变成可浏览 workspace。

### 2.4 V1 存储 PoC 门槛

在固化方案 A 之前，V1 必须对候选 RWX 存储（EFS/CephFS/NFS/云厂商等价服务）跑完以下基准。任一 P0 项失败，都不能进入 V3 工作区产品化。

| 类别 | 测试 | 通过标准 |
|---|---|---|
| 读写可见性 P0 | OpenClaw Pod 写文件，File Service Pod 读取；File Service 写文件，OpenClaw Pod 读取 | p95 可见延迟 < 500ms，p99 < 2s；内容校验和一致 |
| 并发一致性 P0 | 同一 workspace 20 并发小文件创建/覆盖/删除 | 无丢文件、无脏读、无目录项残留 |
| 单写者租约 P0 | 两个 OpenClaw Pod 争抢同一工作区并发写 `MEMORY.md` / 同一文件（D5） | 租约仅授予一个 RW Pod，另一方排队或降级只读；**写入不丢、不交错、不损坏** |
| 大目录 P0 | 单目录 10k 文件 listing、分页、stat | p95 listing < 2s；API 有分页，不一次性返回全量 |
| 文件事件 P1 | inotify/fsnotify 在跨 Pod 场景下传播 | 若不可靠，必须改用 OpenClaw 文件写事件 + File Service 扫描补偿，不能依赖原生 watch |
| 断连恢复 P0 | PVC 短暂不可用/节点迁移后恢复 | Pod 不越权重挂；失败状态可观测；不会把空目录误同步覆盖 S3 |
| 安全 P0 | symlink、`../`、hardlink、绝对路径、特殊文件 | 全部被路径解析与挂载边界拒绝 |
| state 隔离 P0 | 浏览器/API 尝试列出 `/workspace/state`、`.lobsterai-tasks/`、`.trash/`、token、config、日志 | 返回 404/403，不泄露存在性和内容 |

若 V1 证明 RWX/PVC 方案不可接受，替代路线是：File Service 成为唯一文件 IO 代理，OpenClaw 通过受控 FUSE/sidecar/API 访问 workspace，或改为对象存储权威 + 会话级同步。替代路线必须重新评估性能、POSIX 兼容与 OpenClaw 改造量。

## 3. 存储布局与命名约定

### 3.1 卷（PVC）目录布局

```
/data/tenants/{tenantId}/
├── ws-{workspaceId}/               # 一个虚拟工作区（= OpenClaw workspace-{agentId}/workspace-main）
│   ├── state/                      # OpenClaw/LobsterAI 托管状态：AGENTS.md、MEMORY.md、SOUL.md...
│   ├── project/                    # 用户可见 cwd 根（对应旧 ~/lobsterai/project）
│   │   ├── ...(agent 落盘的代码/文档/artifact 文件)
│   │   ├── .lobsterai-tasks/{taskId}/  # 任务型会话容器（沿用桌面端约定，不暴露为普通项目目录）
│   │   └── .trash/                     # 软删除回收站（配额清理前的暂存）
└── quota.json                      # 该租户配额快照（可选，权威在 DB）
```

- **文件服务/API 对外暴露的是相对 `/workspace/project` 的相对路径**，形如 `docs/report.md`；宿主绝对路径、`/workspace/state`、`project/.lobsterai-tasks/`、`project/.trash/` 永不出现在普通文件树 API 与前端。
- OpenClaw Pod 挂载整个 `ws-{workspaceId}` 作为工作区卷；进程 cwd 设置为 `/workspace/project`（见 §6），因此 agent 视角的 `.` = 用户可见项目根；Config Sync / runtime sidecar 只通过受控路径写 `/workspace/state`。

### 3.2 对象存储 key 命名（严格分层，租户前缀在最前）

```
s3://lobsterai-workspaces/
  tenants/{tenantId}/ws-{workspaceId}/files/{relPath}          # 文件镜像
  tenants/{tenantId}/ws-{workspaceId}/artifacts/{artifactId}/  # artifact 快照
  tenants/{tenantId}/ws-{workspaceId}/thumbnails/{hash}.webp   # 缩略图缓存
  tenants/{tenantId}/uploads/{uploadId}/{filename}             # 直传暂存区
  tenants/{tenantId}/backups/ws-{workspaceId}/{ts}.tar.zst     # 备份归档
```

**硬性约束**（详见 `14-安全合规与多租户隔离.md`）：

- 所有 key 第一段必须是 `tenants/{tenantId}/`；文件服务在生成/解析 key 时强制拼接当前请求上下文的 `tenantId`，**绝不接受客户端传入完整 key**。
- 签名 URL 的作用域必须限制到具体对象 key，不得签发前缀级或桶级 URL。
- 桶策略 + IAM 只允许文件服务的 ServiceAccount 访问；前端只拿到短时效对象级签名 URL。

### 3.3 工作区元数据（DB，schema 骨架）

> 完整迁移方案见 `06-数据模型迁移.md`；此处给数据面所需最小 schema。

```prisma
model Workspace {
  id           String   @id @default(uuid()) @db.Uuid
  tenantId     String   @map("tenant_id") @db.Uuid
  name         String
  pvcPath      String   @map("pvc_path")      // 工作区根，如 ws-{workspaceId}；用户文件根为 {pvcPath}/project
  s3Prefix     String   @map("s3_prefix")     // tenants/{tenantId}/ws-{workspaceId}/
  sizeBytes    BigInt   @default(0) @map("size_bytes")   // 后台巡检更新
  quotaBytes   BigInt   @map("quota_bytes")                 // 该工作区配额（可继承租户级）
  status       String   @default("active")       // active | archived | deleting；值域由 contracts 生成常量约束，不用 PG 原生 enum
  lastAccessAt DateTime @default(now()) @map("last_access_at")
  createdAt    DateTime @default(now()) @map("created_at")
  updatedAt    DateTime @updatedAt @map("updated_at")
  deletedAt    DateTime? @map("deleted_at")

  @@index([tenantId])
  @@index([tenantId, status])
  @@index([tenantId, lastAccessAt])
  @@map("workspaces")
}

model WorkspaceFileEvent {   // 供 watch/审计，可选做 outbox
  id          String   @id @default(uuid()) @db.Uuid
  tenantId    String   @map("tenant_id") @db.Uuid
  workspaceId String   @map("workspace_id") @db.Uuid
  relPath     String   @map("rel_path")
  kind        String   // created | modified | deleted | renamed
  actor       String   // agent | user | system
  at          DateTime @default(now())

  @@index([tenantId, workspaceId, at])
  @@map("workspace_file_events")
}
```

- `Workspace` / `WorkspaceFileEvent` 是 tenant-scoped 表，必须按 `06` §2.1 与附录 C D2 启用 `tenant_id` UUID、RLS `ENABLE + FORCE`、`SET LOCAL app.tenant_id`；上方 Prisma 只给数据面最小字段，不得退回 `cuid()` 或无租户前缀索引。
- `cowork_sessions.cwd` 从“绝对路径”改为“`workspaceId` + `project/` 子树内 `relRoot`”二元组；普通会话 cwd 只能落在 `/workspace/project` 下，不能指向 `/workspace/state`、`.trash/` 或配额元数据（迁移细节见 06）。
- 最近工作区/最近文件用 `Workspace.lastAccessAt` 与 `WorkspaceFileEvent` 派生，替代 `get-recent-cwds`。

## 4. 文件服务 API 设计

文件服务是一个 NestJS 模块（`FilesModule`），所有接口都在 `tenantId`（来自 JWT，见 `05-认证与多租户账户.md`）+ `workspaceId`（路径参数，服务端校验归属该租户）双重作用域下工作。

> **契约与错误信封（对齐附录 C D1 / §3.4）**：本节路径 / 方法只作导航，**字段级 request/response/error DTO 以 `libs/shared/contracts` 为唯一权威**。所有 4xx/5xx 一律返回统一 JSON 错误信封（不返回裸文本 / 裸状态码）：
>
> ```jsonc
> { "error": { "code": "PAYLOAD_TOO_LARGE", "message": "…", "requestId": "…", "details": {} } }
> ```
>
> 本文件服务用到的错误码及状态：`PAYLOAD_TOO_LARGE`（**413**，文本 / body 过大，提示改走 `as=url` 或分片上传）、`STORAGE_QUOTA_EXCEEDED`（**507**，超存储配额）、`FORBIDDEN_PATH`（**403**，路径越界 / symlink 逃逸）、`NOT_FOUND`（**404**，工作区 / 文件不存在、跨租户或同租户未授权 workspace，不泄露存在性）。**注意存储配额必须使用 `STORAGE_QUOTA_EXCEEDED`，不得复用模型/账单额度的 `QUOTA_EXCEEDED`**（附录 C §3.4 把二者拆开）。

### 4.2 目录浏览响应（替代原生目录选择器）

```http
GET /api/v1/workspaces/11111111-1111-4111-8111-111111111111/files/tree?path=docs&depth=1&cursor=&limit=1000
```

```jsonc
{
  "workspaceId": "11111111-1111-4111-8111-111111111111",
  "path": "docs",              // project/ 子树内相对路径；根为 ""
  "entries": [
    { "name": "assets",    "type": "dir",  "size": 0,     "mtime": "2026-07-01T10:00:00Z" },
    { "name": "report.md", "type": "file", "size": 20480, "mtime": "2026-07-02T09:00:00Z", "mime": "text/markdown" }
  ],
  "nextCursor": null,         // 非 null 时携带它继续拉下一页；null 表示已到末页
  "truncated": false          // 目录项超过 limit 时为 true，用 nextCursor 续拉
}
```

- `path` 一律相对 `/workspace/project` 子树根；普通文件 API 不接受 `.lobsterai-tasks/`、`.trash/` 这类 `project/` 下控制目录路径，也不接受任何指向 `/workspace/state` 的路径，服务端做 §5 的路径安全校验。
- **稳定排序键（对齐附录 C §3.4）**：目录项按 `(type, name, id)` 排序（`type` 先 `dir` 后 `file`，同类按 `name`，再按稳定 `id` 去歧义），保证分页翻页边界不重不漏。
- **`depth` 语义**：`depth=1` 只返回 `path` 直属子项（默认）；`depth=N` 展开 N 层子树；`depth` 上限受配置约束，超限按上限截断并置 `truncated`。
- **`cursor` 编码**：`cursor` 是对末条记录稳定排序键 `(type, name, id)` 的不透明 base64 编码（不是偏移量 / `sequence`），服务端据此 seek 续拉；`limit` 单次上限（如 1000 项），超出置 `truncated=true` 并回 `nextCursor`。字段级定义以 `libs/shared/contracts` 为准。

### 4.3 读文件（文本 vs 签名 URL）

统一入口按 `as` 分流，避免桌面端 `readFileAsDataUrl` 把二进制塞进 IPC：

```http
GET /api/v1/workspaces/11111111-1111-4111-8111-111111111111/files/download?path=docs/report.md&as=text
-> 200 { "path": "docs/report.md", "encoding": "utf-8", "content": "..." }

GET /api/v1/workspaces/11111111-1111-4111-8111-111111111111/files/download?path=assets/cover.png&as=url
-> 200 { "path": "assets/cover.png", "url": "https://s3.../...&X-Amz-Signature=...", "expiresIn": 300, "mime": "image/png" }
```

- `as=text`：仅当 mime 为文本且 size ≤ 阈值（如 1MB）；否则返回 `413 PAYLOAD_TOO_LARGE`（统一错误信封，`details` 带 `size` / `limit`），并提示改用 `as=url` 拉取——注意这条 413 是「文本过大」而非「超存储配额」，后者是 `507 STORAGE_QUOTA_EXCEEDED`（附录 C §3.4）。
- `as=url`：文件服务确保对象存储镜像是最新的（见 §4.6 同步语义），再签发**对象级、5 分钟时效**的签名 GET URL。前端 `<img>/<video>/iframe` 直连对象存储，绕过应用服务器带宽（见 12 章 Artifact 预览）。

### 4.4 上传（小文件直写 vs 大文件直传/分片）

```mermaid
sequenceDiagram
  participant UI as SPA
  participant FS as File Service
  participant S3 as 对象存储
  participant PVC as PVC

  Note over UI,FS: 小文件 (≤5MB)
  UI->>FS: POST /api/v1/workspaces/:wid/files/upload {path:"notes.txt", body/base64}
  FS->>FS: 路径校验 + 配额检查
  FS->>PVC: 写入
  FS-->>S3: 异步镜像
  FS-->>UI: 200 {etag, size}

  Note over UI,S3: 大文件 / 媒体
  UI->>FS: POST /api/v1/workspaces/:wid/uploads {filename, size, mime}
  FS->>FS: 配额预留 + 生成 uploadId
  FS-->>UI: {uploadId, presignedPut 或 分片列表}
  UI->>S3: PUT 分片 (直传, 绕过应用服务器)
  UI->>FS: POST /api/v1/workspaces/:wid/uploads/:id/complete {parts}
  FS->>S3: CompleteMultipartUpload
  FS->>PVC: 从暂存拉入工作区目标路径
  FS-->>UI: 200 {path, size}
```

- 直传/分片阈值建议 5MB（单请求）/ 8MB 分片；超大媒体（视频）必须分片 + 断点续传。
- **大文件方向定死（附录 C §3.4）**：一律 **S3 直传暂存 → 异步落地 PVC**，不走「浏览器 → File Service → PVC」的服务器中转（避免应用服务器带宽与内存压力）。`complete` 后由后台 worker 把暂存对象拉入工作区目标路径。
- **落地状态机 + 幂等/重试**：`upload` 记录在同步表上以状态机推进 `pending → uploaded(S3 暂存就绪) → landing → landed | failed`；`landing` 失败进 BullMQ 重试队列，重试以 `uploadId` 为**幂等键**（同一 `uploadId` 重复 `complete` / 重试不产生重复落地、不重复累加配额）；`failed` 超阈值告警并保留暂存对象供人工/自动回收。DB `sizeBytes` 仅在状态到达 `landed` 后对账更新（当前用量的权威读源见 §8.1，不以 DB 为准）。完整同步表 DDL 见附录 C §3.4 / 06。
- `complete` 时才把对象“落地”进工作区（PVC 目标路径 + 状态置 `landed` + 发 `files:changed`）。
- 配额在 `POST /api/v1/workspaces/:wid/uploads` 阶段对 **Redis 预留计数**做原子 `INCRBY` 预留（reserve），`landed` 时按真实大小结算回补，超时未 `complete` 则原子回补释放（见 §8.1）。

### 4.5 文件变更订阅（替代 `fs.watch` + `artifact:file:changed`）

传输遵循全局约定：REST 请求/响应、WebSocket 流式（见 `03-前端与传输层改造.md`）。

```mermaid
sequenceDiagram
  participant Agent as OpenClaw Pod
  participant PVC as PVC
  participant FS as File Service (watcher)
  participant Redis as Redis Pub/Sub
  participant UI as SPA (WS)

  Agent->>PVC: 写 report.html
  FS->>PVC: inotify/chokidar 检测变更 (debounce 300ms)
  FS->>Redis: publish files:{tenant}:{ws} {relPath, kind}
  Redis-->>FS: (广播到所有 FS 副本)
  FS-->>UI: WS StreamEnvelope filesChanged {workspaceId, relPath, kind}
  UI->>FS: (若在预览) 重新拉取 as=url 签名 URL
```

- 桌面端 `fs.watch` + 300ms debounce（`src/main/main.ts:8985-9005`）语义**原样保留**，只把“watch 单个 filePath”升级为“watch 工作区/子路径”。
- 多副本文件服务下，watcher 事件先进 **Redis Pub/Sub**（技术栈既定），再由持有该用户 WS 连接的副本推给前端，避免事件丢在别的副本。
- OpenClaw Pod 与文件服务挂**同一 PVC**，因此 agent 落盘能被文件服务的 inotify 捕获（见 §6 卷共享注意事项：ReadWriteMany + inotify 传播）。

### 4.6 PVC ↔ 对象存储同步语义（方案 A 的关键）

| 触发点 | 同步方向 | 时机 |
|--------|---------|------|
| `POST /api/v1/workspaces/:wid/files/upload`（小文件写） | PVC → S3 | 写 PVC 成功后异步镜像；镜像失败进重试队列（BullMQ）|
| `as=url` 读取 | 确保 S3 是最新 | 若 PVC mtime > S3 lastModified，则先同步再签发 |
| agent 落盘（watch 触发） | PVC → S3 | watch 事件入队，后台 worker 镜像热文件/被预览文件 |
| 备份 | PVC → S3 backups | 定时打包（见 §8）|

- 一致性策略：**PVC 为权威**，S3 为最终一致镜像。签发 `as=url` 前做“新鲜度校验”（比对 mtime/etag）以避免预览到旧内容。
- 冷文件不必立即镜像，按“被访问/被预览/被分享”懒同步，控制对象存储写放大。

## 5. 路径安全（防跨租户 / 目录穿越）

这是数据面最关键的安全控制点，直接继承桌面端 `htmlPreviewServer.ts` 的 rootDir 校验思路（`htmlPreviewServer.ts:282-285`）并强化。

### 5.1 相对路径归一化与越界拒绝

所有涉及 `path` 参数的接口，统一走一个 `resolveWorkspacePath` 守卫：

```ts
// 伪代码：文件服务统一路径守卫
function resolveWorkspacePath(projectRootAbs: string, relPath: string): string {
  // 1) 拒绝绝对路径、盘符、协议、NUL
  if (path.isAbsolute(relPath) || /^[a-zA-Z]:[\\/]/.test(relPath) || relPath.includes('\0')) {
    throw new ForbiddenPathError();
  }
  // 2) 归一化后拼接
  const resolved = path.resolve(projectRootAbs, relPath);
  // 3) 越界校验：必须仍在 project/ 根内（末尾补 sep 防前缀混淆）
  const rootWithSep = projectRootAbs.endsWith(path.sep) ? projectRootAbs : projectRootAbs + path.sep;
  if (resolved !== projectRootAbs && !resolved.startsWith(rootWithSep)) {
    throw new ForbiddenPathError();
  }
  return resolved;
}
```

要点（超出桌面端的强化项）：

1. **`..` 穿越**：`path.resolve` 归一化后必须仍 `startsWith(root + sep)`，与 `htmlPreviewServer.ts:285` 同理，但补齐末尾分隔符防 `/data/ws-abc` 前缀混淆 `/data/ws-abcdef`。
2. **符号链接逃逸**：agent 可能在工作区内创建指向 `/etc` 的 symlink。写入/读取前对最终目标做 `fs.realpath` 再次校验仍在 root 内；或挂载卷时禁用 symlink 跟随（`nosymfollow` / 由 07 的 Pod 安全策略保证）。
3. **绝对路径 / 盘符**：直接拒绝，防止 `path.resolve` 把绝对路径“吃掉” root。
4. **租户 / 授权越界**：`workspaceId` 必须先经 DB 校验 `Workspace.tenantId === ctx.tenantId` 且当前用户/会话被授权访问，再取其 `pvcPath`；任何请求都不能用别人的或同租户未授权的 `workspaceId`（统一返回 404 而非 403，避免存在性泄露）。
5. **key 拼接不可信客户端**：对象存储 key 一律由服务端用 `tenants/{ctxTenantId}/ws-{wid}/...` 拼接，客户端只给相对 path。

### 5.2 签名 URL 的边界

| 项 | 约定 |
|----|------|
| 作用域 | 单个对象 key（GET 或分片 PUT），禁止前缀/桶级 |
| 时效 | 读 300s、上传 PUT 900s；过期即失效 |
| 方法 | 读只签 GET；上传只签 PUT，禁止 DELETE/List |
| 内容处置 | 下载类附 `response-content-disposition=attachment`，防 HTML 在存储域执行脚本 |
| 域隔离 | 用户内容存储域与应用主域分离（独立子域/桶），配合 CSP（见 12、14）|

### 5.3 预览与分享的隔离（与 12 衔接）

- HTML/SVG artifact：桌面端用 loopback + token（`htmlPreviewServer.ts`），SaaS 改为“对象存储签名 URL + 内容处置头 + 独立预览域 + iframe sandbox”，具体渲染策略见 `12-Artifacts与预览改造.md`。
- HTML 公开分享（对应桌面端 `htmlShare:*`，youdao 云能力）在自建后端重建：把 artifact 快照写入 `.../artifacts/{artifactId}/`，签发**可选带过期的公开 URL**，访问模式（public/link/private）记入 DB（见 09/12）。

## 6. 与 OpenClaw Pod 的卷共享（细节与 07 一致）

本节只给**存储侧**的接口与约束；Pod 编排、gVisor/Kata 加固、每会话 Pod 生命周期见 `07-OpenClaw运行时编排与沙箱隔离.md`。

### 6.1 共享方式

```mermaid
flowchart LR
  PVC[(租户 PVC<br/>subPath: ws-{workspaceId}<br/>RWX)]
  FS[File Service Pod] -->|mount /data/tenants/T/ws-{workspaceId}| PVC
  OC[OpenClaw Pod] -->|mount /workspace -> ws-{workspaceId}| PVC
  W[Watcher sidecar / FS 内 watcher] -->|inotify| PVC
```

| 约定项 | 值 | 说明 |
|--------|----|----|
| 访问模式 | **ReadWriteMany (RWX)** | RWX 用于「文件服务 + **单个** OpenClaw RW Pod」同挂一卷；**同一工作区不得有第二个 OpenClaw Pod 并发 RW**，多会话经 §6.3 单写者租约串行化（附录 C D5）。桶式 CSI（如基于 NFS/CephFS/EFS 的 RWX）|
| OpenClaw 挂载点 | `/workspace` → `tenants/{tenantId}/ws-{workspaceId}`；进程 cwd = `/workspace/project` | agent 的 `.` = 用户可见项目根（对齐桌面端 `~/lobsterai/project`）；`/workspace/state` 只给 Config Sync/runtime 受控访问 |
| 文件服务挂载点 | `/data/tenants/{T}` | 只挂载本租户子树；跨租户不共享卷 |
| 隔离粒度 | 每租户一个 PVC（GA 主线）| 同租户多工作区共用 PVC 下不同 `ws-{workspaceId}` 子目录；跨租户物理隔离 |
| 只读/受控目录 | 普通 agent / 用户文件 API 不得直接写 `/workspace/state`、`project/.trash/`、`quota.json` 等控制目录 | 普通工具进程 cwd 限在 `/workspace/project`；Config Sync/runtime sidecar 经受控路径写 `/workspace/state`；任务运行器按受控协议写 `project/.lobsterai-tasks/` |
| 写者租约 | 每工作区同一时刻 ≤ 1 个 RW OpenClaw Pod | 分布式租约（Redis + PVC lease 文件），见 §6.3 与附录 C D5 |

### 6.2 关键注意事项（RWX + watch）

1. **RWX + inotify 传播**：并非所有 RWX 存储都保证跨节点 inotify 事件传播。若底层不传播，则退化为“文件服务侧轮询 mtime”或“OpenClaw runtime adapter 在 agent 写文件工具返回时主动上报文件事件”（07 的 adapter 已有事件通道，可复用）。**推荐**：优先由 OpenClaw 侧“文件写工具完成事件”驱动 `files:changed`，inotify 作为兜底，避免依赖存储实现细节。
2. **敏感状态目录不入用户可浏览树**：OpenClaw 的 `AGENTS.md`/`MEMORY.md`/`SOUL.md` 等 state 内容（桌面端在 `{userData}/openclaw/state/workspace-*`）在 SaaS 仍属于同一工作区租约单元，但放在 `/workspace/state` 子树，与**用户可浏览的 `/workspace/project` 子树分离**，避免用户通过文件服务读写 agent 内部记忆/身份文件。归属与同步由 07 + 06 处理，本文件服务**不**把 `/workspace/state` 暴露到 `/tree`。
3. **记忆/托管段落写入归属单一持锁 Pod（附录 C D5）**：`MEMORY.md`、dreaming 产物、`AGENTS.md` 的 LobsterAI 托管段落物理上落在同一工作区卷的 `/workspace/state` 子树，也**只由当前持有 §6.3 写者租约的那个 Pod / Config Sync entrypoint 写入**；未持锁的会话 Pod 对这些文件只读，杜绝两个 Pod 并发写导致记忆交错/损坏。
4. **权限矩阵**：文件服务 SA、OpenClaw Pod SA 均无跨租户卷挂载权限；由 07 的 K8s RBAC + PVC 命名空间隔离保证。

### 6.3 单写者租约（对齐附录 C D5）

因为 `session : workspace` 是多对一，同租户可能有多条会话（多个 OpenClaw Pod）指向**同一工作区**。若放任它们以 RWX 并发写同一批文件与 `MEMORY.md`，会出现丢写/交错/损坏。D5 拍死的口径是**同一工作区同一时刻只允许一个 RW 挂载**。本节给最小可实现协议；字段级 schema 进入 `libs/shared/contracts`。

**租约对象（`WorkspaceLease`）。**

```ts
type WorkspaceLeaseMode = 'rw' | 'readonly';

interface WorkspaceLease {
  workspaceId: string;
  tenantId: string;
  holderPodId: string;
  holderSessionId: string;
  mode: WorkspaceLeaseMode;
  fencingToken: string;      // 单调递增或强随机且可比较；每次写入必须携带
  version: number;           // CAS / 乐观锁版本
  expiresAt: string;         // ISO timestamptz
  renewAfter: string;        // 建议续租时间，通常为 TTL 的 1/3-1/2
  reason?: string;           // acquire / renew / failover / readonly-fallback
}
```

**存储与授予规则。**

- 分布式租约 = Redis 键 `rt:workspace-lease:{tenantId}:{workspaceId}`（TTL、`holderPodId`、`holderSessionId`、`fencingToken`、`version`；与 07/14/16 的 runtime 路由/租约命名空间一致）+ 工作区卷内 `.lobsterai/lease.json`（故障恢复自证）。正常路径以 Redis CAS/SET NX 为准；lease 文件只用于 reconcile 与 Redis 故障后的保守恢复，不能单独授予新 RW；不得另建 `lease:ws:*` 第二套锁，否则会导致同一工作区出现双持有者脑裂。
- 获取租约时必须传 `waitMs` 与 `requestedMode`。`requestedMode='rw'` 且当前已有持有者时，编排器按 FIFO/公平队列等待；`waitMs` 超时返回统一错误信封 `409 WORKSPACE_WRITE_LEASE_HELD`，`details` 至少包含 `workspaceId`、`holderSessionId`（可脱敏）、`estimatedRetryAfterMs`、`canOpenReadonly:true`。
- 只读会话不需要 RW 租约，可浏览树、读文件、拉 `as=url`、接收 `files:changed`；任何写入、上传落地、`MEMORY.md`/`AGENTS.md` 托管段落更新、skill/MCP 对 workspace 的写操作都必须携带当前 `fencingToken`，否则返回 `423 WORKSPACE_READONLY_OR_LEASE_REQUIRED`。
- 正常退出释放租约；Pod crash / node drain 后由 TTL 过期接管。旧 Pod 若恢复并继续写，文件服务 / OpenClaw 文件写代理必须校验 `fencingToken`，token 不匹配即拒绝写入并触发 Pod 自杀或降级只读，防止脑裂迟到写。
- 续租周期不得依赖客户端连接；由 owning Pod 或 Runtime Orchestrator 后台续租。连续两次续租失败时，Pod 进入 drain：停止新 turn、尝试收尾当前写入、对前端广播 `workspaceLeaseLost`，再降级只读或退出。

**用户体验与错误语义。**

- 前端在同一工作区已有活跃 RW 会话时，新的会话启动页显示“等待工作区写入租约”状态，并提供“只读打开”和“取消等待”两个动作；不允许静默卡住。
- `WORKSPACE_WRITE_LEASE_HELD` 用 409，表示可以重试或排队；`WORKSPACE_READONLY_OR_LEASE_REQUIRED` 用 423，表示当前请求需要写锁但调用方只有只读能力。
- 权限与租约错误都必须进入统一错误信封，不能返回裸 409/423；错误码进入 `libs/shared/contracts` 的错误枚举。

**验收。**

- 两个 Pod 同时申请同一工作区 RW：只有一个拿到 `mode='rw'`，另一方在 `waitMs` 内排队或返回 `WORKSPACE_WRITE_LEASE_HELD`。
- 旧持有者 TTL 过期后恢复写入：因 `fencingToken` 失效被拒，不能覆盖新持有者写入。
- 并发写同一文件/`MEMORY.md`/`AGENTS.md` 托管段落不丢写、不交错、不损坏（对应 §2.4「单写者租约 P0」行、§10.3 与 16 的并发写门）。

## 7. 用“虚拟工作区”替代“选本地目录”

桌面端“选一个本机文件夹当 cwd”的交互，在 SaaS 里没有本机文件系统可选，必须彻底改为**服务端虚拟工作区**心智。

### 7.1 交互替换

| 桌面端动作 | SaaS 替代 |
|-----------|----------|
| 点击“选择工作目录” → 原生目录选择器 | 弹“选择/新建工作区”面板：列出本租户工作区（`GET /api/v1/workspaces`），或新建（`POST /api/v1/workspaces`）|
| 看到磁盘绝对路径 | 看到工作区名 + 工作区内相对路径面包屑 |
| “最近使用的目录”（`get-recent-cwds`）| “最近工作区/最近文件”：`Workspace.lastAccessAt` + `WorkspaceFileEvent` 派生 |
| 在资源管理器里翻文件 | 内置文件浏览器组件（调 `GET /tree`），前端渲染目录树 |
| 拖入本地文件 | 拖拽 → 走 §4.4 上传（直传/分片）|

### 7.2 浏览器桥顶替（与 03 衔接）

浏览器桥（顶替 `window.electron`）把这些桌面 IPC 映射到文件服务：

桥的**响应形状必须适配回现状契约**（对齐附录 C §3.4 与源码 P1-7），否则会打断 `ArtifactPanel` / `CodeRenderer` 等调用方：`saveInlineFile({ dataBase64, fileName, mimeType, cwd }) -> { success, path }`（现状 25MB 上限）、`readFileAsDataUrl(...) -> { success, dataUrl }`。桥内部虽改走 REST/签名 URL，但**对上仍要吐旧形状**：

```ts
// browser bridge (示意) —— 见 03 章的桥总体设计；对外形状 = 现状 IPC 契约
window.electron.dialog = {
  // 旧: 弹原生目录选择器
  selectDirectory: async () => openWorkspacePicker(),          // 打开工作区选择 UI，返回 {workspaceId}

  // 旧: 读本地文件为 data URL -> { success, dataUrl }
  // 新: 走 as=url 拿签名 URL，再由桥 fetch 成 data URL 回填旧形状
  readFileAsDataUrl: async ({ path, cwd } = {}) => {
    const r = await filesApi.read(currentWs, path, { as: 'url' });
    return { success: true, dataUrl: await toDataUrl(r.url) };
  },

  // 旧: saveInlineFile({ dataBase64, fileName, mimeType, cwd }) -> { success, path }（25MB 上限）
  // 新: ≤5MB 走 POST /api/v1/workspaces/:wid/files/upload；>5MB(至 25MB) 桥内改走 §4.4 上传流程，再把结果映射回旧形状
  saveInlineFile: async ({ dataBase64, fileName, mimeType, cwd }) => {
    const relPath = joinWsRelPath(cwd, fileName);
    const { path } = await filesApi.writeInline(currentWs, relPath, { dataBase64, mimeType });
    return { success: true, path };
  },

  statFile: async ({ path }) => filesApi.stat(currentWs, path),
  readTextFile: async ({ path }) => filesApi.read(currentWs, path, { as: 'text' }),
  generateThumbnail: async ({ path }) => filesApi.thumbnail(currentWs, path),
};
window.electron.getRecentCwds = async () => filesApi.listWorkspaces(); // 语义改为“最近工作区”
```

- **上限差异**：现状 `saveInlineFile` 允许 25MB，而新 `POST /api/v1/workspaces/:wid/files/upload` inline body 阈值为 5MB（§4.4）；桥须在 5MB–25MB 区间**自动转走上传（直传/分片）流程**，对调用方仍表现为一次成功的 `saveInlineFile`，避免中断 `ArtifactPanel`/`CodeRenderer`。
- `currentWs`（当前工作区 id）取代桌面端“当前 cwd 绝对路径”；会话仍绑定 `workspaceId`（写入 `cowork_sessions`）。
- 所有原来传绝对路径的地方，桥内改传“工作区内相对路径”，桥不知道也拿不到宿主绝对路径（安全上恰好收敛）。

### 7.3 会话与工作区的绑定

- 新会话默认继承租户“默认工作区”（首个 ACTIVE，或用户设定），对应桌面端“默认 `~/lobsterai/project`”。
- 允许一个工作区服务多会话（同租户）；`cowork_sessions.cwd` → `{workspaceId, relRoot}`，其中 `relRoot` 必须相对 `/workspace/project` 子树。
- 任务型会话仍可在 `ws-{workspaceId}/project/.lobsterai-tasks/{taskId}/` 下建独立子目录（沿用桌面端约定，见 §3.1）。

## 8. 配额、清理与备份

### 8.1 配额模型（与 09 计费联动）

| 维度 | 说明 | 权威 |
|------|------|------|
| 租户存储配额 `tenant.quotaBytes` | 该租户所有工作区总量上限 | 配置存 DB（`06`），**当前用量读 Redis** |
| 工作区配额 `Workspace.quotaBytes` | 单工作区上限，默认继承租户 | 配置存 DB，当前用量读 Redis |
| 单文件上限 | 如 2GB（媒体），普通文件更小 | 配置 |
| 上传预留 | `POST /api/v1/workspaces/:wid/uploads` 原子预留、`landed` 结算、超时原子释放 | Redis 预留计数（权威）|

- **当前用量的唯一权威读源 = Redis 预留计数（对齐附录 C §3.4）**：每租户/每工作区一个 Redis 计数键，写入/上传前置检查与预留一律走原子 `INCRBY`（预留）/回补（释放/结算），保证并发下不超发。**DB `sizeBytes` 只作对账/纠偏，不作前置检查的权威读源**——否则并发多副本读 DB 快照会各自放行、集体超配额。
- **写入前置检查**：`POST /api/v1/workspaces/:wid/files/upload` 与 `POST /api/v1/workspaces/:wid/uploads` 前对 Redis 计数原子预留 `当前用量 + 增量`，若 `> 配额` 则回滚预留并返回 `507 STORAGE_QUOTA_EXCEEDED`（统一错误信封，`details` 带 `used` / `quota`）。注意此处是 507，**不是** 413（413 专指文本/body 过大 `PAYLOAD_TOO_LARGE`，见 §4.3）。
- **对账巡检**：后台巡检（BullMQ 定时任务）周期性对工作区做 `du`，把实际占用回写 DB `sizeBytes` 并校正 Redis 计数，纠偏漂移；漂移阈值见 §10.3。
- 超配额行为：拒绝新写入、提示升级/清理；不影响读取（见 09 的降级策略）。

### 8.2 清理策略

```mermaid
flowchart LR
  DEL[DELETE /api/v1/workspaces/:wid/files?path=] -->|软删| TRASH[.trash/]
  TRASH -->|保留 N 天| GC[配额巡检 worker]
  GC -->|物理删除 PVC + S3 镜像| GONE[释放]
  ARCH[长期未访问工作区] -->|status=ARCHIVED| BAK[打包进 S3 backups]
  BAK -->|按策略| DELWS[卸载 PVC 子目录]
```

- **软删 → 回收站**：`DELETE` 移入 `.trash/`，保留期（如 7 天）后由巡检 worker 物理删除 PVC 文件与 S3 镜像。
- **冷工作区归档**：`lastAccessAt` 超阈值的工作区打包进 `backups/` 后释放 PVC 空间（`status=ARCHIVED`），再次访问时从备份恢复（GA 后续的方案 B 思路）。
- **租户注销/删除**：级联删除该租户所有 `tenants/{tenantId}/` 前缀对象 + PVC 子树 + DB 记录（合规删除，见 14）。

### 8.3 备份

- 定时（如每日）对活跃工作区做增量/全量打包 `tar.zst` 到 `backups/`，异地/跨可用区副本。
- 备份包本身受同样的租户前缀隔离与 IAM 约束；恢复接口仅管理员可用。
- 与整体备份/DR 策略衔接见 `15-部署运维与可观测性.md`。

## 10. 验收标准

### 10.1 功能

- [ ] 用户可在 SPA 中列出、创建、切换工作区（无任何宿主绝对路径出现在 UI 或 API 响应）。
- [ ] 文件浏览器可展开工作区目录树，展示名称/类型/大小/mtime，大目录分页正常。
- [ ] 文本文件可读可写（`as=text` / `POST /api/v1/workspaces/:wid/files/upload`）；二进制/图片通过 `as=url` 签名 URL 预览。
- [ ] 小文件（≤5MB）直写成功；大文件/视频经直传或分片上传成功并可断点续传。
- [ ] agent 在 Pod 内落盘的文件，前端在**同一工作区**能通过 `/tree` 与预览看到，且 3s 内收到 `files:changed`。
- [ ] Artifact 文件预览通过对象存储签名 URL 渲染，HTML/PPTX 高保真且沙箱隔离（细节验收在 12）。
- [ ] `dialog:*` / `get-recent-cwds` / `artifact:watchFile` 对应功能在浏览器桥下语义等价可用。

### 10.2 安全（硬性）

- [ ] `../`、绝对路径、盘符、NUL、URL 编码穿越（如 `%2e%2e%2f`）在 tree/read/write/upload 全部被拒（403/404）。
- [ ] 工作区内 symlink 指向 `/etc` 等外部路径时，读写被拒（realpath 校验或挂载禁 symlink）。
- [ ] 用他人或同租户未授权 `workspaceId` 访问均返回 404（不泄露存在性），且服务端从不接受客户端传入完整对象 key。
- [ ] 签名 URL 仅对象级、限时效、方法受限；过期后 403；下载类带 `content-disposition=attachment`。
- [ ] 跨租户：租户 A 无法通过任何接口/URL 读到 `tenants/B/...` 的对象或 PVC 文件。

### 10.3 隔离 / 配额 / 运维

- [ ] 每租户 PVC 物理隔离；文件服务与 OpenClaw Pod 仅挂载本租户卷子树。
- [ ] 超存储配额写入返回 `507 STORAGE_QUOTA_EXCEEDED`（非 413，且不得复用账单额度 `QUOTA_EXCEEDED`）；读取不受影响；Redis 预留计数与实际 `du` 的**漂移 ≤ 用量的 1%，且绝对值 ≤ 100MB**（超阈值触发对账纠偏并告警）。
- [ ] 单写者租约生效：两个 Pod 争抢同一工作区并发写 `MEMORY.md`/同一文件时，写入不丢、不交错、不损坏（对应 §2.4「单写者租约 P0」、§6.3）。
- [ ] 软删文件进 `.trash/`，保留期后物理清理；租户删除级联清空 PVC + S3 前缀 + DB。
- [ ] 备份可生成并可恢复；备份对象受同等租户隔离约束。

### 10.4 性能

- [ ] 目录浏览、文本读取 P95 < 300ms（**工作区常规规模**定义为：单目录 ≤ 1000 项、工作区总文件数 ≤ 5 万、总量 ≤ 该工作区配额；超此规模按 §2.4 大目录标准 P95 listing < 2s）。
- [ ] 大文件下载/预览走对象存储直连，不占用应用服务器出口带宽。
- [ ] `files:changed` 端到端延迟 P95 < 3s（含 300ms debounce）。

## 11. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| RWX 存储不传播 inotify | agent 落盘后前端收不到 `files:changed` | 优先用 OpenClaw 写工具事件驱动；inotify 兜底；必要时轮询 mtime |
| PVC↔S3 双写不一致 | 预览到旧内容 | PVC 为权威 + `as=url` 前新鲜度校验 + 镜像重试队列 |
| 路径穿越/symlink 逃逸 | 跨租户或读宿主文件 | §5 统一守卫 + realpath + 挂载禁 symlink（由 07 保证）|
| 客户端拼 key 绕过隔离 | 跨租户对象访问 | key 只由服务端按 `ctx.tenantId` 拼接，签名限对象级 |
| 大文件占满 PVC | 影响同租户其他工作区 | 上传预留 + 单文件上限 + 配额前置检查 + 巡检纠偏 |
| RWX 存储成本/性能 | 大规模下昂贵或慢 | 冷工作区归档到对象存储（方案 B / GA 后续冷热分层）|
| 存量桌面数据迁移遗漏 | 用户老文件丢失 | 提供可选一键打包上传；不自动改动用户本机数据 |
